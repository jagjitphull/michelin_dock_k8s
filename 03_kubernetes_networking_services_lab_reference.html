3. Cluster Communications

Kubernetes networking connects several communication paths. Containers in the same Pod can use localhost. Pods normally talk to other Pods through Services, and external users access applications through NodePort, LoadBalancer, or Ingress.

Communication PathExampleTypical Object
Container to container in same PodApp talks to sidecar on localhostPod
Pod to PodFrontend talks to backendPod IP, usually via Service
Pod to ServiceApp calls backend.netlab.svc.cluster.localService + DNS
External user to appBrowser opens applicationNodePort, LoadBalancer, Ingress

The moving dot represents a request packet. In a real cluster, kube-proxy or eBPF dataplane, CNI routing, DNS, and node networking participate in the flow.

4. Pod and Node Networking

Every Node has a node IP. Every Pod gets its own Pod IP from the cluster Pod network. Pods are ephemeral, so when a Pod is recreated it can receive a different IP.

Pod and Node Networking View

Node 1
192.168.1.11

Pod A
10.244.1.5
Node 2
192.168.1.12

Pod B
10.244.2.8
Important: Application code should talk to a Service name, not to an individual Pod IP. A Service gives a stable virtual IP and DNS name while Pods behind it can change.

5. Container Network Interface (CNI)

CNI is the plugin mechanism used by Kubernetes networking solutions to connect Pods to the cluster network. Examples include Calico, Cilium, Flannel, Weave Net, Antrea, and cloud provider CNIs.

What CNI does

  • Assigns Pod IP addresses.
  • Connects Pod network interfaces.
  • Enables Pod-to-Pod communication.
  • May enforce NetworkPolicies, depending on plugin support.

What CNI does not replace

  • It does not replace Services.
  • It does not automatically create application DNS names.
  • It does not expose applications externally by itself.
# Check which CNI/networking pods are running
kubectl get pods -n kube-system -o wide

6. Understanding Pod IP Assignment is Dynamic

A Pod IP is assigned when the Pod is created. If the Pod is deleted and recreated, the new Pod can receive a different IP.

Never hardcode Pod IP addresses in application code. Hardcoded Pod IPs can break after restart, rescheduling, scaling, or rolling update.

Mini Demonstration

kubectl create namespace netlab
kubectl -n netlab run temp-nginx --image=nginx:1.27 --restart=Never
kubectl -n netlab get pod temp-nginx -o wide

# Delete and recreate the Pod
kubectl -n netlab delete pod temp-nginx
kubectl -n netlab run temp-nginx --image=nginx:1.27 --restart=Never
kubectl -n netlab get pod temp-nginx -o wide

The IP may change. Even if it appears unchanged in a small local cluster, production design must still assume Pod IPs are temporary.

7. Understanding How Services Route Traffic to Pods

A Service selects Pods using labels. Traffic sent to the Service is forwarded to one of the matching backend Pod endpoints.

Animated Concept: Service Selector to Pods

The moving orange packets show Service traffic being distributed to Pods that match the selector app=web. The Service does not care about Pod names; it finds backend Pods through labels.

ObjectPurposeCommand
ServiceStable virtual access point for Podskubectl get svc
EndpointSliceStores backend Pod IPs selected by the Servicekubectl get endpointslice
kube-proxy / dataplaneImplements forwarding rules on nodeskubectl get pods -n kube-system

8. Understanding How DNS Names are Generated for Services

Kubernetes creates DNS records for Services. Applications should use these names instead of Pod IP addresses.

Short name

web-clusterip

Works from Pods in the same namespace.

Namespace-qualified name

web-clusterip.netlab

Useful from another namespace.

Full service DNS name

web-clusterip.netlab.svc.cluster.local

Fully qualified cluster DNS name.

# Test DNS and HTTP from a temporary BusyBox Pod
kubectl -n netlab run dns-demo \
  --image=busybox:1.36 \
  --restart=Never \
  --rm -i \
  -- nslookup web-clusterip

kubectl -n netlab run http-demo \
  --image=busybox:1.36 \
  --restart=Never \
  --rm -i \
  -- wget -qO- http://web-clusterip

10. Ingress Controller and Ingress Rules

An Ingress defines HTTP/HTTPS routing rules. It does not process traffic by itself; the cluster must have an Ingress Controller such as NGINX Ingress Controller, Traefik, HAProxy, or a cloud provider controller.

Animated Concept: Ingress HTTP Routing

Browser
training.local
Ingress Controller
Service
Pods

Sample Ingress Rule

apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: web-ingress
spec:
  ingressClassName: nginx
  rules:
    - host: training.local
      http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service:
                name: web-clusterip
                port:
                  number: 80
Important: Create this Ingress only after installing an Ingress Controller. Without a controller, the Ingress object exists but traffic will not be handled.

11. NodePort or kubectl port-forward?

OptionBest ForNot Best ForExample
kubectl port-forwardDeveloper testing, debugging, temporary local accessSharing with many users or production accesskubectl -n netlab port-forward svc/web-clusterip 8080:80
NodePortTraining labs, bare-metal demos, simple external accessPolished production exposure, TLS routing, hostname-based routinghttp://NodeIP:30080
Practical rule: Use kubectl port-forward for quick local troubleshooting. Use NodePort when trainees need to access the app using node IP and port. Use Ingress or LoadBalancer for production-style HTTP/HTTPS exposure.

12. Lab: Exposing Applications using ClusterIP Service

Goal: Deploy an NGINX web application and expose it only inside the Kubernetes cluster using a stable ClusterIP Service.

Use case: A frontend Pod, backend API Pod, or batch job inside the cluster needs to call an internal web service. External users should not directly access this Service.

What trainees will prove in this lab

  • Pods receive dynamic IP addresses.
  • A Service gives a stable virtual IP and DNS name.
  • The Service selector finds matching Pods using labels.
  • Traffic continues to work even when Pods scale up or down.

Step 1: Check cluster access

kubectl version --client
kubectl cluster-info
kubectl get nodes -o wide

Expected: kubectl get nodes should show at least one node in Ready status.

Step 2: Create the complete ClusterIP lab manifest

cat <<'EOF' > web-clusterip-lab.yaml
apiVersion: v1
kind: Namespace
metadata:
  name: netlab
---
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web
  namespace: netlab
  labels:
    app: web
spec:
  replicas: 3
  selector:
    matchLabels:
      app: web
  template:
    metadata:
      labels:
        app: web
    spec:
      containers:
        - name: nginx
          image: nginx:1.27-alpine
          imagePullPolicy: IfNotPresent
          ports:
            - name: http
              containerPort: 80
---
apiVersion: v1
kind: Service
metadata:
  name: web-clusterip
  namespace: netlab
  labels:
    app: web
spec:
  type: ClusterIP
  selector:
    app: web
  ports:
    - name: http
      protocol: TCP
      port: 80
      targetPort: http
EOF

This single file creates a namespace, a 3-replica NGINX Deployment, and a ClusterIP Service named web-clusterip.

Step 3: Apply the manifest and verify the workload

kubectl apply -f web-clusterip-lab.yaml
kubectl -n netlab rollout status deployment/web --timeout=120s
kubectl -n netlab get deployment web
kubectl -n netlab get pods -l app=web -o wide
kubectl -n netlab get svc web-clusterip -o wide
Expected: Deployment should become available, three Pods should be Running, and Service type should be ClusterIP.

Step 4: Verify labels, selector, and endpoints

kubectl -n netlab get pods --show-labels
kubectl -n netlab describe svc web-clusterip
kubectl -n netlab get endpointslice -l kubernetes.io/service-name=web-clusterip -o wide
kubectl -n netlab get endpoints web-clusterip

Key check: The Service selector should show app=web. EndpointSlice should contain Pod IPs. These Pod IPs are backend targets and must not be hardcoded in application code.

Step 5: Test Kubernetes DNS from inside the cluster

kubectl -n netlab run dns-test \
  --image=busybox:1.36 \
  --restart=Never \
  --rm -i \
  -- nslookup web-clusterip

kubectl -n netlab run dns-fqdn-test \
  --image=busybox:1.36 \
  --restart=Never \
  --rm -i \
  -- nslookup web-clusterip.netlab.svc.cluster.local
Expected: Both short name web-clusterip and FQDN web-clusterip.netlab.svc.cluster.local should resolve.

Step 6: Test HTTP access from inside the cluster

kubectl -n netlab run http-test \
  --image=busybox:1.36 \
  --restart=Never \
  --rm -i \
  -- wget -qO- http://web-clusterip

kubectl -n netlab run http-fqdn-test \
  --image=busybox:1.36 \
  --restart=Never \
  --rm -i \
  -- wget -qO- http://web-clusterip.netlab.svc.cluster.local
Expected: The command should print the NGINX welcome HTML. This proves that internal cluster clients can access the app using the Service name.

Step 7: Prove that Service backends update when Pods change

kubectl -n netlab scale deployment web --replicas=5
kubectl -n netlab rollout status deployment/web --timeout=120s
kubectl -n netlab get pods -l app=web -o wide
kubectl -n netlab get endpointslice -l kubernetes.io/service-name=web-clusterip -o wide

kubectl -n netlab scale deployment web --replicas=2
kubectl -n netlab rollout status deployment/web --timeout=120s
kubectl -n netlab get endpointslice -l kubernetes.io/service-name=web-clusterip -o wide

Observation: The Service name and ClusterIP do not change. EndpointSlice backend Pod IPs change as replicas are added or removed.

Step 8: Optional local access using port-forward

# Terminal 1
kubectl -n netlab port-forward svc/web-clusterip 8080:80

# Terminal 2
curl http://127.0.0.1:8080

Important: Port-forward is only for local debugging. It does not expose the Service to other users like NodePort, LoadBalancer, or Ingress.

Step 9: Optional selector troubleshooting mini-lab

# Break the Service selector intentionally
kubectl -n netlab patch svc web-clusterip \
  -p '{"spec":{"selector":{"app":"wrong-label"}}}'

kubectl -n netlab get endpointslice -l kubernetes.io/service-name=web-clusterip
kubectl -n netlab describe svc web-clusterip

# Restore the correct selector
kubectl -n netlab patch svc web-clusterip \
  -p '{"spec":{"selector":{"app":"web"}}}'

kubectl -n netlab get endpointslice -l kubernetes.io/service-name=web-clusterip -o wide
Learning point: A Service with the wrong selector has no endpoints. If a Service is not working, first compare kubectl get pods --show-labels with kubectl describe svc.

Step 10: Cleanup

kubectl delete -f web-clusterip-lab.yaml
rm -f web-clusterip-lab.yaml

13. Lab: Exposing Applications using NodePort Service

Goal: Expose an NGINX application outside the cluster through NodeIP:NodePort.

Use case: A trainee needs to access an app from a browser or terminal outside the Kubernetes cluster in a lab setup without configuring Ingress or a cloud LoadBalancer.

What trainees will prove in this lab

  • NodePort creates a port on every Kubernetes node.
  • Traffic reaches the Service using NodeIP:30080.
  • The Service still routes traffic to Pods using labels and EndpointSlices.
  • NodePort reachability depends on the node network, VM network, local firewall, and cloud security rules.

Step 1: Check cluster access

kubectl version --client
kubectl cluster-info
kubectl get nodes -o wide

Expected: Nodes should be Ready. Note the node INTERNAL-IP or reachable external IP.

Step 2: Create the complete NodePort lab manifest

cat <<'EOF' > web-nodeport-lab.yaml
apiVersion: v1
kind: Namespace
metadata:
  name: netlab
---
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web
  namespace: netlab
  labels:
    app: web
spec:
  replicas: 3
  selector:
    matchLabels:
      app: web
  template:
    metadata:
      labels:
        app: web
    spec:
      containers:
        - name: nginx
          image: nginx:1.27-alpine
          imagePullPolicy: IfNotPresent
          ports:
            - name: http
              containerPort: 80
---
apiVersion: v1
kind: Service
metadata:
  name: web-nodeport
  namespace: netlab
  labels:
    app: web
spec:
  type: NodePort
  selector:
    app: web
  ports:
    - name: http
      protocol: TCP
      port: 80
      targetPort: http
      nodePort: 30080
EOF

This file is self-contained. It creates the namespace, Deployment, and NodePort Service. The static NodePort is 30080, which is inside the default Kubernetes NodePort range 30000-32767.

Step 3: Apply the manifest and verify the Service

kubectl apply -f web-nodeport-lab.yaml
kubectl -n netlab rollout status deployment/web --timeout=120s
kubectl -n netlab get pods -l app=web -o wide
kubectl -n netlab get svc web-nodeport -o wide
kubectl -n netlab describe svc web-nodeport
kubectl -n netlab get endpointslice -l kubernetes.io/service-name=web-nodeport -o wide
Expected: Service type should be NodePort, port mapping should show 80:30080/TCP, and EndpointSlice should show backend Pod IPs.

Step 4: Access using node IP on Linux/macOS shell

kubectl get nodes -o wide

# Linux/macOS helper: first node InternalIP
NODE_IP=$(kubectl get nodes -o jsonpath='{.items[0].status.addresses[?(@.type=="InternalIP")].address}')
echo $NODE_IP
curl http://$NODE_IP:30080
Expected: The command should return the NGINX welcome page.

Step 5: Manual access from browser or terminal

# Replace NODE_IP with a reachable node IP from: kubectl get nodes -o wide
curl http://NODE_IP:30080

# Example only:
# curl http://192.168.56.20:30080

Open http://NODE_IP:30080 in a browser if the node IP is reachable from your workstation.

Step 6: Minikube-specific test

minikube status
minikube ip
curl http://$(minikube ip):30080

# Alternative command that prints the service URL:
minikube service web-nodeport -n netlab --url

Minikube usually runs Kubernetes inside a local VM or container. minikube service is often the simplest way to open or print the correct URL.

Step 7: kind-specific working setup

cat <<'EOF' > kind-nodeport-config.yaml
kind: Cluster
apiVersion: kind.x-k8s.io/v1alpha4
nodes:
  - role: control-plane
    extraPortMappings:
      - containerPort: 30080
        hostPort: 30080
        protocol: TCP
EOF

kind create cluster --name netlab-kind --config kind-nodeport-config.yaml
kubectl apply -f web-nodeport-lab.yaml
kubectl -n netlab rollout status deployment/web --timeout=120s
curl http://127.0.0.1:30080
Important: In a default kind cluster, NodePort is often not reachable from the host unless you create the cluster with extraPortMappings. Create the kind cluster with the mapping before applying the Service.

Step 8: Firewall and networking checks when NodePort fails

# On Linux node, check whether kube-proxy opened NodePort handling
kubectl -n kube-system get pods -l k8s-app=kube-proxy -o wide
kubectl -n netlab get svc web-nodeport -o yaml

# If node firewall is active, allow TCP 30080 in your lab environment.
# Ubuntu example:
sudo ufw allow 30080/tcp

# RHEL/Rocky example:
sudo firewall-cmd --add-port=30080/tcp --permanent
sudo firewall-cmd --reload

For cloud VMs, also open TCP 30080 in the cloud security group or network security rule. For corporate laptops, local firewall/VPN policy may block access.

Step 9: Fallback test using port-forward

# Useful when NodePort is blocked by laptop VM networking, firewall, or kind without port mapping
kubectl -n netlab port-forward svc/web-nodeport 8080:80

# In another terminal:
curl http://127.0.0.1:8080

If port-forward works but NodePort does not, the application and Service are healthy. The issue is outside the Pod path, usually node reachability, firewall, VM NAT, kind mapping, or cloud security rules.

Step 10: Cleanup

kubectl delete -f web-nodeport-lab.yaml
rm -f web-nodeport-lab.yaml

# Only if you created the optional kind cluster:
kind delete cluster --name netlab-kind
rm -f kind-nodeport-config.yaml

Reference Notes

  • Kubernetes Services provide stable access to dynamic Pods and route to selected backends using labels.
  • Kubernetes DNS creates records for Services and Pods, allowing applications to use names instead of IPs.
  • CNI plugins implement the Kubernetes network model for Pod networking.
  • Ingress handles HTTP/HTTPS routing rules but requires an Ingress Controller to process them.

Sources used for validation: Kubernetes official documentation on Services, DNS for Services and Pods, Network Plugins/CNI, Cluster Networking, Ingress, Ingress Controllers, and Connecting Applications with Services.
