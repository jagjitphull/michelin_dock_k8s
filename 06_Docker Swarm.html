1. What is Docker Swarm?

🧩

Simple meaning

Docker Swarm is Docker’s built-in clustering feature. It allows many Docker machines to work together as one group called a swarm.

Instead of logging in to every server and starting containers manually, you tell Swarm what you want. Example: “Run 3 copies of nginx.” Swarm decides where to run them.

🎯

One-line definition

Docker Swarm = many Docker hosts + automatic scheduling + scaling + self-healing + service discovery.

It is useful for learning orchestration and for simple Docker-native clusters.

2. Why do we need Docker Swarm?

📦

Deploy services

Run an application as a service, not as many manually managed containers.

📈

Scale easily

Increase or decrease replicas with one command.

🩹

Self-healing

If a task fails, Swarm schedules a replacement task.

🌐

Multi-host network

Services on different hosts can communicate using overlay networking.

Reality check: Docker Swarm is simpler than Kubernetes. Kubernetes is more common in large enterprise platforms, but Swarm is easier for beginners to understand orchestration concepts.

3. Docker Swarm Architecture

A Swarm cluster contains nodes. A node is a machine, VM, or cloud server running Docker Engine and participating in the swarm.

Animated Concept 2 — Manager and worker nodes

Managers are the control plane. Workers execute assigned tasks. The moving dots show scheduling and cluster-state communication.

Manager node

The manager is the brain of Swarm. It accepts service definitions, stores cluster state, schedules tasks, and checks whether the real state matches the desired state.

Worker node

The worker runs containers assigned by the manager. It reports task status back to the manager.

Service

A service is the desired application definition. It includes the image, replicas, ports, networks, constraints, and update policy.

Task

A task is one scheduled unit of work. In normal use, one task runs one container.

4. How a Service Runs in Swarm

Swarm uses a desired state model. You describe the final state you want. The manager checks the real cluster and takes action if reality does not match your desired state.

Animated Concept 3 — Desired state and task scheduling

The service definition stays with the manager; small dots show tasks being assigned to workers.
You create a service. Example: docker service create --replicas 3 nginx:alpine.
Manager stores desired state. Desired state says: “3 nginx tasks should run.”
Manager schedules tasks. It chooses available nodes.
Workers run containers. Each task becomes a running container.
Manager keeps checking. If a task stops, the manager creates another task to restore the desired state.

5. Swarm Networking and Load Balancing

Swarm networking helps services communicate across nodes. The two beginner concepts are overlay network and ingress/routing mesh.

Animated Concept 4 — Published port, routing mesh, and overlay network

Request enters the published port, then Swarm can route traffic to one of the running service tasks.

Overlay network

An overlay network connects services running on different Docker hosts.

docker network create --driver overlay app-net

Ingress / routing mesh

When a service port is published, Swarm can accept traffic on swarm nodes and route it to an available task.

docker service create \
  --name web \
  --replicas 3 \
  --publish published=8080,target=80 \
  nginx:alpine

6. Common Troubleshooting

ProblemLikely causeFix
docker swarm init asks for advertise addressHost has multiple network interfacesRun docker swarm init --advertise-addr <IP>
Worker cannot joinWrong IP, expired/wrong token, or port blockedRun docker swarm join-token worker again and allow TCP 2377
Service stuck in pendingNo suitable node, image pull problem, resource issue, or constraint issueRun docker service ps --no-trunc <service>
Cannot access published portFirewall, wrong port, or service not runningCheck docker service ls, docker service ps, and firewall rules
Cannot remove overlay networkService is still attachedRemove services first, then remove the network
This node is already part of a swarmSwarm already initialized or joinedLeave the swarm carefully with docker swarm leave --force only when you intentionally want to remove this node from the swarm
Trainer tip: Always ask trainees to run docker service ps --no-trunc SERVICE_NAME. It shows the real reason a task failed, such as image pull failure, port conflict, constraint mismatch, or container crash.

Answer key

Show answers
  1. A node is a Docker Engine participating in the swarm.
  2. Managers control and schedule; workers run assigned tasks.
  3. A service is the desired definition of an application workload.
  4. A task is the smallest scheduled unit, usually one container.
  5. Desired state lets Swarm continuously reconcile actual state with what the user requested.
  6. An overlay network lets services communicate across Docker hosts.
  7. Routing mesh routes traffic from a published port to available service tasks.
  8. Odd manager counts reduce split-vote risk and improve quorum behavior.

Reference Notes

  • Swarm mode is part of Docker Engine.
  • For manager high availability, prefer an odd number of manager nodes.
  • For multi-node Swarm networking, open TCP 2377, TCP/UDP 7946, and UDP 4789 between participating hosts.
