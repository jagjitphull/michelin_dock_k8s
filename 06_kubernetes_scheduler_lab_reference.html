Working with Kubernetes Scheduler

This reference explains how Kubernetes decides where a Pod should run, then walks trainees through practical labs for manual scheduling, node labels with nodeSelector, and taints with tolerations.

Recommended lab environment: a non-production Kubernetes cluster with at least two nodes. A single-node cluster can still run most commands, but placement behavior is easier to observe with two or more nodes.

1. Learning objectives

1

Understand scheduling

Explain how kube-scheduler watches unscheduled Pods and selects a suitable Node.

2

Compare scheduling methods

Differentiate between dynamic scheduling, direct Pod-to-Node binding using nodeName, and label-based scheduling using nodeSelector.

3

Control placement safely

Use taints and tolerations to keep unsuitable Pods away from special-purpose Nodes.

2. Understanding kube-scheduler and its features

The kube-scheduler is a control-plane component. It watches for newly created Pods that do not yet have a Node assigned, evaluates available Nodes, and then binds each schedulable Pod to a selected Node.

Animated concept: scheduler decision flow

1. Pod createdPod has no spec.nodeName.
2. FilterRemove Nodes that cannot run the Pod.
3. ScoreRank feasible Nodes.
4. BindAssign Pod to the chosen Node.
API ServerPod is stored
SchedulerWatches unscheduled Pods
FilterResources, labels, taints
ScoreBest feasible Node
KubeletRuns Pod on Node

Important kube-scheduler features

FeatureMeaningSimple example
Scheduling queueNew Pods without an assigned Node wait for scheduling.A Deployment creates three Pods; scheduler processes each Pod.
FilteringNodes that cannot satisfy the Pod requirements are removed from consideration.A Pod needs a Node label disktype=ssd; Nodes without that label are filtered out.
ScoringFeasible Nodes are ranked; the best ranked Node is selected.If multiple Nodes match, the scheduler chooses the most suitable Node based on active scoring rules.
BindingThe selected Node is written back to the Pod as spec.nodeName.After scheduling, kubectl get pod -o wide shows the Node name.
ConstraintsScheduling can consider resources, labels, affinity, taints/tolerations, topology, volumes, and policies.A GPU workload can be restricted to GPU Nodes.
ExtensibilityKubernetes supports scheduler configuration and multiple schedulers for advanced cases.A custom scheduler can be used for special workloads.
Trainer explanation: Think of kube-scheduler as a placement decision maker. It does not run containers. It only chooses a Node and records that decision. The kubelet on that Node then starts the containers.

3. Manual Scheduling vs Dynamic Scheduling

Dynamic scheduling

The normal Kubernetes behavior. You create a Pod without spec.nodeName. The scheduler filters, scores, and binds the Pod to a suitable Node.

  • Recommended for normal workloads.
  • Honors scheduler rules and constraints.
  • Works well with Deployments, ReplicaSets, Jobs, and StatefulSets.

Manual scheduling

You directly set spec.nodeName in the Pod manifest. This bypasses scheduler placement.

  • Useful for demos, testing, and debugging.
  • Not recommended for regular application deployment.
  • If the selected Node is wrong or unavailable, the Pod can remain Pending.
Production warning: Avoid hardcoding nodeName in production manifests. It removes scheduler flexibility and can break during node replacement, upgrades, or autoscaling.

5. Taints and Tolerations

Node labels attract Pods when used with selectors or affinity. Taints do the opposite: they repel Pods from a Node unless the Pod has a matching toleration.

Without toleration

Node has taint dedicated=training:NoSchedule. Pod has no matching toleration.

Pod

With toleration

Pod tolerates dedicated=training:NoSchedule, so the scheduler may place it there.

Pod

Taint effects

EffectBehaviorTraining note
NoScheduleNew Pods without a matching toleration are not scheduled onto the tainted Node.Best effect for this lab.
PreferNoScheduleScheduler tries to avoid the Node, but it is not a strict rule.Useful when avoidance is preferred but not mandatory.
NoExecuteNew Pods without toleration are not scheduled, and existing Pods without toleration can be evicted.Avoid in beginner/shared labs unless you intentionally test eviction.
Important detail: A toleration allows scheduling onto a tainted Node, but it does not force scheduling there. To target a tainted Node in a lab, combine a toleration with a nodeSelector label.

6. Lab setup

Option A: Use an existing training cluster

Use this option if your trainees already have a kubeadm, minikube, kind, OpenShift-compatible Kubernetes, or cloud Kubernetes cluster.

Check cluster access
kubectl cluster-info
kubectl get nodes -o wide
kubectl version --short 2>/dev/null || kubectl version

Option B: Create a local multi-node kind cluster

Use this only on a local lab machine with Docker and kind installed.

Create local kind cluster
cat <<'EOF' > kind-scheduler-lab.yaml
kind: Cluster
apiVersion: kind.x-k8s.io/v1alpha4
nodes:
- role: control-plane
- role: worker
- role: worker
EOF

kind create cluster --name scheduler-lab --config kind-scheduler-lab.yaml
kubectl get nodes -o wide

Create namespace and select a lab node

Namespace and helper variables
kubectl create namespace scheduler-lab --dry-run=client -o yaml | kubectl apply -f -

# Prefer a worker node. If the cluster has no explicit worker label, fall back to the first available node.
NODE1=$(kubectl get nodes --selector='!node-role.kubernetes.io/control-plane,!node-role.kubernetes.io/master' -o jsonpath='{.items[0].metadata.name}')
if [ -z "$NODE1" ]; then
  NODE1=$(kubectl get nodes -o jsonpath='{.items[0].metadata.name}')
fi

echo "Selected lab node: $NODE1"
kubectl get node "$NODE1" --show-labels
Do not use a production cluster for these labs. The taint lab modifies Node scheduling behavior. The cleanup commands are provided, but trainees should still work on a disposable training cluster.

Lab 1: Manually scheduling Pod

nodeName

Goal

Create one Pod that lets kube-scheduler choose the Node, then create another Pod that is manually assigned to a Node using spec.nodeName.

Step 1: Create a dynamically scheduled Pod

01-dynamic-pod.yaml
cat <<'EOF' > 01-dynamic-pod.yaml
apiVersion: v1
kind: Pod
metadata:
  name: dynamic-pod
  namespace: scheduler-lab
  labels:
    lab: scheduler
    scheduling: dynamic
spec:
  containers:
  - name: pause
    image: registry.k8s.io/pause:3.10
    imagePullPolicy: IfNotPresent
    resources:
      requests:
        cpu: "20m"
        memory: "32Mi"
      limits:
        cpu: "100m"
        memory: "64Mi"
EOF

kubectl apply -f 01-dynamic-pod.yaml
kubectl get pod dynamic-pod -n scheduler-lab -o wide

Expected observation

The NODE column should show the Node selected by kube-scheduler.

Validate scheduler assignment
kubectl get pod dynamic-pod -n scheduler-lab \
  -o custom-columns=NAME:.metadata.name,PHASE:.status.phase,NODE:.spec.nodeName

kubectl describe pod dynamic-pod -n scheduler-lab | sed -n '/Events:/,$p'

Step 2: Create a manually scheduled Pod using nodeName

01-manual-pod.yaml
# Make sure NODE1 is set in your shell
if [ -z "$NODE1" ]; then
  NODE1=$(kubectl get nodes --selector='!node-role.kubernetes.io/control-plane,!node-role.kubernetes.io/master' -o jsonpath='{.items[0].metadata.name}')
  [ -z "$NODE1" ] && NODE1=$(kubectl get nodes -o jsonpath='{.items[0].metadata.name}')
fi

cat <<EOF > 01-manual-pod.yaml
apiVersion: v1
kind: Pod
metadata:
  name: manual-pod
  namespace: scheduler-lab
  labels:
    lab: scheduler
    scheduling: manual
spec:
  nodeName: ${NODE1}
  containers:
  - name: pause
    image: registry.k8s.io/pause:3.10
    imagePullPolicy: IfNotPresent
    resources:
      requests:
        cpu: "20m"
        memory: "32Mi"
      limits:
        cpu: "100m"
        memory: "64Mi"
EOF

kubectl apply -f 01-manual-pod.yaml
kubectl get pod manual-pod -n scheduler-lab -o wide

Step 3: Compare both Pods

Compare placement
kubectl get pods -n scheduler-lab -o wide

kubectl get pod dynamic-pod manual-pod -n scheduler-lab \
  -o custom-columns=NAME:.metadata.name,SCHEDULER:.spec.schedulerName,NODE:.spec.nodeName,PHASE:.status.phase
Learning point: The dynamic Pod is placed by the scheduler. The manual Pod already contains a Node assignment, so normal scheduler placement is bypassed.

Optional negative test: wrong nodeName

This shows why hardcoding Node names is fragile.

Wrong nodeName test
cat <<'EOF' > 01-wrong-node-pod.yaml
apiVersion: v1
kind: Pod
metadata:
  name: wrong-node-pod
  namespace: scheduler-lab
spec:
  nodeName: node-does-not-exist
  containers:
  - name: pause
    image: registry.k8s.io/pause:3.10

kubectl apply -f 01-wrong-node-pod.yaml
sleep 5

kubectl delete pod wrong-node-pod -n scheduler-lab --ignore-not-found=true

Lab 2: Scheduling Pod based on Node Selector and Labels

nodeSelector

Goal

Label a Node and create a Pod that can run only on Nodes matching that label.

Step 1: Label the selected Node

Apply Node labels
# Re-detect NODE1 if needed
if [ -z "$NODE1" ]; then
  NODE1=$(kubectl get nodes --selector='!node-role.kubernetes.io/control-plane,!node-role.kubernetes.io/master' -o jsonpath='{.items[0].metadata.name}')
  [ -z "$NODE1" ] && NODE1=$(kubectl get nodes -o jsonpath='{.items[0].metadata.name}')
fi

kubectl label node "$NODE1" disktype=ssd scheduler-lab=enabled --overwrite
kubectl get nodes -L disktype,scheduler-lab

Step 2: Create a Pod with nodeSelector

02-selector-pod.yaml
cat <<'EOF' > 02-selector-pod.yaml
apiVersion: v1
kind: Pod
metadata:
  name: selector-pod
  namespace: scheduler-lab
  labels:
    lab: scheduler
    scheduling: nodeselector
spec:
  nodeSelector:
    disktype: ssd
  containers:
  - name: pause
    image: registry.k8s.io/pause:3.10
    imagePullPolicy: IfNotPresent
    resources:
      requests:
        cpu: "20m"
        memory: "32Mi"
EOF

kubectl apply -f 02-selector-pod.yaml
kubectl get pod selector-pod -n scheduler-lab -o wide

Step 3: Verify that it ran on the labelled Node

Validate nodeSelector placement
echo "Expected labelled node: $NODE1"

kubectl get pod selector-pod -n scheduler-lab \
  -o custom-columns=NAME:.metadata.name,NODE:.spec.nodeName,PHASE:.status.phase

ACTUAL_NODE=$(kubectl get pod selector-pod -n scheduler-lab -o jsonpath='{.spec.nodeName}')
echo "Actual node: $ACTUAL_NODE"
kubectl get node "$ACTUAL_NODE" -L disktype,scheduler-lab

Step 4: Negative test with a missing label

Create a Pod that requires disktype=nvme. Because no Node has that label, the Pod should remain Pending.

02-selector-mismatch-pod.yaml
cat <<'EOF' > 02-selector-mismatch-pod.yaml
apiVersion: v1
kind: Pod
metadata:
  name: selector-mismatch-pod
  namespace: scheduler-lab
spec:
  nodeSelector:
    disktype: nvme
  containers:
  - name: pause
    image: registry.k8s.io/pause:3.10
EOF

kubectl apply -f 02-selector-mismatch-pod.yaml
sleep 5
kubectl get pod selector-mismatch-pod -n scheduler-lab -o wide
kubectl describe pod selector-mismatch-pod -n scheduler-lab | sed -n '/Events:/,$p'

  • selector-pod should be Running on a Node labelled disktype=ssd.

Fix missing label or delete mismatch Pod

Lab 3: Taints and Tolerations

NoSchedule

Goal

Apply a taint to a Node, prove that a Pod without toleration cannot schedule there, then create a Pod with a matching toleration.

Safety rule: This lab uses NoSchedule, not NoExecute. Do not use NoExecute on shared clusters unless you are intentionally testing eviction behavior.

Step 1: Label and taint one Node

Apply lab label and taint
# Re-detect NODE1 if needed
if [ -z "$NODE1" ]; then
  NODE1=$(kubectl get nodes --selector='!node-role.kubernetes.io/control-plane,!node-role.kubernetes.io/master' -o jsonpath='{.items[0].metadata.name}')
  [ -z "$NODE1" ] && NODE1=$(kubectl get nodes -o jsonpath='{.items[0].metadata.name}')
fi

kubectl label node "$NODE1" taint-lab=yes --overwrite
kubectl taint node "$NODE1" dedicated=training:NoSchedule --overwrite

kubectl get node "$NODE1" -L taint-lab
kubectl describe node "$NODE1" | grep -i "Taints:"

Step 2: Create a Pod without toleration

This Pod targets the tainted Node using nodeSelector, but it does not tolerate the taint. It should remain Pending.

03-no-toleration-pod.yaml
cat <<'EOF' > 03-no-toleration-pod.yaml
apiVersion: v1
kind: Pod
metadata:
  name: no-toleration-pod
  namespace: scheduler-lab
spec:
  nodeSelector:
    taint-lab: "yes"
  containers:
  - name: pause
    image: registry.k8s.io/pause:3.10
EOF

kubectl apply -f 03-no-toleration-pod.yaml
sleep 5
kubectl get pod no-toleration-pod -n scheduler-lab -o wide
kubectl describe pod no-toleration-pod -n scheduler-lab | sed -n '/Events:/,$p'

Expected observation

The Pod should stay Pending. The Events section should mention that the Node had an untolerated taint similar to dedicated=training:NoSchedule.

Step 3: Create a Pod with matching toleration

03-with-toleration-pod.yaml
cat <<'EOF' > 03-with-toleration-pod.yaml
apiVersion: v1
kind: Pod
metadata:
  name: with-toleration-pod
  namespace: scheduler-lab
spec:
  nodeSelector:
    taint-lab: "yes"
  tolerations:
  - key: "dedicated"
    operator: "Equal"
    value: "training"
    effect: "NoSchedule"
  containers:
  - name: pause
    image: registry.k8s.io/pause:3.10
    imagePullPolicy: IfNotPresent
EOF

kubectl apply -f 03-with-toleration-pod.yaml
kubectl get pod with-toleration-pod -n scheduler-lab -o wide

Step 4: Verify both results

Verify taint behavior
kubectl get pods -n scheduler-lab -o wide

kubectl get pod with-toleration-pod -n scheduler-lab \
  -o custom-columns=NAME:.metadata.name,NODE:.spec.nodeName,PHASE:.status.phase

kubectl describe pod with-toleration-pod -n scheduler-lab | sed -n '/Tolerations:/,/Events:/p'
Learning point: The taint repels normal Pods. The toleration permits the scheduler to place the Pod on the tainted Node. The nodeSelector in this lab ensures the Pod tries to use that exact labelled Node.

Step 5: Cleanup this lab immediately

Remove taint and lab objects
kubectl delete pod no-toleration-pod with-toleration-pod -n scheduler-lab --ignore-not-found=true
kubectl taint node "$NODE1" dedicated=training:NoSchedule- || true
kubectl label node "$NODE1" taint-lab- || true
kubectl describe node "$NODE1" | grep -i "Taints:"

7. Troubleshooting guide

SymptomLikely reasonCommand to checkFix
Pod remains PendingNo Node satisfies selector, resources, taints, or other constraints.kubectl describe pod POD -n scheduler-labRead Events. Remove bad selector, add label, add toleration, or free resources.
selector-pod runs on unexpected NodeMore than one Node has the same matching label.kubectl get nodes -L disktypeUse a unique label for lab, for example scheduler-lab=enabled.
Manual Pod does not startWrong nodeName, Node not Ready, image pull issue, or kubelet problem.kubectl get nodes
kubectl describe pod manual-pod -n scheduler-lab		Use a valid Ready Node and reachable image.
Toleration Pod is still PendingToleration does not exactly match taint key/value/effect, or another constraint is failing.kubectl describe node NODE | grep Taints
kubectl describe pod POD		Match key, value, and effect. Check labels and resources.
All Pods avoid one NodeThe Node may have an existing taint such as control-plane taint.kubectl describe node NODE | grep TaintsUse a worker Node, or add correct toleration only in lab manifests.

8. Assessment questions

  1. What does kube-scheduler do when a new Pod has no spec.nodeName?
  2. What are the two major stages of scheduler node selection?
  3. Why is nodeName usually a bad choice for production manifests?
  4. How is nodeSelector different from nodeName?
  5. A Pod has nodeSelector: disktype=ssd, but no Node has that label. What phase will the Pod usually remain in?
  6. What command adds a label to a Node?
  7. What is the purpose of a Node taint?
  8. What is the purpose of a Pod toleration?
  9. Does a toleration force a Pod to run on a tainted Node?
  10. Which taint effect can evict existing Pods that do not tolerate it?
Suggested answers
  1. It watches the unscheduled Pod, finds a suitable Node, and binds the Pod to that Node.
  2. Filtering and scoring.
  3. It bypasses scheduler flexibility and breaks if the Node name changes or becomes unavailable.
  4. nodeName directly binds to one Node; nodeSelector lets the scheduler choose any Node with matching labels.
  5. Pending.
  6. kubectl label node NODE key=value --overwrite
  7. To repel Pods from a Node unless they tolerate the taint.
  8. To allow a Pod to be scheduled onto a Node with a matching taint.
  9. No. It only permits scheduling; other scheduler rules still apply.
  10. NoExecute.
