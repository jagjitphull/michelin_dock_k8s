2. Understanding Docker Images and Containers

Docker image

An image is a read-only template containing application files, libraries, runtime, environment settings, and default startup command.

Think of it as a blueprint.

Docker container

A container is a running instance of an image. It adds a thin writable layer on top of the read-only image layers.

Think of it as a running process created from the blueprint.

3. Working with Docker Hub

Docker Hub is a public container registry used to store, pull, and push images. Common image names follow this pattern:

[registry-host/]namespace/repository:tag
nginx:alpine
python:3.12-slim
mydockerid/flask-demo:v1

Developer Machine

docker build
docker tag

Docker Hub

Image registry

Training VM

docker pull
docker run

Pull an image

docker pull alpine:3.20
docker run --rm alpine:3.20 cat /etc/os-release

Push your own image

# Replace mydockerid with your Docker Hub username
docker login
docker tag flask-image-demo:1.0 mydockerid/flask-image-demo:1.0
docker push mydockerid/flask-image-demo:1.0
Training note

Do not push secrets, credentials, private certificates, or internal source code into public registries.

4. Dockerfile Syntax and Best Practices

InstructionPurposeExample
FROMSets the base image. Required for normal Dockerfiles.FROM python:3.12-slim
WORKDIRSets working directory for later commands.WORKDIR /app
COPYCopies files from build context into image.COPY app.py .
RUNRuns commands during image build.RUN pip install flask
ENVSets environment variables.ENV PORT=5000
EXPOSEDocuments the container port. It does not publish the port by itself.EXPOSE 5000
CMDDefault command; can be overridden at runtime.CMD ["python", "app.py"]
ENTRYPOINTMain executable; runtime arguments are appended.ENTRYPOINT ["python"]
Good practices
  • Use official, minimal base images where possible.
  • Copy dependency files before application files to improve caching.
  • Use .dockerignore.
  • Use exec form: CMD ["python", "app.py"].
  • Keep final images small.
Avoid
  • Hardcoding passwords in Dockerfile.
  • Copying the whole project without .dockerignore.
  • Running unnecessary package managers in the final image.
  • Using latest tag in production without control.

6. Image Layers and Caching

Each Dockerfile instruction normally creates a layer. Docker can reuse unchanged layers from previous builds, which makes rebuilds faster.

Dockerfile

FROM
COPY requirements.txt
RUN pip install
COPY app.py

Image Cache

Unchanged layers are reused. Changed layers and later layers rebuild.

Cache-friendly Dockerfile

FROM python:3.12-slim
WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
COPY app.py .
CMD ["python", "app.py"]

Less cache-friendly Dockerfile

FROM python:3.12-slim
WORKDIR /app
COPY . .
RUN pip install --no-cache-dir -r requirements.txt
CMD ["python", "app.py"]
Why the first version is better

If only app.py changes, Docker can reuse the dependency installation layer. In the second version, copying the whole project before installing dependencies often invalidates the cache unnecessarily.

# Rebuild once
docker build -t flask-image-demo:1.0 .

# Edit app.py message, then rebuild
docker build -t flask-image-demo:1.1 .

# Observe CACHED steps in build output when BuildKit is enabled

Lab 2: Multi-stage Go image

mkdir docker-go-multistage
cd docker-go-multistage
package main

import (
    "fmt"
    "net/http"
)

func main() {
    http.HandleFunc("/", func(w http.ResponseWriter, r *http.Request) {
        fmt.Fprintln(w, "Hello from a multi-stage Docker build!")
    })
    http.ListenAndServe(":8080", nil)
}
FROM golang:1.22-alpine AS builder
WORKDIR /src
COPY main.go .
RUN go build -o /out/app main.go

FROM alpine:3.20
WORKDIR /app
COPY --from=builder /out/app /app/app
EXPOSE 8080
CMD ["/app/app"]
docker build -t go-multistage-demo:1.0 .
docker run -d --name go-demo -p 8080:8080 go-multistage-demo:1.0
curl http://localhost:8080
docker image ls go-multistage-demo
docker stop go-demo && docker rm go-demo
Validation

The curl command should print Hello from a multi-stage Docker build!. The final image should not contain the full Go compiler toolchain.

8. Docker Networking Concepts

Docker networking lets containers communicate with each other, the host, and external networks. For single-host labs, a user-defined bridge network is the most useful and predictable option.

Docker Host
Web Containercontainer name: webport: 80
API Containercontainer name: apiport: 5000
DB Containercontainer name: dbport: 5432
User-defined bridge network: app-net
Containers on the same user-defined bridge network can reach each other by container name using Docker DNS.
Internal DNS
Use http://web, not container IP.
Isolation
Separate bridge networks are isolated by default.
Port publishing
Use -p 8081:80 for host access.
Network typeUse caseCommand example
bridgeDefault single-host container networking. Best for local labs and container-to-container DNS on the same host.docker network create app-net
hostContainer uses host network namespace. Useful for specific performance/debug cases. Less isolated.docker run --network host nginx
noneNo network access. Useful for isolation tests.docker run --network none alpine:3.20 ip addr
overlayMulti-host networking, usually with Docker Swarm.docker network create -d overlay app-overlay
Important

EXPOSE in Dockerfile only documents a port. To access a container from the host, publish the port with -p hostPort:containerPort.

9. Docker Networking Labs

Lab 3: Custom bridge network and DNS

docker network create app-net

docker run -d --name web --network app-net nginx:alpine
docker run -it --rm --name client --network app-net alpine:3.20 sh

Inside the client container:

wget -qO- http://web
Validation

The client container should resolve web by container name and print the NGINX welcome page HTML.

Lab 4: Port publishing

docker run -d --name published-web -p 8081:80 nginx:alpine
curl http://localhost:8081
docker ps --filter name=published-web
Validation

PORTS in docker ps should show something like 0.0.0.0:8081->80/tcp.

Lab 5: Network isolation

This lab intentionally creates two separate user-defined bridge networks. The client container is started in detached mode without --rm, so it remains available for Lab 6.

docker network create net-a
docker network create net-b

docker run -d --name web-a --network net-a nginx:alpine
docker run -d --name client-b --network net-b alpine:3.20 sleep 1d

docker ps --filter name=client-b

Test connectivity from client-b to web-a:

docker exec client-b wget -qO- http://web-a || echo "Expected failure: client-b cannot resolve or reach web-a yet"
Expected behavior

This should fail because client-b is only on net-b and web-a is only on net-a. User-defined bridge networks isolate containers from other user-defined bridge networks unless you explicitly connect them.

Lab 6: Connect a running container to another network

Now attach the already-running client-b container to net-a.

docker network connect net-a client-b

docker network inspect net-a --format '{{range .Containers}}{{.Name}} {{end}}'

Retry the request from client-b:

docker exec client-b wget -qO- http://web-a | head
Validation

The request should now return the NGINX welcome HTML because client-b is connected to net-a and can resolve web-a using Docker's embedded DNS.

Why the earlier version failed

If client-b was started with docker run -it --rm ... sh, closing the shell removed the container. Then docker network connect net-a client-b correctly failed with No such container: client-b. A container must exist and be running before it can be connected to another network.

Lab 7: No network mode

docker run --rm --network none alpine:3.20 ip addr
Validation

You should see only loopback networking inside the container. This confirms network isolation.

Cleanup

docker rm -f web published-web web-a client-b 2>/dev/null || true
docker network rm app-net net-a net-b 2>/dev/null || true

11. Troubleshooting

ProblemLikely causeFix
Cannot connect to the Docker daemonDocker service not running or permission issue.Start Docker Desktop or run sudo systemctl start docker. On Linux, add user to docker group if allowed.
Port already allocatedAnother service is using the host port.Change host port, e.g. -p 5001:5000.
curl localhost failsWrong port mapping or app binding only to localhost inside container.Bind app to 0.0.0.0 and verify -p host:container.
Container DNS name not resolvingContainers are not on the same user-defined network.Use docker network inspect network-name and docker network connect.
Image rebuild is slowCache invalidated too early.Copy dependency files first, install dependencies, then copy source code.
Push deniedWrong Docker Hub namespace or not logged in.Use docker login and tag as username/repo:tag.
