1. Introduction to Docker Compose

Docker Compose lets you describe a complete application stack in a YAML file, then create, start, stop, inspect, and remove that stack with one command set.

Without Compose

You manually run many docker run commands, create networks, mount volumes, remember ports, and pass environment variables one by one.

With Compose

You write compose.yaml. Compose creates the project containers, network, service DNS, and named volumes consistently.

Best for training

Web + database, API + cache, local development stacks, monitoring demos, and repeatable classroom labs.

Command correction: current Compose uses docker compose. Older tutorials often show docker-compose. The old binary may exist, but the labs use v2.

2. Writing Docker Compose YAML files

The recommended filename is compose.yaml. Compose Specification style does not require the old top-level version: key.

Basic file shape

yaml
name: my-training-app
services:
  web:
    image: nginx:1.27-alpine
    ports:
      - "8080:80"
  redis:
    image: redis:7-alpine
volumes:
  redis-data:
networks:
  backend:

YAML rules trainees must remember

  • Indent with spaces, not tabs.
  • Quote port mappings like "8080:80".
  • Service names become DNS names on the Compose network.
  • Named volumes must be declared under top-level volumes:.
  • Run docker compose config before up.

3. Defining services, networks, and volumes

Services

A service is a container role such as web, api, db, or redis. It can use an existing image or build from a Dockerfile.

Networks

Compose creates a default network. Containers can talk to each other by service name, for example redis or db.

Volumes

Volumes store data outside the container lifecycle. Use them for databases and persistent application state.

4. Docker Compose CLI commands

Modern v2 command list

bash
docker compose version                 # Check Compose v2 plugin
docker compose config                  # Render and validate compose.yaml
docker compose up -d                   # Create/start services in background
docker compose up -d --build           # Rebuild local images, then start
docker compose ps                      # Show project containers
docker compose logs -f                 # Follow all service logs
docker compose logs web --tail=50      # Show last 50 logs from web service
docker compose exec web sh             # Open shell inside running web container
docker compose stop                    # Stop services, keep containers
docker compose start                   # Start stopped services
docker compose restart web             # Restart one service
docker compose pull                    # Pull service images
docker compose build                   # Build services that use build:
docker compose down                    # Remove containers and networks
docker compose down -v                 # Also remove named volumes

Old vs current command form

bash
# Preferred current syntax
docker compose up -d
docker compose down
docker compose ps

# Legacy standalone syntax, seen in older material
docker-compose up -d
docker-compose down
docker-compose ps

5. Environment variables in Docker Compose

Interpolation variables

Variables like ${APP_PORT} are resolved by Compose while reading compose.yaml. They can come from the shell or the project .env file.

Container environment

The environment: and env_file: keys inject variables into the running container process.

Do not confuse them: a variable in .env is not automatically inside the container unless you pass it with environment: or env_file:.

2 Lab 2 — Multi-container Flask + Redis app

Goal: run a small web app and Redis cache. The app connects to Redis using hostname redis.

bash
mkdir -p compose-labs/lab2-counter
cd compose-labs/lab2-counter

cat > Dockerfile <<'EOF'
FROM python:3.12-alpine
WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
COPY app.py .
EXPOSE 5000
CMD ["python", "app.py"]
EOF

cat > requirements.txt <<'EOF'
flask>=3,<4
redis>=5,<7
EOF

cat > app.py <<'APP_PY'
import os
from flask import Flask
from redis import Redis, RedisError

app = Flask(__name__)
redis_host = os.getenv("REDIS_HOST", "redis")
redis_port = int(os.getenv("REDIS_PORT", "6379"))
app_message = os.getenv("APP_MESSAGE", "Docker Compose Flask + Redis demo")
r = Redis(host=redis_host, port=redis_port, decode_responses=True)

@app.get("/")
def home():
    try:
        count = r.incr("hits")
        return (
            f"{app_message}<br>"
            f"Redis host: {redis_host}<br>"
            f"Page views: {count}\n"
        )
    except RedisError as exc:
        return f"Redis is not ready or unreachable: {exc}\n", 503

if __name__ == "__main__":
    app.run(host="0.0.0.0", port=5000)
APP_PY

cat > compose.yaml <<'YAML'
name: compose-counter-lab
services:
  web:
    build: .
    ports:
      - "5000:5000"
    environment:
      REDIS_HOST: redis
      REDIS_PORT: "6379"
      APP_MESSAGE: "Hello trainees - Compose connected Flask to Redis"
    depends_on:
      redis:
        condition: service_healthy
  redis:
    image: redis:7-alpine
    command: ["redis-server", "--appendonly", "yes"]
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
      interval: 5s
      timeout: 3s
      retries: 5
    volumes:
      - redis-data:/data
volumes:
  redis-data:
YAML

docker compose config
docker compose up -d --build
docker compose ps
curl http://localhost:5000
curl http://localhost:5000
docker compose exec redis redis-cli GET hits
docker compose down

Expected verification: refreshing http://localhost:5000 increases the page view count. docker compose exec redis redis-cli GET hits returns the stored counter value.

Why this works: Compose creates an internal DNS name for each service. The Flask container reaches Redis using hostname redis, not localhost. The Redis healthcheck avoids the common classroom issue where the web app starts before Redis is ready.

3 Lab 3 — PostgreSQL + Adminer with networks and volume

Goal: demonstrate database persistence and frontend/backend networks.

bash
mkdir -p compose-labs/lab3-db
cd compose-labs/lab3-db
cat > compose.yaml <<'YAML'
name: compose-db-lab
services:
  db:
    image: postgres:16-alpine
    environment:
      POSTGRES_USER: training
      POSTGRES_PASSWORD: trainingpass
      POSTGRES_DB: trainingdb
    volumes:
      - pgdata:/var/lib/postgresql/data
    networks:
      - backend
  adminer:
    image: adminer:4
    ports:
      - "8081:8080"
    networks:
      - frontend
      - backend
    depends_on:
      - db
networks:
  frontend:
  backend:
volumes:
  pgdata:
YAML

docker compose config
docker compose up -d
docker compose ps
docker compose exec db pg_isready -U training -d trainingdb
docker compose exec db psql -U training -d trainingdb -c "CREATE TABLE IF NOT EXISTS trainees (id serial PRIMARY KEY, name text);"
docker compose exec db psql -U training -d trainingdb -c "INSERT INTO trainees (name) VALUES ('Asha');"
docker compose exec db psql -U training -d trainingdb -c "SELECT * FROM trainees;"

# Volume persistence test: this removes containers but keeps pgdata.
docker compose down
docker compose up -d
docker compose exec db psql -U training -d trainingdb -c "SELECT * FROM trainees;"

# Final cleanup including the named volume.
docker compose down -v

Browser verification: open http://localhost:8081. Use system PostgreSQL, server db, username training, password trainingpass, database trainingdb.

Persistence: plain docker compose down keeps the named volume; docker compose down -v removes it.

4 Lab 4 — Environment variables with .env and env_file

Goal: see the difference between Compose interpolation and variables injected into a container.

bash
mkdir -p compose-labs/lab4-env
cd compose-labs/lab4-env

cat > .env <<'EOF'
APP_MESSAGE=Hello from the project .env file
APP_MODE=training
EOF

cat > app.env <<'EOF'
TRAINING_BATCH=batch-01
EOF

cat > compose.yaml <<'YAML'
name: compose-env-lab
services:
  app:
    image: alpine:3.20
    command: sh -c 'echo "APP_MESSAGE=$$APP_MESSAGE"; echo "APP_MODE=$$APP_MODE"; echo "TRAINING_BATCH=$$TRAINING_BATCH"; sleep 3600'
    environment:
      APP_MESSAGE: "${APP_MESSAGE}"
      APP_MODE: "${APP_MODE:-dev}"
    env_file:
      - app.env
YAML

docker compose config
docker compose up -d
docker compose logs app
docker compose exec app printenv | grep -E 'APP_MESSAGE|APP_MODE|TRAINING_BATCH'
docker compose down

Expected verification: logs show APP_MESSAGE, APP_MODE, and TRAINING_BATCH. The double dollar signs $$ prevent Compose from substituting variables too early.

8. Troubleshooting quick reference

Error / SymptomLikely causeFix
permission denied on Docker socketUser cannot access Docker daemon.sudo usermod -aG docker $USER, then log out and back in.
docker compose: command not foundCompose v2 plugin missing.Install Docker Compose plugin and verify with docker compose version.
Port already allocatedHost port is already used.Change the left side of the mapping, for example "8099:80".
App cannot reach Redis/DBWrong hostname or network.Use the Compose service name, such as redis or db. Check logs and docker compose ps.
Data disappearsVolume removed with down -v.Use plain docker compose down when you need to keep data.
Environment variable is blankVariable was used for interpolation but not injected into the container.Use environment: or env_file:.
