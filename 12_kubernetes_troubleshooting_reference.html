1. Troubleshooting Overview

Kubernetes troubleshooting should be systematic. Do not randomly delete objects. First identify the failing layer: application, service/networking, node, control plane, storage, or external dependency.

1.1 Troubleshooting mindset 1.2 First commands to run 2. Control plane failures 3. Node failure 4. Application failure 5. Component manifests and errors 6. Scenarios and solutions 7. Lab: Troubleshooting in Kubernetes 8. Assessment questions 9. Command cheat sheet

1.1 Troubleshooting Mindset

1Observe

Check symptoms: Pending pods, NotReady nodes, failing probes, missing endpoints, API server unreachable, or controller errors.

2Scope

Decide whether it affects one pod, one namespace, one node, one service, or the whole cluster.

3Verify

After fixing, validate with events, logs, endpoints, rollouts, pod status, and an actual application request.

1.2 First Commands to Run

Run these commands before changing anything. They quickly identify whether the issue is at the cluster, node, workload, or service layer.

Initial cluster triage 
kubectl cluster-info
kubectl get nodes -o wide
kubectl get pods -A -o wide
kubectl get events -A --sort-by=.metadata.creationTimestamp | tail -40
kubectl get componentstatuses 2>/dev/null || true
kubectl version --short 2>/dev/null || kubectl version
Note: kubectl get componentstatuses is deprecated in many clusters. Prefer checking the health of control plane pods, static pod manifests, kubelet logs, and API server health endpoints where available.

2. Control Plane Failures

Control plane failures affect scheduling, desired-state reconciliation, API access, service discovery, or cluster-wide operations. In kubeadm-based clusters, kube-apiserver, kube-controller-manager, kube-scheduler, and etcd commonly run as static pods managed by the kubelet.

Animated Concept: Control Plane Flow

2.1 Common Symptoms

SymptomLikely AreaFirst Checks
API kubectl times out or connection refusedAPI server, load balancer, control plane node, certificate, firewallkubectl cluster-info, control plane node health, journalctl -u kubelet, API server static pod logs
New pods remain Pending but existing pods runScheduler, taints, resource pressure, PVC, affinitykubectl describe pod, scheduler pod, events, node resources
Deployments do not create or replace podsController manager, API access, admission, quotasController-manager logs, deployment events, replica set status
DNS lookup fails inside podsCoreDNS, kube-proxy, CNI, service networkingkubectl get pods -n kube-system -l k8s-app=kube-dns, CoreDNS logs, service endpoints

2.2 Control Plane Diagnostic Commands

Control plane checks for kubeadm/static-pod clusters 
# Control plane pods
kubectl get pods -n kube-system -o wide
kubectl get pods -n kube-system -l tier=control-plane

# Static pod manifest directory on kubeadm control-plane node
sudo ls -l /etc/kubernetes/manifests
sudo grep -R "image:\|command:\|--advertise-address\|--etcd-servers" /etc/kubernetes/manifests

# Kubelet and container runtime checks
sudo systemctl status kubelet --no-pager
sudo journalctl -u kubelet -n 120 --no-pager
sudo systemctl status containerd --no-pager
sudo crictl ps -a | grep -E "kube-apiserver|kube-scheduler|kube-controller|etcd"
sudo crictl logs <CONTAINER_ID>
Production caution: Do not edit or delete files under /etc/kubernetes/manifests directly on production clusters unless you have a backup, change approval, and recovery access. A bad static pod manifest can take down the component.

3. Node Failure

Node failures usually appear as NotReady, Unknown, scheduling failures, pod evictions, image pull failures, CNI issues, disk pressure, memory pressure, or kubelet/runtime errors.

Animated Concept: Node Heartbeat and Failure

3.1 Node Status Conditions

ConditionMeaningUseful Checks
Ready=False / UnknownKubelet is unhealthy, stopped, unreachable, or cannot update node status.systemctl status kubelet, journalctl -u kubelet, network, certificates
MemoryPressureNode memory is low; kubelet may evict pods.free -h, kubectl top node, pod requests/limits
DiskPressureDisk or inode pressure; image garbage collection or evictions may happen.df -h, df -i, container runtime storage

Node troubleshooting commands

kubectl get pods -A -o wide --field-selector spec.nodeName=<node-name>
kubectl get events -A --field-selector involvedObject.kind=Node --sort-by=.metadata.creationTimestamp

# On the node itself
sudo systemctl status kubelet --no-pager
sudo journalctl -u kubelet -n 150 --no-pager

sudo crictl info
sudo crictl ps -a
sudo crictl images
sudo df -h
sudo df -i
free -h
ip addr
ip route

3.3 Safe Node Maintenance Workflow

Cordon, drain, fix, uncordon 
kubectl cordon <node-name>
kubectl drain <node-name> --ignore-daemonsets --delete-emptydir-data

# Fix node issue: kubelet, runtime, disk, CNI, package, certificate, OS patch, etc.

kubectl uncordon <node-name>
kubectl get nodes
kubectl get pods -A -o wide

4. Application Failure

Most Kubernetes incidents start at the workload layer. A correct troubleshooting path checks object status, events, container logs, probes, service selectors, endpoints, DNS, configuration, resource limits, and node placement.

5. Component Manifests and Errors

In kubeadm clusters, control plane components are commonly static pods. The kubelet watches manifest files and creates mirror pods visible through the Kubernetes API. Manifest syntax, wrong flags, bad certificates, missing files, or invalid image tags can break a component.

Animated Concept: Static Pod Manifest to Running Component

5.1 Static Manifest Troubleshooting Checklist

  • Check whether the expected manifest exists under /etc/kubernetes/manifests.
  • Validate YAML indentation and fields before saving changes.
  • Check kubelet logs because kubelet reads static pod manifests.
  • Check the container runtime logs with crictl when the API server is unavailable.
  • Confirm certificates, mounted host paths, command flags, and image tags.
  • Back up the original manifest before making any change.
Safe static manifest inspection 
sudo cp -a /etc/kubernetes/manifests /etc/kubernetes/manifests.backup.$(date +%F-%H%M%S)
sudo ls -lh /etc/kubernetes/manifests
sudo sed -n '1,220p' /etc/kubernetes/manifests/kube-apiserver.yaml
sudo journalctl -u kubelet -n 150 --no-pager
sudo crictl ps -a | grep kube-apiserver
sudo crictl logs <APISERVER_CONTAINER_ID>

7. Lab: Troubleshooting in Kubernetes

These labs use a disposable kind cluster. Do not run destructive control plane simulations on production clusters. For kubeadm clusters, use only the diagnostic commands unless the instructor explicitly provides a lab VM.

Recommended lab environment: Linux VM with Docker, kind, and kubectl. The examples also work on minikube with small adjustments, but node/container failure simulations are easiest with kind.

Lab Setup: Create a Multi-node kind Cluster

15 minutes

This creates one control-plane node and two worker nodes. The cluster name is trouble.

kind-trouble.yaml 
cat > kind-trouble.yaml <<'EOF'
kind: Cluster
apiVersion: kind.x-k8s.io/v1alpha4
nodes:
  - role: control-plane
  - role: worker
  - role: worker
