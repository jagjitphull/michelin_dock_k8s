Kubernetes: Managing Pod Lifecycle

A trainee-friendly HTML lab reference covering Pod creation, inspection, lifecycle, labels, annotations, multi-container patterns, init containers, static Pods, and practical kubectl exercises.

Recommended lab setup: Kubernetes cluster using Minikube, kind, kubeadm, or a managed cluster. Trainees need kubectl configured and working. Verify with kubectl cluster-info and kubectl get nodes.

1. Pod Overview

A Pod is the smallest deployable compute object in Kubernetes. A Pod can contain one container or multiple tightly coupled containers that share the same Pod network namespace and can share volumes.

User
kubectl
API Server
Scheduler
Node / Kubelet
Pod Running

Single-container Pod

nginx container
Shared Pod IP + localhost namespace

Most beginner examples use this model. It is simple and enough for many labs.

Multi-container Pod

main app
helper sidecar
Shared volume + network

Use only when containers must be co-located and managed as one unit.

Trainer note: Do not teach trainees to put unrelated applications in the same Pod. Separate lifecycle usually means separate Pods and a Service between them.

3. Creating a Pod with Imperative Syntax

Imperative commands are useful for quick tests, demos, and troubleshooting. They are not ideal as the final source of truth for production workloads.

Lab 2: Create Pods using imperative commands

Pod
  1. Create an nginx Pod. 
    kubectl run web-imperative --image=nginx:1.27 --port=80
  2. Check the Pod status. 
    kubectl get pods
kubectl get pod web-imperative -o wide
  3. Describe the Pod. 
    kubectl describe pod web-imperative
  4. Generate a YAML manifest without creating the object. This is a good bridge from imperative to declarative work. 
    kubectl run web-dryrun --image=nginx:1.27 --port=80 \
  --dry-run=client -o yaml > pod-web-dryrun.yaml

4. Creating a Pod with Declarative Syntax

Declarative syntax means you write the desired state in YAML and apply it. This is the preferred way for repeatable labs and real-world infrastructure.

Lab 3: Create a Pod using declarative YAML

YAML
  1. Create a file named pod-web.yaml. 
    cat <<'YAML' > pod-web.yaml
apiVersion: v1
kind: Pod
metadata:
  name: web-declarative
  labels:
    app: web
    env: dev
spec:
  containers:
  - name: nginx
    image: nginx:1.27
    ports:
    - containerPort: 80
YAML
  2. Apply the manifest. 
    kubectl apply -f pod-web.yaml
  3. Verify the Pod. 
    kubectl get pod web-declarative -o wide
Simple use case: A developer wants to test a container image quickly before creating a Deployment. A standalone Pod is acceptable for the lab. For production, use a controller such as Deployment, StatefulSet, DaemonSet, or Job.

5. Reading Pod Information, Metadata, JSON/YAML, and Backup

Read basic info

kubectl get pods
kubectl get pod web-declarative -o wide
kubectl describe pod web-declarative

Read metadata

kubectl get pod web-declarative \
  -o jsonpath='{.metadata.name}{"\n"}{.metadata.labels}{"\n"}'

Listing objects in JSON or YAML

kubectl get pod web-declarative -o yaml
kubectl get pod web-declarative -o json
kubectl get pods -o yaml
kubectl get pods -o json

Backing up resources using list operation

mkdir -p backup
kubectl get pods -o yaml > backup/pods-backup.yaml
kubectl get pod web-declarative -o yaml > backup/web-declarative-pod.yaml
Important: A raw backup from kubectl get -o yaml includes runtime/status fields. For clean reusable YAML, remove fields such as status, resourceVersion, uid, creationTimestamp, and managedFields.

Getting more information from the list operation

kubectl get pods -o wide
kubectl get pods --show-labels
kubectl get pods --sort-by=.metadata.name
kubectl get pods -L app,env
kubectl get pods -o custom-columns=NAME:.metadata.name,PHASE:.status.phase,NODE:.spec.nodeName,IP:.status.podIP

6. Accessing a Pod from the Outside World

A Pod IP is usually reachable only inside the cluster network. For labs, use port-forward. For cluster-level exposure, create a Service.

Browser
localhost:8080
kubectl port-forward
Pod :80

Option A: Port-forward

kubectl port-forward pod/web-declarative 8080:80

Open http://localhost:8080 from your browser or run:

curl http://localhost:8080

Option B: NodePort Service

kubectl expose pod web-declarative --type=NodePort --port=80 --target-port=80 --name=web-svc
kubectl get svc web-svc

For Minikube:

minikube service web-svc -n pod-lifecycle-lab

7. Entering a Container and Creating Temporary Pods

Entering into a container inside a Pod

kubectl exec -it web-declarative -- /bin/sh
# inside the container
hostname
exit

Running commands in containers

kubectl exec web-declarative -- nginx -v
kubectl logs web-declarative

Creating a temporary Pod

kubectl run tmp-shell --rm -it --image=busybox:1.36 --restart=Never -- sh
# inside temporary Pod
wget -qO- http://web-svc
exit
Use case: Temporary Pods are useful for DNS checks, connectivity checks, curl/wget tests, and short-lived troubleshooting without permanently adding tools into application containers.

8. Multi-container Pod

Concrete scenarios where you need multi-container Pods

Sidecar

Helper container enhances the main container, for example log shipping, file sync, proxy, or hot reload helper.

Adapter

Transforms application output into a standard format, for example converting custom metrics into Prometheus-friendly output.

Ambassador

Acts as a local proxy for remote service access, so the app talks to localhost while the ambassador handles routing.

When not to create a multi-container Pod

SituationBetter designReason
Frontend and databaseSeparate Pods + ServiceDifferent lifecycle, scaling, storage, and security needs.
Two unrelated appsSeparate workloadsA crash or restart should not bind their fate together.
Independent scaling requirementDeployment per componentKubernetes scales Pods, not individual containers inside a Pod.
Separate ownership teamsSeparate servicesDeployment, rollback, and access control become cleaner.

What happens when Kubernetes fails to launch one container in a Pod?

If one container cannot start, the Pod usually remains unhealthy. Depending on restart policy and failure type, you may see states such as ImagePullBackOff, CrashLoopBackOff, or a Pod that is Running but not Ready. The healthy container may still run, but the Pod should not be treated as fully ready.

Lab 4: Multi-container Pod sharing a volume

cat <<'YAML' > pod-multi.yaml
apiVersion: v1
kind: Pod
metadata:
  name: multi-demo
  labels:
    app: multi-demo
spec:
  volumes:
  - name: shared-data
    emptyDir: {}
  containers:
  - name: writer
    image: busybox:1.36
    command: ["sh", "-c"]
    args:
    - while true; do date >> /data/time.txt; sleep 5; done
    volumeMounts:
    - name: shared-data
      mountPath: /data
  - name: reader
    image: busybox:1.36
    command: ["sh", "-c"]
    args:
    - while true; do echo '--- file content ---'; cat /data/time.txt 2>/dev/null || true; sleep 5; done
    volumeMounts:
    - name: shared-data
      mountPath: /data
YAML
kubectl apply -f pod-multi.yaml
kubectl get pod multi-demo
kubectl logs multi-demo -c reader

Accessing a specific container inside a multi-container Pod

kubectl exec -it multi-demo -c writer -- sh
kubectl exec -it multi-demo -c reader -- sh
kubectl logs multi-demo -c writer
kubectl logs multi-demo -c reader

Understanding the Pod deletion grace period

When a Pod is deleted, Kubernetes gives containers time to stop gracefully before force killing them. The default grace period is commonly 30 seconds unless changed in the Pod or delete command.

kubectl delete pod multi-demo --grace-period=10
# Force deletion only for stuck lab Pods, not normal production cleanup:
kubectl delete pod multi-demo --grace-period=0 --force

9. Overriding Default Commands Run by Containers

Container images define default command behavior, but Kubernetes lets you override it using command and args.

cat <<'YAML' > pod-command-override.yaml
apiVersion: v1
kind: Pod
metadata:
  name: command-demo
spec:
  restartPolicy: Never
  containers:
  - name: busybox
    image: busybox:1.36
    command: ["sh", "-c"]
    args: ["echo Starting custom command; for i in 1 2 3; do echo Loop-$i; sleep 2; done; echo Finished"]
YAML
kubectl apply -f pod-command-override.yaml
kubectl logs command-demo
FieldMeaningDocker comparison
commandOverrides executable/entrypointSimilar to Docker ENTRYPOINT override
argsArguments passed to commandSimilar to Docker CMD override

10. Labeling and Annotating Pods

What are labels and why do we need them?

Labels are key-value pairs used to organize, query, and select Kubernetes objects. Services, controllers, and CLI filters commonly use labels.

What are annotations and how do they differ from labels?

FeatureLabelsAnnotations
PurposeIdentification, grouping, filtering, selectionNon-identifying metadata
Used by selectors?YesNo
Examplesapp=web, env=dev, tier=frontendowner=training-team, description=demo pod
Best forAutomation and object selectionNotes, tooling hints, build info, URLs

Lab 5: Working with labels and annotations

Metadata
  1. Create a labeled Pod. 
    kubectl run label-demo --image=nginx:1.27 --labels=app=nginx,env=dev,tier=frontend
  2. List labels attached to a Pod. 
    kubectl get pod label-demo --show-labels
kubectl get pods -L app,env,tier
  3. Filter by labels. 
    kubectl get pods -l app=nginx
kubectl get pods -l 'env in (dev,test)'
kubectl get pods -l app=nginx,tier=frontend
  4. Add or update a label on a running Pod. 
    kubectl label pod label-demo version=v1
kubectl label pod label-demo env=test --overwrite
  5. Delete a label attached to a running Pod. 
    kubectl label pod label-demo tier-
  6. Add an annotation. 
    kubectl annotate pod label-demo owner='training-team'
kubectl annotate pod label-demo description='Pod lifecycle lab example' --overwrite
kubectl describe pod label-demo

11. Understanding the Pod Lifecycle

Pods follow a lifecycle from creation through scheduling, container startup, running, termination, and final phase.

Pending
Running
Succeeded
Failed
Unknown
Pod PhaseMeaningTypical trainee observation
PendingAccepted by cluster, but one or more containers are not yet running.Image pulling, scheduling delay, insufficient CPU/memory, PVC pending.
RunningPod is bound to a node and at least one container is running or starting/restarting.Use kubectl get pods, logs, and describe.
SucceededAll containers terminated successfully and will not restart.Common for one-time Pods with restartPolicy: Never.
FailedAt least one container terminated in failure and will not restart.Check exit code, events, and logs.
UnknownPod state could not be obtained, often due to node communication problems.Check node health and kubelet connectivity.

12. Pod Controllers

A standalone Pod is not self-healing if the node fails. Controllers create and manage Pods on your behalf.

Desired State
Controller
ReplicaSet / Job
Pods
ControllerUse caseExample
DeploymentStateless long-running appsWeb server, API service
ReplicaSetMaintains replica count; usually managed by DeploymentInternal part of Deployment workflow
StatefulSetStable identity and storageDatabases, clustered systems
DaemonSetOne Pod per node or selected nodesLog agent, node exporter, CNI agent

Training rule: Use standalone Pods to learn concepts. Use controllers for real deployment patterns.

13. Pod Patterns: Sidecar, Adapter, Ambassador

Sidecar Pattern

main applog sidecar

The sidecar supports the main application. Example: app writes logs to a shared volume, sidecar reads and ships them.

Adapter Pattern

app outputadapterstandard output

The adapter normalizes output. Example: transform custom metrics into a consistent HTTP metrics endpoint.

Ambassador Pattern

applocalhost proxyremote service

The ambassador hides remote access details. Example: app connects to localhost while the proxy routes to an external service.

Lab 7: Sidecar container Pod

cat <<'YAML' > pod-sidecar.yaml
apiVersion: v1
kind: Pod
metadata:
  name: sidecar-demo
spec:
  volumes:
  - name: logs
    emptyDir: {}
  containers:
  - name: app
    image: busybox:1.36
    command: ["sh", "-c"]
    args: ["while true; do echo app-log-$(date) >> /var/log/app/app.log; sleep 3; done"]
    volumeMounts:
    - name: logs
      mountPath: /var/log/app
  - name: log-reader
    image: busybox:1.36
    command: ["sh", "-c"]
    args: ["tail -n+1 -f /var/log/app/app.log"]
    volumeMounts:
    - name: logs
      mountPath: /var/log/app
YAML
kubectl apply -f pod-sidecar.yaml
kubectl logs sidecar-demo -c log-reader

Lab 8: Adapter container Pod

cat <<'YAML' > pod-adapter.yaml
apiVersion: v1
kind: Pod
metadata:
  name: adapter-demo
spec:
  volumes:
  - name: data
    emptyDir: {}
  containers:
  - name: app
    image: busybox:1.36
    command: ["sh", "-c"]
    args: ["while true; do echo temperature=$(date +%S) > /data/raw.txt; sleep 5; done"]
    volumeMounts:
    - name: data
      mountPath: /data
  - name: adapter
    image: busybox:1.36
    command: ["sh", "-c"]
    args: ["while true; do awk -F= '{print \"app_temperature_seconds \"$2}' /data/raw.txt 2>/dev/null; sleep 5; done"]
    volumeMounts:
    - name: data
      mountPath: /data
YAML
kubectl apply -f pod-adapter.yaml
kubectl logs adapter-demo -c adapter

14. Init Containers

Init containers run before the main application containers. Each init container must complete successfully before the next one starts. They are useful for setup, checks, permissions, migrations, and dependency waiting.

init-1
init-2
main app
Pod Ready

Lab 10: Init container Pod

Startup Control
  1. Create a Pod where init container prepares content for nginx. 
    cat <<'YAML' > pod-init.yaml
apiVersion: v1
kind: Pod
metadata:
  name: init-demo
spec:
  volumes:
  - name: web-content
    emptyDir: {}
  initContainers:
  - name: init-html
    image: busybox:1.36
    command: ["sh", "-c"]
    args: ["echo 'Hello from init container' > /work/index.html"]
    volumeMounts:
    - name: web-content
      mountPath: /work
  containers:
  - name: nginx
    image: nginx:1.27
    ports:
    - containerPort: 80
    volumeMounts:
    - name: web-content
      mountPath: /usr/share/nginx/html
YAML
kubectl apply -f pod-init.yaml
  2. Inspect init container status. 
    kubectl get pod init-demo
kubectl describe pod init-demo

15. Static Pods

Static Pods are managed directly by the kubelet on a specific node, not by the API server. The kubelet watches a local manifest directory, commonly /etc/kubernetes/manifests in kubeadm-based clusters. The API server may show mirror Pods for visibility.

Production caution: Do not casually edit static Pod manifests on production control-plane nodes. kubeadm control plane components are often static Pods. A wrong edit can break the cluster.

Lab 11: Understanding Static Pods

Node-level
  1. SSH to a lab node where you have sudo access. Check kubelet static Pod path. 
    sudo grep -R "staticPodPath" /var/lib/kubelet/config.yaml /etc/kubernetes/kubelet.conf 2>/dev/null || true
  2. Create a static Pod manifest only on a safe lab node. 
    sudo tee /etc/kubernetes/manifests/static-web.yaml >/dev/null <<'YAML'
apiVersion: v1
kind: Pod
metadata:
  name: static-web
  labels:
    app: static-web
spec:
  containers:
  - name: nginx
    image: nginx:1.27
    ports:
    - containerPort: 80
YAML
  3. From your kubectl machine, check for the mirror Pod. Its name usually includes the node name. 
    kubectl get pods -A | grep static-web
  4. Remove the static Pod by deleting the manifest from the node. 
    sudo rm -f /etc/kubernetes/manifests/static-web.yaml
