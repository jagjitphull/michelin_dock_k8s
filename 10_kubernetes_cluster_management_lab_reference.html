How to use this document

  • Use the concept sections before labs so trainees understand the control-plane behavior.
  • Use kind for the safe Node Maintenance lab.
  • Use a disposable kubeadm VM cluster for upgrade and etcd backup/restore labs. Do not run those labs on a production cluster.
  • Use K9s only as an optional visual tool. Every K9s task has a kubectl alternative.
Production warning: Kubernetes upgrade and etcd restore are destructive administrative operations if performed incorrectly. The commands here are for controlled training clusters. Always take backups, confirm version skew, CNI compatibility, and maintenance windows before production work.

Concept

2. Cluster Management

Kubernetes cluster management means continuously checking whether the cluster is healthy, whether nodes are available, whether workloads are running as desired, and whether cluster state is safely stored in etcd.

The control plane does not manually “push” every container. It stores desired state in the API server, controllers compare desired state with actual state, and node agents report health and run containers.

AreaWhat admin checksTypical command
Cluster healthNodes Ready, control plane pods runningkubectl get nodes, kubectl -n kube-system get pods
Workload healthDeployments available, pods not crashingkubectl get deploy,pods -A
SchedulingPods have nodes assigned, no unschedulable nodeskubectl describe pod
Cluster stateetcd healthy and backed upetcdctl endpoint health, etcdctl snapshot save
Daily cluster management checks
kubectl cluster-info
kubectl get nodes -o wide
kubectl get componentstatuses 2>/dev/null || true   # deprecated on many clusters; use pod checks below
kubectl -n kube-system get pods -o wide
kubectl get events -A --sort-by=.lastTimestamp | tail -30

3. Node Status

A Node is a Kubernetes API object representing a worker or control-plane machine. Its status includes addresses, capacity, allocatable resources, node conditions, and system information.

ConditionMeaningAdmin action
ReadyKubelet is healthy and posting status.Check Ready=True.
MemoryPressureNode memory is low.Check pod requests/limits, evictions.
DiskPressureNode disk is low.Clean images/logs, expand disk.
PIDPressureToo many processes.Investigate runaway pods/processes.
NetworkUnavailableNetwork route/plugin problem.Check CNI pods and routes.
Node status inspection
  kubectl get nodes
  kubectl describe node <node-name>
  kubectl get node <node-name> -o jsonpath='{.status.conditions[*].type}' ; echo
  kubectl get node <node-name> -o jsonpath='{range .status.conditions[*]}{.type}{"="}{.status}{"
"}{end}'
  kubectl top nodes   # requires Metrics Server
Concept

4. Node Heartbeats

Heartbeats are how nodes tell the control plane that they are alive. Kubernetes uses two heartbeat forms: updates to the Node .status and lightweight Lease objects in the kube-node-lease namespace.

The kubelet creates and updates Node status and the related Lease object. Lease-based heartbeats reduce API-server load in large clusters.

Inspect node lease heartbeats
kubectl get leases -n kube-node-lease
kubectl get lease <node-name> -n kube-node-lease -o yaml
kubectl get node <node-name> -o yaml | less
Concept

5. Node Controller

The Node Controller is part of the controller-manager. It watches node heartbeats and node status. If heartbeats stop, Kubernetes can mark the node condition as Unknown or NotReady, stop scheduling new pods there, and eventually trigger pod eviction behavior depending on configuration and workload rules.

Important: Existing containers may keep running on an unreachable node, but the control plane cannot reliably manage them until node communication returns.
Check node-controller symptoms
kubectl get nodes
kubectl describe node <node-name> | sed -n '/Conditions:/,/Addresses:/p'
kubectl get events -A --field-selector involvedObject.kind=Node --sort-by=.lastTimestamp

6. Node Maintenance

Node maintenance is the controlled process of removing a node from scheduling, evicting movable workloads, performing maintenance, and returning the node to service.

  1. Cordon: mark node unschedulable for new pods.
  2. Drain: evict application pods safely.
  3. Maintain: patch OS, reboot, fix hardware, or upgrade kubelet.
  4. Uncordon: allow scheduling again.

Lab: Node Maintenance using kind

Step 1: Create sample workload

Create workload
kubectl create namespace k8s-mgmt
kubectl -n k8s-mgmt create deployment web-demo --image=nginx:1.27-alpine --replicas=6
kubectl -n k8s-mgmt expose deployment web-demo --port=80 --target-port=80
kubectl -n k8s-mgmt rollout status deployment/web-demo
kubectl -n k8s-mgmt get pods -o wide

Step 2: Select a worker node

Choose node
kubectl get nodes
export NODE_TO_MAINTAIN=k8s-mgmt-lab-worker
kubectl describe node $NODE_TO_MAINTAIN | grep -i "Unschedulable" || true

Step 3: Cordon the node

Cordon
kubectl cordon $NODE_TO_MAINTAIN
kubectl get nodes
kubectl describe node $NODE_TO_MAINTAIN | grep -i "Unschedulable"

Expected: the node status may show Ready,SchedulingDisabled.

Step 4: Drain the node

Drain
kubectl drain $NODE_TO_MAINTAIN --ignore-daemonsets --delete-emptydir-data
kubectl -n k8s-mgmt get pods -o wide
--ignore-daemonsets is normally required because DaemonSet-managed pods are not drained like ordinary workload pods. --delete-emptydir-data allows deletion of pods using temporary emptyDir data; do not use it blindly for important data.

Step 5: Simulate maintenance

Simulate node restart
# With kind, the node is a Docker container. This simulates node maintenance/restart.
docker restart $NODE_TO_MAINTAIN

# Wait and verify the node returns Ready
kubectl get nodes -w

Step 6: Uncordon the node

Return node to service
kubectl uncordon $NODE_TO_MAINTAIN
kubectl get nodes
kubectl -n k8s-mgmt scale deployment web-demo --replicas=8
kubectl -n k8s-mgmt get pods -o wide

Cleanup

Cleanup
kubectl delete namespace k8s-mgmt
# Optional: delete the complete kind lab cluster
# kind delete cluster --name k8s-mgmt-lab

7. Upgrading the Kubernetes Cluster

Kubernetes cluster upgrades depend on how the cluster was built. For kubeadm clusters, the safe sequence is: upgrade the primary control-plane node, upgrade additional control-plane nodes, then upgrade worker nodes.

Do not skip minor versions. Upgrade from 1.N to 1.N+1, then continue if another minor upgrade is required.

Disposable lab only: Upgrade commands below are for kubeadm-managed clusters, not kind/minikube managed clusters. Use a test cluster that you can rebuild.

Pre-upgrade checklist

  • Read release notes for the target version.
  • Confirm current version: kubectl get nodes.
  • Confirm all nodes are Ready.
  • Back up etcd before upgrade.
  • Check CNI plugin upgrade notes: Calico, Cilium, Flannel, etc.
  • Check deprecated/removed API usage.
Pre-upgrade health checks
kubectl get nodes -o wide
kubectl -n kube-system get pods -o wide
kubectl get --raw='/readyz?verbose'
kubectl get apiservices | grep False || true
kubectl get events -A --sort-by=.lastTimestamp | tail -30

Lab: Upgrade a kubeadm cluster on Ubuntu/Debian

Replace 1.36.x-* and v1.36.x with the exact patch version shown by your package manager and supported by your current cluster path. Example only: do not copy blindly.

Step 1: On the first control-plane node, find target version

Select target patch version
sudo apt update
sudo apt-cache madison kubeadm

# Example variable. Replace x with the patch version selected from apt-cache output.
export TARGET_DEB_VERSION='1.36.x-*'
export TARGET_K8S_VERSION='v1.36.x'

Step 2: Upgrade kubeadm and run upgrade plan

Upgrade kubeadm and plan
sudo apt-mark unhold kubeadm
sudo apt-get update
sudo apt-get install -y kubeadm="$TARGET_DEB_VERSION"
sudo apt-mark hold kubeadm

kubeadm version
sudo kubeadm upgrade plan

Step 3: Apply control-plane upgrade

Upgrade first control-plane node
sudo kubeadm upgrade apply "$TARGET_K8S_VERSION"

Step 4: Drain the control-plane node and upgrade kubelet/kubectl

Upgrade kubelet/kubectl on control-plane
kubectl get nodes -o wide
export NODE_TO_UPGRADE=<control-plane-node-name>
kubectl drain "$NODE_TO_UPGRADE" --ignore-daemonsets

sudo apt-mark unhold kubelet kubectl
sudo apt-get update
sudo apt-get install -y kubelet="$TARGET_DEB_VERSION" kubectl="$TARGET_DEB_VERSION"
sudo apt-mark hold kubelet kubectl

sudo systemctl daemon-reload
sudo systemctl restart kubelet
kubectl uncordon "$NODE_TO_UPGRADE"

Step 5: Upgrade additional control-plane nodes

Additional control-plane node
# Run on each additional control-plane node, one at a time.
sudo apt-mark unhold kubeadm
sudo apt-get update
sudo apt-get install -y kubeadm="$TARGET_DEB_VERSION"
sudo apt-mark hold kubeadm

sudo kubeadm upgrade node

kubectl get nodes -o wide
export NODE_TO_UPGRADE=<control-plane-node-name>
kubectl drain "$NODE_TO_UPGRADE" --ignore-daemonsets
sudo apt-mark unhold kubelet kubectl
sudo apt-get install -y kubelet="$TARGET_DEB_VERSION" kubectl="$TARGET_DEB_VERSION"
sudo apt-mark hold kubelet kubectl
sudo systemctl daemon-reload
sudo systemctl restart kubelet
kubectl uncordon "$NODE_TO_UPGRADE"

Step 6: Upgrade worker nodes

Worker node upgrade
# Run on each worker node. Drain from an admin terminal before package changes.
kubectl drain <worker-node-name> --ignore-daemonsets --delete-emptydir-data

# On the worker node:
sudo apt-mark unhold kubeadm
sudo apt-get update
sudo apt-get install -y kubeadm="$TARGET_DEB_VERSION"
sudo apt-mark hold kubeadm
sudo kubeadm upgrade node

sudo apt-mark unhold kubelet kubectl
sudo apt-get install -y kubelet="$TARGET_DEB_VERSION" kubectl="$TARGET_DEB_VERSION"
sudo apt-mark hold kubelet kubectl
sudo systemctl daemon-reload
sudo systemctl restart kubelet

# From admin terminal:
kubectl uncordon <worker-node-name>

Step 7: Validate upgrade

Post-upgrade validation
kubectl get nodes -o wide
kubectl -n kube-system get pods -o wide
kubectl get --raw='/readyz?verbose'
kubectl version

8. Kubernetes Cluster Backup and Restore

For a kubeadm cluster using stacked etcd, the most important cluster-level backup is the etcd snapshot. It contains Kubernetes API state: namespaces, deployments, services, secrets, configmaps, RBAC, and other objects.

It does not back up application data stored inside external databases, PersistentVolumes, object storage, or cloud disks. Those need separate backup processes.

Lab: Backup and restore ETCD on a disposable kubeadm cluster

Do not perform the restore section on a production cluster. Restoring etcd rolls the cluster state back to the snapshot. This can delete later API objects and confuse controllers if not handled correctly.

Step 1: Create a test object before backup

Create object to prove restore
kubectl create namespace restore-test
kubectl -n restore-test create configmap important-config --from-literal=state=before-backup
kubectl -n restore-test get configmap important-config -o yaml

Step 2: Confirm etcd pod and certificates on control-plane node

Inspect etcd static pod
sudo crictl ps | grep etcd
sudo ls -l /etc/kubernetes/pki/etcd/
sudo grep -n "data-dir\|listen-client-urls\|advertise-client-urls" /etc/kubernetes/manifests/etcd.yaml

Step 3: Take etcd snapshot

Backup etcd
export SNAPSHOT=/root/etcd-snapshot-$(date +%F-%H%M).db

sudo env ETCDCTL_API=3 etcdctl     --endpoints=https://127.0.0.1:2379     --cacert=/etc/kubernetes/pki/etcd/ca.crt     --cert=/etc/kubernetes/pki/etcd/server.crt     --key=/etc/kubernetes/pki/etcd/server.key     snapshot save "$SNAPSHOT"

sudo etcdutl snapshot status "$SNAPSHOT" -w table

Step 4: Simulate accidental deletion

Delete test object
kubectl delete namespace restore-test
kubectl get namespace restore-test || true

Step 5: Restore from snapshot on single-node kubeadm control plane

This simplified restore lab assumes a single stacked-etcd control-plane. For multi-control-plane etcd, restore all etcd members consistently and stop all API servers first.
Restore etcd snapshot
# 1) Stop API server and etcd static pods by moving their manifests away.
sudo mkdir -p /root/k8s-static-pod-manifests-backup
sudo mv /etc/kubernetes/manifests/kube-apiserver.yaml /root/k8s-static-pod-manifests-backup/
sudo mv /etc/kubernetes/manifests/etcd.yaml /root/k8s-static-pod-manifests-backup/

# 2) Wait until both containers disappear.
watch -n 2 'sudo crictl ps | egrep "kube-apiserver|etcd" || true'
# Press Ctrl+C after both kube-apiserver and etcd containers have disappeared.

# 3) Move old etcd data aside and restore snapshot to /var/lib/etcd.
sudo mv /var/lib/etcd /var/lib/etcd.bak-$(date +%F-%H%M)
sudo etcdutl snapshot restore "$SNAPSHOT" --data-dir /var/lib/etcd
sudo chown -R root:root /var/lib/etcd

# 4) Bring etcd back, then API server.
sudo mv /root/k8s-static-pod-manifests-backup/etcd.yaml /etc/kubernetes/manifests/
sleep 30
sudo mv /root/k8s-static-pod-manifests-backup/kube-apiserver.yaml /etc/kubernetes/manifests/

Step 6: Validate restore

Verify restored object
# Wait for API server to return
kubectl get nodes
kubectl get namespace restore-test
kubectl -n restore-test get configmap important-config -o yaml
kubectl -n kube-system get pods

9. Working with ETCD datastore

etcd is the consistent key-value store used as Kubernetes backing storage. Admins normally interact through the Kubernetes API, not by editing etcd keys directly.

Never manually edit etcd keys in a live cluster unless you are performing a documented disaster-recovery operation and fully understand the consequences.

Useful etcd read-only commands

etcd health and key inspection
# Run on kubeadm control-plane node.
sudo env ETCDCTL_API=3 etcdctl     --endpoints=https://127.0.0.1:2379     --cacert=/etc/kubernetes/pki/etcd/ca.crt     --cert=/etc/kubernetes/pki/etcd/server.crt     --key=/etc/kubernetes/pki/etcd/server.key     endpoint health -w table

sudo env ETCDCTL_API=3 etcdctl     --endpoints=https://127.0.0.1:2379     --cacert=/etc/kubernetes/pki/etcd/ca.crt     --cert=/etc/kubernetes/pki/etcd/server.crt     --key=/etc/kubernetes/pki/etcd/server.key     endpoint status -w table

sudo env ETCDCTL_API=3 etcdctl     --endpoints=https://127.0.0.1:2379     --cacert=/etc/kubernetes/pki/etcd/ca.crt     --cert=/etc/kubernetes/pki/etcd/server.crt     --key=/etc/kubernetes/pki/etcd/server.key     get /registry/namespaces --prefix --keys-only | head

10. Create a K8S component through a YAML descriptor file

Declarative YAML descriptors define the desired state of Kubernetes objects. The recommended trainee workflow is: write YAML, validate with dry-run, apply, verify, update, and clean up.

Descriptor files

namespace.yaml
apiVersion: v1
kind: Namespace
metadata:
  name: k8s-mgmt
deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web-demo
  namespace: k8s-mgmt
  labels:
    app: web-demo
spec:
  replicas: 3
  selector:
    matchLabels:
      app: web-demo
  template:
    metadata:
      labels:
        app: web-demo
    spec:
      containers:
        - name: nginx
          image: nginx:1.27-alpine
          ports:
            - containerPort: 80
          readinessProbe:
            httpGet:
              path: /
              port: 80
            initialDelaySeconds: 5
            periodSeconds: 5
          resources:
            requests:
              cpu: 50m
              memory: 64Mi
            limits:
              cpu: 200m
              memory: 128Mi
service.yaml
apiVersion: v1
kind: Service
metadata:
  name: web-demo-svc
  namespace: k8s-mgmt
spec:
  type: ClusterIP
  selector:
    app: web-demo
  ports:
    - name: http
      port: 80
      targetPort: 80
configmap.yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: app-settings
  namespace: k8s-mgmt
data:
  APP_MODE: training
  LOG_LEVEL: info
pdb.yaml
apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: web-demo-pdb
  namespace: k8s-mgmt
spec:
  minAvailable: 2
  selector:
    matchLabels:
      app: web-demo

Apply and verify

Apply YAML descriptors
kubectl apply --dry-run=server -f namespace.yaml
kubectl apply -f namespace.yaml
kubectl apply --dry-run=server -f deployment.yaml
kubectl apply -f deployment.yaml
kubectl apply -f service.yaml
kubectl apply -f configmap.yaml
kubectl apply -f pdb.yaml

kubectl -n k8s-mgmt get deploy,rs,pods,svc,cm,pdb -o wide
kubectl -n k8s-mgmt rollout status deployment/web-demo
kubectl -n k8s-mgmt describe deployment web-demo

Access the application

Port-forward test
kubectl -n k8s-mgmt port-forward service/web-demo-svc 8080:80
# Open another terminal:
curl http://127.0.0.1:8080

Cleanup

Cleanup descriptors
kubectl delete -f pdb.yaml
kubectl delete -f configmap.yaml
kubectl delete -f service.yaml
kubectl delete -f deployment.yaml
kubectl delete -f namespace.yaml

11. Kustomize deployment

Kustomize lets trainees keep reusable base manifests and apply environment-specific overlays without template syntax. kubectl includes Kustomize support through kubectl kustomize and kubectl apply -k.

Directory structure

Kustomize layout
kustomize-web-demo/
├── base/
│   ├── deployment.yaml
│   ├── service.yaml
│   └── kustomization.yaml
└── overlays/
    └── dev/
        ├── namespace.yaml
        └── kustomization.yaml

Create files

Create folders
mkdir -p kustomize-web-demo/base kustomize-web-demo/overlays/dev
cd kustomize-web-demo

      app: web-demo

      labels:
        app: web-demo
    spec:
      containers:
        - name: nginx
          image: nginx:1.27-alpine
          ports:
            - containerPort: 80
          readinessProbe:
            httpGet:
              path: /
              port: 80
            initialDelaySeconds: 5
            periodSeconds: 5
          resources:
            requests:
              cpu: 50m
              memory: 64Mi
            limits:
              cpu: 200m
              memory: 128Mi
base/service.yaml
apiVersion: v1
kind: Service
metadata:
  name: web-demo-svc

  type: ClusterIP
  selector:
    app: web-demo

    - name: http

base/kustomization.yaml
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization
resources:
  - deployment.yaml
  - service.yaml
commonLabels:
  training: cluster-management
overlays/dev/namespace.yaml
apiVersion: v1
kind: Namespace
metadata:
  name: k8s-mgmt-dev
overlays/dev/kustomization.yaml
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization
namespace: k8s-mgmt-dev
resources:
  - namespace.yaml
  - ../../base
patches:
  - target:
      kind: Deployment
      name: web-demo
    patch: |-
      - op: replace
        path: /spec/replicas
        value: 2
      - op: replace
        path: /spec/template/spec/containers/0/image
        value: nginx:1.27-alpine

Build, apply, verify

Deploy with Kustomize
kubectl kustomize overlays/dev
kubectl apply --dry-run=client -k overlays/dev
kubectl apply -k overlays/dev

kubectl -n k8s-mgmt-dev get deploy,rs,pods,svc -o wide
kubectl -n k8s-mgmt-dev rollout status deployment/web-demo

Cleanup

Cleanup Kustomize lab
kubectl delete -k overlays/dev
cd ..
rm -rf kustomize-web-demo
Concept

Deployment: desired state, ReplicaSet, Pods

A Deployment manages ReplicaSets and Pods for stateless applications. You declare replicas, image, labels, probes, and resource requests. The Deployment controller moves the actual state toward the desired state and supports rollout and rollback.

Deployment rollout commands
kubectl -n k8s-mgmt rollout history deployment/web-demo
kubectl -n k8s-mgmt set image deployment/web-demo nginx=nginx:1.27-alpine
kubectl -n k8s-mgmt rollout status deployment/web-demo
kubectl -n k8s-mgmt rollout undo deployment/web-demo
