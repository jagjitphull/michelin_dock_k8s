1. What is Containerization?

Containerization packages an application with its dependencies so it can run consistently across laptop, VM, test, and production environments.

Simple picture: application packaging before and after containers

Without containers

Application may work on a developer laptop but fail on a server due to different library versions or OS settings.

With containers

The image contains the application and dependencies. The runtime starts the container from the same image everywhere.

Docker basics: image to running container

Container vs virtual machine

Docker review: Docker is commonly used to build and push container images. Kubernetes nodes do not require Docker Engine as the runtime; modern clusters commonly use CRI-compatible runtimes such as containerd or CRI-O.

2. Why Kubernetes?

Self-healing

Restarts failed containers and reschedules Pods when nodes fail.

Scaling

Runs multiple replicas and scales them up or down.

Service discovery

Applications use stable Service names instead of changing Pod IPs.

Rolling updates

Deploys new versions gradually and supports rollback.

Declarative operation

You define desired state in YAML; Kubernetes continuously reconciles reality.

5. Kubernetes Cluster Architecture and Directional Flow

The diagram below shows what happens when a trainee runs kubectl create deployment web --image=nginx.

StepWhat happens
1kubectl sends the desired state to kube-apiserver.
2API server validates the request and records state in etcd.
3Scheduler assigns the pending Pod to a suitable worker node.
4Kubelet on the worker sees the assigned Pod.
5Kubelet uses CRI/containerd to pull image and start containers.
6CNI configures Pod networking; kube-proxy handles Service routing.
7Application runs as a Pod and can be exposed through a Service.

6. Understanding Kubernetes Control Plane Components

kube-apiserver

Front door of Kubernetes. All users, nodes, and controllers communicate through it.

etcd

Distributed key-value database storing cluster state. Back it up before risky changes.

kube-scheduler

Selects nodes for new Pods using resources, taints, tolerations, affinity, and constraints.

kube-controller-manager

Runs controllers that reconcile desired and actual state.

CoreDNS

Provides DNS service discovery inside the cluster.

7. Understanding Kubernetes Node Components

kubelet

Node agent. Ensures Pods assigned to the node are running.

kube-proxy

Maintains Service networking rules and routes traffic to Pod backends.

CRI

Container Runtime Interface between kubelet and runtimes such as containerd or CRI-O.

CNI

Container Network Interface used by plugins such as Flannel, Calico, and Cilium.

CSI

Container Storage Interface for storage provisioning and attachment.

8. Kubernetes Cluster Setup and Validation Using kubeadm

Validation status: The commands in this document were checked for shell syntax and aligned with current official kubeadm/container runtime/package repository guidance. They were not executed as a real privileged multi-node cluster inside this chat environment because that requires root VMs, networking, and reboot-level system changes.

Lab 1: Environment Setup

# Run on each node, changing only the hostname line per VM
sudo hostnamectl set-hostname k8s-master    # control-plane only
# sudo hostnamectl set-hostname k8s-worker1 # worker1 only
# sudo hostnamectl set-hostname k8s-worker2 # worker2 only

# Run on ALL nodes. Replace IPs with your lab IPs.
sudo tee -a /etc/hosts >/dev/null <<'EOF'
192.168.56.10 k8s-master
192.168.56.11 k8s-worker1
192.168.56.12 k8s-worker2
EOF

ping -c 2 k8s-master
ping -c 2 k8s-worker1
ping -c 2 k8s-worker2

Lab 2: Install Prerequisites on All Nodes

# Run on ALL nodes: control-plane and workers
set -euo pipefail

# 1) Disable swap immediately and permanently
sudo swapoff -a
sudo sed -i.bak '/\sswap\s/s/^/#/' /etc/fstab

# 2) Load required kernel modules
cat <<'EOF' | sudo tee /etc/modules-load.d/k8s.conf
overlay
br_netfilter
EOF
sudo modprobe overlay
sudo modprobe br_netfilter

# 3) Kubernetes networking sysctl settings
cat <<'EOF' | sudo tee /etc/sysctl.d/99-kubernetes-cri.conf
net.bridge.bridge-nf-call-iptables  = 1
net.bridge.bridge-nf-call-ip6tables = 1
net.ipv4.ip_forward                 = 1
EOF
sudo sysctl --system

# 4) Install containerd
sudo apt-get update
sudo apt-get install -y ca-certificates curl gnupg apt-transport-https
sudo apt-get install -y containerd

# 5) Configure containerd with systemd cgroups
sudo mkdir -p /etc/containerd
containerd config default | sudo tee /etc/containerd/config.toml >/dev/null
sudo sed -i 's/SystemdCgroup = false/SystemdCgroup = true/' /etc/containerd/config.toml
sudo systemctl restart containerd
sudo systemctl enable containerd

# 6) Add Kubernetes v1.36 package repository
sudo mkdir -p /etc/apt/keyrings
curl -fsSL https://pkgs.k8s.io/core:/stable:/v1.36/deb/Release.key \
  | sudo gpg --dearmor -o /etc/apt/keyrings/kubernetes-apt-keyring.gpg

echo 'deb [signed-by=/etc/apt/keyrings/kubernetes-apt-keyring.gpg] https://pkgs.k8s.io/core:/stable:/v1.36/deb/ /' \
  | sudo tee /etc/apt/sources.list.d/kubernetes.list

# 7) Install kubeadm, kubelet, kubectl and lock versions
sudo apt-get update
sudo apt-get install -y kubelet kubeadm kubectl
sudo apt-mark hold kubelet kubeadm kubectl

# 8) Validate node prerequisites
sudo systemctl is-active --quiet containerd && echo "containerd: OK"
lsmod | grep -E 'overlay|br_netfilter'
sysctl net.ipv4.ip_forward net.bridge.bridge-nf-call-iptables
free -h | grep -i swap
kubeadm version
kubectl version --client

Lab 3: Initialize Cluster Using kubeadm

# Run only on the CONTROL-PLANE node
set -euo pipefail

sudo kubeadm init \
  --apiserver-advertise-address=192.168.56.10 \
  --pod-network-cidr=10.244.0.0/16 \
  --cri-socket=unix:///run/containerd/containerd.sock

# Configure kubectl for the normal user
mkdir -p $HOME/.kube
sudo cp -i /etc/kubernetes/admin.conf $HOME/.kube/config
sudo chown $(id -u):$(id -g) $HOME/.kube/config

# Install Flannel CNI. The 10.244.0.0/16 pod CIDR above matches Flannel default.
kubectl apply -f https://github.com/flannel-io/flannel/releases/latest/download/kube-flannel.yml

# Save the worker join command
kubeadm token create --print-join-command | tee $HOME/kubeadm-join-command.txt

Lab 4: Add Worker Nodes

# Run on EACH worker node.
# Use the exact command printed by kubeadm init or saved in kubeadm-join-command.txt.
sudo kubeadm join 192.168.56.10:6443 \
  --token <token-from-control-plane> \
  --discovery-token-ca-cert-hash sha256:<hash-from-control-plane> \
  --cri-socket=unix:///run/containerd/containerd.sock

9. Final Validation and Troubleshooting

Cluster Validation

# Run on the control-plane node after joining workers
kubectl get nodes -o wide
kubectl get pods -A -o wide
kubectl cluster-info

# Expected:
# 1) All nodes show STATUS=Ready
# 2) kube-system pods such as coredns, kube-proxy, kube-flannel are Running
# 3) No pods are stuck in ContainerCreating or CrashLoopBackOff

Application Test

# Deploy a simple test application
kubectl create deployment web --image=nginx:stable --replicas=2
kubectl expose deployment web --port=80 --type=NodePort
kubectl get deploy,pod,svc -o wide

# Test from any node. Replace <node-ip> and <node-port> from kubectl get svc web.
curl http://<node-ip>:<node-port>

Common Troubleshooting Commands

# Node not Ready
kubectl describe node <node-name>
journalctl -u kubelet -xe --no-pager | tail -80
sudo crictl ps -a

# CNI issue
kubectl get pods -n kube-system -o wide
kubectl logs -n kube-flannel -l app=flannel --tail=100

# Join token expired
kubeadm token create --print-join-command

# Reset a failed lab node carefully. This destroys local cluster state on that node.
sudo kubeadm reset -f
sudo rm -rf /etc/cni/net.d $HOME/.kube
sudo systemctl restart containerd kubelet
Common trainee mistakes: forgetting to disable swap, using a mismatched pod CIDR and CNI, not configuring containerd systemd cgroups, using old apt.kubernetes.io repositories, running kubeadm init on every node, or copying an expired join token.

Trainer Notes

  • Use clean Ubuntu Server VMs. Avoid machines already used for Docker Desktop, minikube, kind, or previous kubeadm labs unless reset carefully.
  • Use one Kubernetes minor version for the whole batch. This document uses v1.36 package repository.
  • Run all prerequisite commands on every node before kubeadm init or kubeadm join.
  • Flannel is used for simplicity. For production/security-focused training, replace with Calico or Cilium and adjust the pod CIDR accordingly.
