1. Understanding Kubernetes Storage

Container filesystems are temporary. When a container restarts, files written inside the container layer can be lost. Kubernetes volumes solve this by mounting storage into containers. A volume can be temporary, configuration-backed, secret-backed, or persistent beyond the Pod lifecycle.

Volume

A directory or block device made available inside one or more containers in a Pod.

PersistentVolume

Cluster-level storage resource provided statically by an admin or dynamically by a provisioner.

PersistentVolumeClaim

A namespace-level request for storage. Pods usually consume storage through a PVC.

Key idea: Pods should not care whether the storage is NFS, cloud disk, Ceph, vSphere, local-path, or SAN. The PV/PVC/StorageClass model hides backend details from application YAML.

3. Types of Volumes

4. Static vs Dynamic Storage

Static provisioning

Administrator creates a PV first. Users create PVCs. Kubernetes binds a matching PVC to an available PV.

  • Good for pre-existing NFS/SAN/manual volumes.
  • Requires admin work before users request storage.
  • Useful for teaching PV/PVC binding clearly.

Dynamic provisioning

User creates a PVC. StorageClass and its provisioner create the PV automatically.

  • Preferred for cloud-native clusters.
  • Requires a working StorageClass and provisioner/CSI driver.
  • Often uses WaitForFirstConsumer for topology-aware placement.

5. Storage Classes and Dynamic Volume Provisioning

A StorageClass describes a class or profile of storage offered by the cluster. It names a provisioner and may include parameters such as disk type, filesystem type, reclaim policy, expansion support, and binding mode.

FieldMeaningExample
provisionerPlugin/CSI driver that creates the volume.rancher.io/local-path, ebs.csi.aws.com, pd.csi.storage.gke.io
reclaimPolicyWhat happens to the PV after PVC deletion.Delete or Retain
volumeBindingModeWhen binding/provisioning happens.Immediate or WaitForFirstConsumer
allowVolumeExpansionWhether PVC size can be expanded.true
Training note: local-path provisioning is convenient for labs and single-node/development clusters, but it is not a production-grade replicated storage solution. Production storage normally uses a cloud/block/file CSI driver, Ceph/Rook, Longhorn, Portworx, NetApp, vSphere CSI, EBS/EFS CSI, Azure Disk/File CSI, or GCE PD CSI.

6. Volume Snapshots

VolumeSnapshot is the Kubernetes API pattern for point-in-time copies of a PVC. Snapshot support is not automatic in every cluster. It requires snapshot CRDs, a snapshot controller, and a CSI driver that supports snapshots.

VolumeSnapshot

Namespace object created by a user to request a snapshot of a PVC.

VolumeSnapshotContent

Cluster object representing the actual snapshot in the backend.

VolumeSnapshotClass

Snapshot profile, similar in spirit to a StorageClass.

Practical rule: do not promise snapshot labs unless your training cluster has snapshot CRDs and a CSI driver that supports snapshots. The document includes an optional detection and sample manifest section.

7. Volume Modes and Access Modes

Volume modes

ModeMeaningWhen to use
FilesystemThe default. Kubernetes mounts the volume as a directory.Most applications, logs, uploads, databases using filesystems.
BlockKubernetes exposes the raw block device to the container.Specialized databases or storage software that manage the block device themselves.

Access modes

ModeShort nameMeaningImportant note
ReadWriteOnceRWOMounted read-write by a single node.Multiple Pods can still use it if they run on the same node.
ReadOnlyManyROXMounted read-only by many nodes.Backend must support it.
ReadWriteManyRWXMounted read-write by many nodes.Usually file storage such as NFS, EFS, Azure Files, CephFS.
ReadWriteOncePodRWOPMounted read-write by only one Pod across the cluster.Useful when exactly one writer must exist.
Common misconception: access modes are mainly used for PV/PVC matching and mount constraints. They are not a substitute for application-level locking, database clustering, or filesystem permissions.

8. ConfigMaps and Secrets

ConfigMaps and Secrets are commonly consumed as environment variables or mounted files. ConfigMaps are for non-confidential configuration. Secrets are intended for sensitive data, but default Kubernetes Secrets still need proper RBAC and encryption-at-rest controls.

9. Understanding CSI

CSI means Container Storage Interface. It is the standard plugin model that lets storage vendors integrate with Kubernetes without building storage logic into Kubernetes core. CSI drivers commonly handle provisioning, attaching, mounting, resizing, snapshotting, and deleting volumes.

What Kubernetes does

  • Watches PVC, Pod, PV, StorageClass objects.
  • Schedules Pods with volume requirements.
  • Calls CSI sidecars and kubelet volume logic.

What CSI driver does

  • Talks to actual storage backend.
  • Creates and deletes volumes.
  • Attaches/mounts volumes to nodes.
  • May support expansion, snapshots, health monitoring.

Lab 1A: Working with Kubernetes Volume Service using emptyDir

This lab shows two containers in the same Pod sharing files through an ephemeral emptyDir volume.

apiVersion: v1
kind: Pod
metadata:
  name: emptydir-demo
  namespace: storage-lab
spec:
  restartPolicy: Never
  containers:
  - name: writer
    image: busybox:1.36
    command: ["/bin/sh", "-c"]
    args:
    - |
      i=1
      while [ $i -le 5 ]; do
        echo "$(date) - writer message $i" >> /shared/messages.txt
        i=$((i+1))
        sleep 2
      done
      sleep 3600
    volumeMounts:
    - name: shared-cache
      mountPath: /shared
  - name: reader
    image: busybox:1.36
    command: ["/bin/sh", "-c"]
    args:
    - |
      while true; do
        echo "---- reader view ----"
        cat /shared/messages.txt 2>/dev/null || echo "file not created yet"
        sleep 3
      done
    volumeMounts:
    - name: shared-cache
      mountPath: /shared
  volumes:
  - name: shared-cache
    emptyDir: {}

kubectl apply -f emptydir-demo.yaml
kubectl wait --for=condition=Ready pod/emptydir-demo -n storage-lab --timeout=120s
kubectl logs emptydir-demo -n storage-lab -c reader --tail=20

Expected result: the reader container displays messages written by the writer container under /shared/messages.txt.

kubectl exec -n storage-lab emptydir-demo -c writer -- cat /shared/messages.txt
kubectl delete pod emptydir-demo -n storage-lab
After deleting the Pod, the emptyDir data is gone. That is expected because emptyDir follows the Pod lifetime.

Lab 1B: Static PV/PVC using hostPath for classroom demonstration

Do not use hostPath as a production storage design. This lab is for understanding PV/PVC binding. It pins the PV to one node using node affinity.

Find one node hostname label and use it in the PV and Pod:

NODE_HOSTNAME=$(kubectl get nodes -o jsonpath='{.items[0].metadata.labels.kubernetes\.io/hostname}')
echo "$NODE_HOSTNAME"

Create a static PV and matching PVC:

apiVersion: v1
kind: PersistentVolume
metadata:
  name: manual-pv-hostpath
spec:
  capacity:
    storage: 1Gi
  volumeMode: Filesystem
  accessModes:
  - ReadWriteOnce
  persistentVolumeReclaimPolicy: Retain
  storageClassName: manual
  hostPath:
    path: /tmp/k8s-static-pv
    type: DirectoryOrCreate
  nodeAffinity:
    required:
      nodeSelectorTerms:
      - matchExpressions:
        - key: kubernetes.io/hostname
          operator: In
          values:
          - "${NODE_HOSTNAME}"
---
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: manual-pvc
  namespace: storage-lab
spec:
  storageClassName: manual
  volumeName: manual-pv-hostpath
  accessModes:
  - ReadWriteOnce
  resources:
    requests:
      storage: 200Mi

Lab 2: Configuring Dynamic Volume Provisioning

First check whether your cluster already has a default StorageClass:

kubectl get storageclass
kubectl get storageclass -o custom-columns=NAME:.metadata.name,PROVISIONER:.provisioner,DEFAULT:.metadata.annotations.storageclass\.kubernetes\.io/is-default-class,BINDING:.volumeBindingMode

If your cluster already has a suitable default StorageClass, set STORAGE_CLASS to that name. Example:

STORAGE_CLASS=$(kubectl get sc -o jsonpath='{.items[?(@.metadata.annotations.storageclass\.kubernetes\.io/is-default-class=="true")].metadata.name}' | awk '{{print $1}}')
echo "$STORAGE_CLASS"

If there is no StorageClass in your lab cluster, install Rancher local-path provisioner for training:

kubectl apply -f https://raw.githubusercontent.com/rancher/local-path-provisioner/v0.0.36/deploy/local-path-storage.yaml
kubectl -n local-path-storage rollout status deploy/local-path-provisioner --timeout=120s
kubectl get storageclass local-path
STORAGE_CLASS=local-path

Create a PVC that asks the StorageClass to provision storage automatically, then start a Pod that writes into it:

Lab 4: Projected Volume using ConfigMap + Secret + Downward API

This lab maps multiple sources into one mount path called /projected.

apiVersion: v1
kind: Pod
metadata:
  name: projected-volume-demo
  namespace: storage-lab
spec:
  restartPolicy: Never

      echo "Files in projected volume:"

      - secret:
          name: db-secret
          items:

            path: secret/db-user
      - downwardAPI:
          items:
          - path: pod/name
            fieldRef:
              fieldPath: metadata.name
          - path: pod/namespace

              fieldPath: metadata.namespace

Optional Lab 5: Volume Snapshot readiness check

Run these checks before attempting snapshots:

kubectl api-resources | grep -i volumesnapshot || true
kubectl get volumesnapshotclass || true
kubectl get crd | grep snapshot.storage.k8s.io || true

Only if your cluster has snapshot CRDs, a snapshot controller, and a compatible CSI driver, create a snapshot like this after replacing the class name:

apiVersion: snapshot.storage.k8s.io/v1
kind: VolumeSnapshot
metadata:
  name: dynamic-pvc-snapshot
  namespace: storage-lab
spec:
  volumeSnapshotClassName: YOUR_VOLUME_SNAPSHOT_CLASS
  source:
    persistentVolumeClaimName: dynamic-pvc

kubectl apply -f volume-snapshot.yaml
kubectl get volumesnapshot -n storage-lab
kubectl describe volumesnapshot dynamic-pvc-snapshot -n storage-lab
Expected classroom outcome: in many local clusters this optional lab will not work unless snapshot components are installed. Treat it as a concept and readiness-check lab unless the environment is prepared.
