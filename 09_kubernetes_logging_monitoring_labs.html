Kubernetes Logging and Monitoring
Trainee Lab Reference

A practical, beginner-friendly guide for describing resources, reading Pod/container logs, debugging failed applications, using events, inspecting nodes with crictl, and deploying Metrics Server for kubectl top.

1. Training Objectives

By the end of this module, trainees can:

  • Use kubectl get and kubectl describe to inspect current state, desired state, related resources, and events.
  • Read single-container, multi-container, previous, streaming, and time-filtered logs.
  • Debug common failures: ImagePullBackOff, CrashLoopBackOff, init container failure, service selector mismatch, and OOMKilled.
  • Use kubectl exec, kubectl debug, ephemeral debug containers, node debug pods, and crictl.
  • Install Metrics Server and validate kubectl top nodes / kubectl top pods.

Lab Assumptions

  • A working Kubernetes cluster: Minikube, Kind, kubeadm, EKS, AKS, GKE, OpenShift-compatible Kubernetes, or similar.
  • kubectl is configured and can run kubectl get nodes.
  • Internet access is available to pull public container images.
  • For node-level and crictl labs, trainees need node shell access and suitable permissions.
Do not run failure labs in production namespaces. These labs intentionally create broken Pods.

2. Conceptual Mental Model

Kubernetes troubleshooting is evidence collection. Start broad, then narrow down: resource state → events → logs → shell/debug container → node/runtime inspection → metrics.

Animated Concept: Debugging Evidence Flow
kubectl getWhat exists?
describe/eventsWhy changed?
logs/execWhat app says?
debug/crictlRuntime view
topCPU/memory
Animated Concept: Logs, Metrics, and Events Are Different Signals

Logs

Application text output from containers. Best for error messages, stack traces, request traces, and custom application messages.

{"level":"INFO","msg":"request received"}
{"level":"INFO","msg":"db connected"}
{"level":"ERROR","msg":"timeout"}
{"level":"INFO","msg":"retrying"}
{"level":"WARN","msg":"slow response"}
{"level":"INFO","msg":"request completed"}
{"level":"ERROR","msg":"missing config"}

Metrics

Numeric measurements such as CPU and memory. Best for load, capacity, autoscaling, and resource pressure signals.

Metrics Server provides recent CPU/memory metrics for kubectl top and autoscaling.

Events

Kubernetes-generated lifecycle messages. Best for scheduling, image pull, volume, probe, and object state-change issues.

Pod lifecycle

Scheduled → Pulling → Failed

3. Describe Resources

kubectl describe gives a human-readable, aggregated view of a resource. For Pods, it commonly shows labels, node placement, container states, image details, restart count, probes, volumes, conditions, and recent events.

CommandPurpose
kubectl get pods -n k8s-debug-lab -o wideList Pods with node name, IP, readiness, restarts, and age.
kubectl describe pod POD -n k8s-debug-labDetailed Pod state and related events.
kubectl describe deploy/log-demo -n k8s-debug-labDeployment rollout, ReplicaSet, labels, selectors, and Pod template.
kubectl get pod POD -o yamlMachine-readable full object specification and status.
kubectl get pod POD -o jsonpath='{.status.phase}'Extract a precise field for scripts.
Rule: Use get to find the object and status quickly. Use describe to understand why the object is in that state.

4. Pod and Container Logs

In Kubernetes, the common logging pattern is for applications to write logs to standard output and standard error. The container runtime captures those streams, and kubectl logs retrieves them.

Use caseCommand
Current logskubectl logs POD -n NAMESPACE
Follow logskubectl logs -f POD -n NAMESPACE
Last few lineskubectl logs POD --tail=50
Logs from last 5 minuteskubectl logs POD --since=5m
Specific containerkubectl logs POD -c CONTAINER
Previous crashed container logskubectl logs POD -c CONTAINER --previous
All containers in a Podkubectl logs POD --all-containers=true
Deployment logskubectl logs deploy/DEPLOYMENT -n NAMESPACE
Important: kubectl logs is not a long-term log storage system. For production, send logs to a backend such as Loki, Elasticsearch/OpenSearch, Splunk, Cloud Logging, or another centralized logging platform.

5. Debugging Your Application

Debug Pods

  1. kubectl get pod — identify status.
  2. kubectl describe pod — inspect state, conditions, restart reason, events.
  3. kubectl logs — inspect application output.
  4. kubectl logs --previous — inspect prior failed container instance.
  5. kubectl exec — inspect inside running container.
  6. kubectl debug — attach a temporary debug container or create a debug copy.

Determine Reason for Pod Failure

Status / ReasonLikely meaningFirst commands
PendingScheduling, volume, quota, taint, or resource issue.describe pod, events
ImagePullBackOffBad image name/tag, registry auth, network, pull policy.describe pod
CrashLoopBackOffContainer starts then exits repeatedly.logs --previous, describe pod
Init:CrashLoopBackOffInit container failed repeatedly.logs -c init-name
OOMKilledContainer exceeded memory limit.describe pod, top pod

6. Inspect Kubernetes Node with crictl

crictl is a command-line tool for CRI-compatible container runtimes. Use it when kubectl is not enough and you need a node/runtime-level view.

Permission boundary: crictl normally runs on the node with sudo. It is not a regular developer command for shared production clusters.
CommandPurpose
sudo crictl infoRuntime information.
sudo crictl podsList Pod sandboxes known to the runtime.
sudo crictl ps -aList containers including exited containers.
sudo crictl logs CONTAINER_IDRead logs directly from runtime.
sudo crictl inspect CONTAINER_IDInspect runtime container details.

7. Metrics Server and the top Command

Metrics Server collects recent CPU and memory metrics from kubelets and exposes them through the Kubernetes Metrics API. kubectl top depends on Metrics Server being installed and healthy.

Animated Concept: Metrics Server Flow
Compatibility note: Metrics Server 0.8.x targets Kubernetes 1.31+. For Kubernetes 1.27 to 1.30 lab clusters, use Metrics Server 0.7.2.
Scope note: Metrics Server is useful for kubectl top and autoscaling signals. It is not a replacement for Prometheus/Grafana or a long-term monitoring system.

8. Events

Events are Kubernetes-generated records that explain what happened to objects: scheduled, pulling image, failed mount, failed probe, back-off restarting container, service endpoint changes, and similar lifecycle activity.

CommandPurpose
kubectl events -n k8s-debug-labList recent events in a namespace.
kubectl events -n k8s-debug-lab --types=WarningShow warning events only.
kubectl events --for pod/POD -n k8s-debug-labShow events for one Pod.
kubectl events -n k8s-debug-lab --watchWatch new events as they occur.
kubectl get events --sort-by=.lastTimestampOlder-compatible fallback.

LAB 0

Prepare Lab Namespace and Base Workloads

Create the namespace and two working workloads: a log generator Deployment and a two-container Pod.

00-namespace.yaml
apiVersion: v1
kind: Namespace
metadata:
  name: k8s-debug-lab
  labels:
    purpose: logging-monitoring-training
01-log-demo.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: log-demo
  namespace: k8s-debug-lab
spec:
  replicas: 1
  selector:
    matchLabels:
      app: log-demo
  template:
    metadata:
      labels:
        app: log-demo
    spec:
      containers:
      - name: logger
        image: busybox:1.36.1
        command: ["sh", "-c"]
        args:
        - |
          i=0
          while true; do
            echo "{\"ts\":\"$(date -Iseconds)\",\"level\":\"INFO\",\"msg\":\"heartbeat from log-demo\",\"count\":$i}"
            i=$((i+1))
            sleep 3
          done
        resources:
          requests:
            cpu: "20m"
            memory: "32Mi"
          limits:
            cpu: "100m"
            memory: "64Mi"

kind: Pod
metadata:

    app: two-container-logs
spec:

LAB 2B

Debug CrashLoopBackOff

This creates a Pod that starts, writes an error, exits with code 1, and restarts repeatedly.

04-crashloop-pod.yaml
apiVersion: v1
kind: Pod
metadata:
  name: crashloop-pod
  namespace: k8s-debug-lab
  labels:
    app: crashloop-demo
spec:
  restartPolicy: Always
  containers:
  - name: app
    image: busybox:1.36.1
    command: ["sh", "-c"]
    args:
    - |
      echo "Starting application"
      echo "FATAL: missing required environment variable APP_MODE"
      exit 1
debug-crashloop.sh
kubectl apply -f 04-crashloop-pod.yaml
kubectl get pod crashloop-pod -n k8s-debug-lab -w

# Press Ctrl+C after you see CrashLoopBackOff, then inspect:
kubectl describe pod crashloop-pod -n k8s-debug-lab
kubectl logs crashloop-pod -n k8s-debug-lab -c app
kubectl logs crashloop-pod -n k8s-debug-lab -c app --previous
kubectl get pod crashloop-pod -n k8s-debug-lab -o jsonpath='{.status.containerStatuses[0].lastState.terminated.reason}{"\n"}'

# Cleanup
kubectl delete pod crashloop-pod -n k8s-debug-lab
What trainees should learn
  • CrashLoopBackOff means the container starts and then exits repeatedly.
  • kubectl logs --previous is important because the current container instance may have already restarted.
  • lastState.terminated.reason and exit code help confirm the failure reason.
LAB 2C

Debug Init Container Failure

This demonstrates that application containers do not start until init containers complete successfully.

05-init-failure-pod.yaml
apiVersion: v1
kind: Pod
metadata:
  name: init-failure-pod
  namespace: k8s-debug-lab
  labels:
    app: init-failure-demo
spec:
  volumes:
  - name: workdir
    emptyDir: {}
  initContainers:
  - name: init-check
    image: busybox:1.36.1
    command: ["sh", "-c"]
    args:
    - |
      echo "Init container checking for /work/ready"
      test -f /work/ready
    volumeMounts:
    - name: workdir
      mountPath: /work
  containers:
  - name: app
    image: busybox:1.36.1
    command: ["sh", "-c", "echo Application started; sleep 3600"]
    volumeMounts:
    - name: workdir
      mountPath: /work
06-init-fixed-pod.yaml
apiVersion: v1
kind: Pod
metadata:
  name: init-fixed-pod
  namespace: k8s-debug-lab
  labels:
    app: init-fixed-demo
spec:
  volumes:
  - name: workdir
    emptyDir: {}
  initContainers:
  - name: init-create-file
    image: busybox:1.36.1
    command: ["sh", "-c", "echo ready > /work/ready && cat /work/ready"]
    volumeMounts:
    - name: workdir
      mountPath: /work
  containers:
  - name: app
    image: busybox:1.36.1
    command: ["sh", "-c", "echo Application started after init; sleep 3600"]

    - name: workdir
      mountPath: /work

LAB 3

Get a Shell, Use Ephemeral Debug Containers, and Debug Node

Use this after the base workloads are running. Some commands require higher privileges and may not work in locked-down managed clusters.

exec-and-ephemeral-debug.sh
POD=$(kubectl get pod -n k8s-debug-lab -l app=log-demo -o jsonpath='{.items[0].metadata.name}')

# Get a shell in the existing container.
kubectl exec -it "$POD" -n k8s-debug-lab -c logger -- sh

# Add an ephemeral debug container to a running Pod.
kubectl debug -it "$POD" -n k8s-debug-lab --image=busybox:1.36.1 --target=logger -- sh

# If --target is unsupported by your runtime, retry without it:
kubectl debug -it "$POD" -n k8s-debug-lab --image=busybox:1.36.1 -- sh

# Create a debug copy of the Pod with a different command.
kubectl debug "$POD" -n k8s-debug-lab -it --copy-to=log-demo-debug --container=logger -- sh
kubectl delete pod log-demo-debug -n k8s-debug-lab --ignore-not-found=true
debug-node-and-crictl.sh
NODE=$(kubectl get pod -n k8s-debug-lab -l app=log-demo -o jsonpath='{.items[0].spec.nodeName}')
echo "$NODE"

# Requires cluster-admin style permissions. This creates a privileged node debug Pod.
kubectl debug node/$NODE -it --image=busybox:1.36.1

# Inside the debug shell, the host filesystem is mounted at /host.
# Example commands inside the node debug shell:
chroot /host sh
ps aux | grep kubelet
ls /var/log/pods | head
exit
exit

# crictl commands must be run on the Kubernetes node, not usually on your laptop.
sudo tee /etc/crictl.yaml >/dev/null <<'EOF'
runtime-endpoint: unix:///run/containerd/containerd.sock
image-endpoint: unix:///run/containerd/containerd.sock
timeout: 10
debug: false
EOF

sudo crictl info
sudo crictl pods
sudo crictl ps -a
sudo crictl images

# Filter a container and inspect it.
CID=$(sudo crictl ps -a --name logger -q | head -n 1)
sudo crictl inspect "$CID" | less
sudo crictl logs "$CID" | tail
Common issues
  • If kubectl exec fails, the container may not be running or may not have the shell requested.
  • If kubectl debug --target fails, retry without --target.
  • If node debug is denied, your account lacks privileges.
  • If crictl cannot connect, verify the runtime endpoint. For containerd, common paths are unix:///run/containerd/containerd.sock and unix:///var/run/containerd/containerd.sock.
