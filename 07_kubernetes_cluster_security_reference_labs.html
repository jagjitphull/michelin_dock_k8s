1. Objectives and Prerequisites

Learning objectives

  • Understand how Kubernetes decides whether a request is allowed.
  • Create service accounts, Roles, RoleBindings, ClusterRoles, and ClusterRoleBindings.
  • Understand user certificates and kubeconfig entries.
  • Harden Pods using security context, Linux capabilities, seccomp, and SELinux labels.
  • Validate access using kubectl auth can-i and runtime checks from inside containers.

Lab prerequisites

  • A working Kubernetes cluster: kind, minikube, kubeadm, or cloud Kubernetes.
  • kubectl configured with admin access for lab creation.
  • Internet access from nodes to pull nginx:1.27-alpine and busybox:1.36.
  • Optional: SELinux-enabled worker nodes for the SELinux label section.

2. Animated Concept: Kubernetes Security Request Flow

Every API request passes through a security pipeline. A request may be syntactically valid but still denied if the identity is not authenticated, not authorized, or violates an admission policy.

3. Securing the Kubernetes Cluster

Cluster security is layered. Do not rely only on RBAC or only on container restrictions. Secure the control plane, nodes, identities, network, workloads, and supply chain together.

Control plane hardening
Least-privilege RBAC
Pod security controls
Network segmentation

Control plane

  • Expose the API server only to trusted networks.
  • Use TLS everywhere: API server, kubelet, etcd, admission webhooks.
  • Encrypt Secrets at rest using Kubernetes encryption configuration.
  • Back up etcd and protect etcd certificates and snapshots.
  • Enable audit logging and forward logs to a secure log platform.

Worker nodes

  • Limit SSH access; prefer immutable or managed nodes.
  • Patch OS, kubelet, container runtime, and CNI plugins.
  • Restrict kubelet API access and avoid anonymous kubelet access.
  • Use SELinux/AppArmor/seccomp where supported.
  • Separate high-risk workloads using taints, node pools, or runtime classes.

Workloads

  • Run as non-root and disable privilege escalation.
  • Drop Linux capabilities by default.
  • Use read-only root filesystems where possible.
  • Use explicit service accounts; avoid using the default service account.
  • Use NetworkPolicies to control pod-to-pod traffic.

Supply chain

  • Use trusted image registries and immutable image tags/digests.
  • Scan images for vulnerabilities and secrets.
  • Sign images where feasible.
  • Restrict who can deploy privileged workloads.
  • Use admission controls to enforce baseline or restricted standards.

5. Cluster Authentication and Authorization

Authentication

Authentication verifies identity. Kubernetes supports multiple authenticators such as client certificates, bearer tokens, service account tokens, OIDC, and webhook token authentication.

  • Certificate user example: CN=trainee, group O=dev-team.
  • Service account example: system:serviceaccount:rbac-lab:trainee-viewer.
  • Groups are important because RBAC can bind permissions to users, groups, or service accounts.

Authorization

Authorization checks whether an authenticated identity can perform a verb on a resource in a namespace or across the cluster.

  • Verb: get, list, watch, create, update, patch, delete.
  • Resource: pods, deployments, secrets, nodes.
  • Namespace: applies to namespaced resources such as Pods and Deployments.
Useful command: kubectl auth can-i VERB RESOURCE -n NAMESPACE --as=IDENTITY is the safest way to validate RBAC during training.

6. Creating TLS Certificates and kubeconfig

TLS in Kubernetes has two common meanings: securing component communication and authenticating clients. This section demonstrates a client certificate identity for a trainee user. Creating the identity does not grant permissions; RBAC must still be added.

Important: In a real production cluster, normal users are usually integrated through an external identity provider such as OIDC/SSO. Client certificate users are useful for learning and controlled administrative use, but certificate lifecycle management must be handled carefully.

6.1 Create a trainee client certificate using the Kubernetes CSR API

Client certificate request
mkdir -p ~/k8s-security-lab/certs
cd ~/k8s-security-lab/certs

# 1) Create private key and CSR for a normal Kubernetes user identity.
openssl genrsa -out trainee.key 2048
openssl req -new -key trainee.key -out trainee.csr -subj "/CN=trainee/O=dev-team"

# 2) Convert CSR to base64 without line wrapping.
CSR_B64=$(base64 -w0 < trainee.csr 2>/dev/null || base64 < trainee.csr | tr -d '\n')

# 3) Submit CSR to Kubernetes certificates.k8s.io API.
cat > trainee-csr.yaml <<EOF
apiVersion: certificates.k8s.io/v1
kind: CertificateSigningRequest
metadata:
  name: trainee
spec:
  request: ${CSR_B64}
  signerName: kubernetes.io/kube-apiserver-client
  expirationSeconds: 86400
  usages:
  - client auth
EOF

kubectl apply -f trainee-csr.yaml
kubectl certificate approve trainee

# 4) Download the signed certificate.
kubectl get csr trainee -o jsonpath='{.status.certificate}' | base64 -d > trainee.crt
ls -l trainee.key trainee.csr trainee.crt

6.2 Create a dedicated kubeconfig for the trainee user

kubeconfig creation
cd ~/k8s-security-lab/certs

CLUSTER_NAME=$(kubectl config view --minify -o jsonpath='{.clusters[0].name}')
SERVER=$(kubectl config view --minify -o jsonpath='{.clusters[0].cluster.server}')
CA_DATA=$(kubectl config view --raw --minify -o jsonpath='{.clusters[0].cluster.certificate-authority-data}')
echo "$CA_DATA" | base64 -d > ca.crt

kubectl config set-cluster "$CLUSTER_NAME" \
  --server="$SERVER" \
  --certificate-authority=ca.crt \
  --embed-certs=true \
  --kubeconfig=trainee.kubeconfig

kubectl config set-credentials trainee \
  --client-certificate=trainee.crt \
  --client-key=trainee.key \
  --embed-certs=true \
  --kubeconfig=trainee.kubeconfig

kubectl config set-context trainee-context \
  --cluster="$CLUSTER_NAME" \
  --user=trainee \
  --namespace=rbac-lab \
  --kubeconfig=trainee.kubeconfig

kubectl config use-context trainee-context --kubeconfig=trainee.kubeconfig

# Expected before RBAC is added: the user authenticates but is not authorized.
kubectl --kubeconfig=trainee.kubeconfig get pods -n rbac-lab

7. Service Accounts and Users

Normal users

  • Normal users are not stored as Kubernetes API objects.
  • They come from client certificates, OIDC, external identity systems, or authentication webhooks.
  • RBAC can refer to them by username or group.
  • Example certificate subject: /CN=trainee/O=dev-team.

Service accounts

  • ServiceAccounts are Kubernetes API objects.
  • They are namespaced and commonly used by Pods.
  • Use one service account per application or automation function.
  • Short-lived tokens can be created using kubectl create token.
Do not use the default service account for everything. Create explicit service accounts and bind only the minimum permissions required.

8. Roles, RoleBindings, ClusterRoles, and ClusterRoleBindings

RBAC separates permission definition from permission assignment. A Role or ClusterRole defines allowed actions. A RoleBinding or ClusterRoleBinding assigns those permissions to a subject.

Namespace: rbac-lab
SA
Role: pod-reader

Permission applies only here.

Namespace: default
SA
No binding

Request is denied.

ObjectScopeUse when
RoleNamespaceYou want permissions only inside one namespace.
RoleBindingNamespaceYou want to grant a Role or ClusterRole only inside the RoleBinding namespace.
ClusterRoleClusterYou need cluster-scoped permissions or a reusable permission template.
ClusterRoleBindingClusterYou want to grant ClusterRole permissions across the whole cluster.
Common spelling issue: The correct object is ClusterRole, not CluserRole.

9. Pod and Container Security Context

A security context defines privilege and access-control settings for a Pod or container. Pod-level settings apply to all containers unless overridden at container level.

Pod securityContext: runAsNonRoot, runAsUser, fsGroup, seccompProfile
Container securityContext: allowPrivilegeEscalation, readOnlyRootFilesystem
Linux capabilities: drop ALL, add only the minimum required
Kernel controls: seccomp, SELinux/AppArmor, namespaces, cgroups
FieldLevelRecommended training valueWhy it matters
runAsNonRootPod/containertruePrevents running as UID 0.
runAsUserPod/container1000Runs process with a known non-root UID.
allowPrivilegeEscalationContainerfalseBlocks gaining more privileges through setuid or file capabilities.
capabilities.dropContainer["ALL"]Removes unnecessary Linux privileges.
seccompProfile.typePod/containerRuntimeDefaultUses runtime default syscall filtering.
readOnlyRootFilesystemContainertrueReduces damage if the process is compromised.
seLinuxOptionsPod/containertype: container_tApplies SELinux labels where SELinux is supported.
Pod Security Standards: Use namespace labels such as pod-security.kubernetes.io/enforce=restricted to prevent unsafe Pods from being created.

Lab 1: Working with Namespaces and RBAC

In this lab, trainees create a namespace, service accounts, a Role, RoleBinding, ClusterRole, and ClusterRoleBinding. They validate what is allowed and denied.

Expected outcome: Trainees can explain why one identity can read Pods but cannot create Deployments, and why a ClusterRole grants cluster-wide access only when bound with ClusterRoleBinding.

Step 1: Create namespace and sample Pod

Namespace and Pod
mkdir -p ~/k8s-security-lab/rbac
cd ~/k8s-security-lab/rbac

kubectl create namespace rbac-lab --dry-run=client -o yaml | kubectl apply -f -

cat > demo-pod.yaml <<'EOF'
apiVersion: v1
kind: Pod
metadata:
  name: web-demo
  namespace: rbac-lab
  labels:
    app: web-demo
spec:
  containers:
  - name: web
    image: nginx:1.27-alpine
    ports:
    - containerPort: 80
EOF

kubectl apply -f demo-pod.yaml
kubectl -n rbac-lab get pods -o wide

Step 2: Create service accounts

ServiceAccounts
kubectl -n rbac-lab create serviceaccount trainee-viewer --dry-run=client -o yaml | kubectl apply -f -
kubectl -n rbac-lab create serviceaccount trainee-deployer --dry-run=client -o yaml | kubectl apply -f -

kubectl -n rbac-lab get serviceaccounts

Step 3: Create a namespaced Role and RoleBinding

Role + RoleBinding
cat > pod-reader-role.yaml <<'EOF'
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: pod-reader
  namespace: rbac-lab
rules:
- apiGroups: [""]
  resources: ["pods", "pods/log"]
  verbs: ["get", "list", "watch"]
EOF

cat > pod-reader-binding.yaml <<'EOF'
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: trainee-viewer-can-read-pods
  namespace: rbac-lab
subjects:
- kind: ServiceAccount
  name: trainee-viewer
  namespace: rbac-lab
roleRef:
  kind: Role
  name: pod-reader
  apiGroup: rbac.authorization.k8s.io
EOF

kubectl apply -f pod-reader-role.yaml
kubectl apply -f pod-reader-binding.yaml

Step 4: Validate allowed and denied actions

RBAC validation

kubectl auth can-i create deployments \
  -n rbac-lab \
  --as=system:serviceaccount:rbac-lab:trainee-viewer

# Expected:
# yes
# no

Lab 2: Working with Security Context

In this lab, trainees enforce the restricted Pod Security Standard in a namespace, observe a rejected privileged Pod, and deploy hardened Pods using security context settings.

Expected outcome: Trainees can verify UID/GID, read-only filesystem behavior, dropped capabilities, seccomp status, and understand when SELinux labels apply.

Step 1: Create a restricted namespace

Pod Security namespace
mkdir -p ~/k8s-security-lab/security-context
cd ~/k8s-security-lab/security-context

kubectl create namespace secure-lab --dry-run=client -o yaml | kubectl apply -f -

# Enforce Kubernetes Pod Security Standards at the namespace level.
# This requires Kubernetes v1.25+ where Pod Security Admission is stable.
kubectl label namespace secure-lab \
  pod-security.kubernetes.io/enforce=restricted \
  pod-security.kubernetes.io/audit=restricted \
  pod-security.kubernetes.io/warn=restricted \
  --overwrite

kubectl describe namespace secure-lab | grep -i pod-security -A5

Step 2: Try an unsafe privileged Pod

Expected rejection
cat > bad-privileged-pod.yaml <<'EOF'
apiVersion: v1
kind: Pod
metadata:
  name: bad-privileged-pod
  namespace: secure-lab
spec:
  containers:
  - name: bad
    image: busybox:1.36
    command: ["sh", "-c", "sleep 3600"]
    securityContext:
      privileged: true
      runAsUser: 0
EOF

kubectl apply -f bad-privileged-pod.yaml

# Expected: rejected by Pod Security Admission because the namespace enforces "restricted".

Step 3: Deploy a hardened Pod

Safe Pod securityContext
cat > safe-pod.yaml <<'EOF'
apiVersion: v1
kind: Pod
metadata:
  name: safe-pod
  namespace: secure-lab
spec:
  securityContext:
    runAsNonRoot: true
    runAsUser: 1000
    runAsGroup: 3000
    fsGroup: 2000
    seccompProfile:
      type: RuntimeDefault
  containers:
  - name: app
    image: busybox:1.36
    command: ["sh", "-c", "id; touch /tmp/write-ok; sleep 3600"]
    securityContext:
      allowPrivilegeEscalation: false
      readOnlyRootFilesystem: true
      capabilities:
        drop: ["ALL"]
    volumeMounts:
    - name: tmp
      mountPath: /tmp
  volumes:
  - name: tmp
    emptyDir: {}
EOF

kubectl apply -f safe-pod.yaml
kubectl -n secure-lab wait --for=condition=Ready pod/safe-pod --timeout=90s
kubectl -n secure-lab get pod safe-pod -o wide

Step 4: Verify runtime behavior

Runtime checks
kubectl -n secure-lab exec safe-pod -- id
kubectl -n secure-lab exec safe-pod -- sh -c 'ls -l /tmp/write-ok'
kubectl -n secure-lab exec safe-pod -- sh -c 'touch /root/should-fail || echo "root filesystem write blocked"'
kubectl -n secure-lab exec safe-pod -- sh -c 'grep Seccomp /proc/1/status; grep CapEff /proc/1/status'

# Expected observations:
# - uid=1000 and gid=3000
# - write to /tmp works because emptyDir is mounted there
# - write to /root fails because root filesystem is read-only
# - Seccomp is active and effective capabilities are dropped

Step 5: Add a narrow Linux capability

Capabilities
cat > capability-pod.yaml <<'EOF'
apiVersion: v1
kind: Pod
metadata:
  name: capability-pod
  namespace: secure-lab
spec:
  securityContext:
    runAsNonRoot: true
    runAsUser: 1000
    seccompProfile:
      type: RuntimeDefault
  containers:
  - name: app
    image: busybox:1.36
    command: ["sh", "-c", "sleep 3600"]
    securityContext:
      allowPrivilegeEscalation: false
      capabilities:
        drop: ["ALL"]
        add: ["NET_BIND_SERVICE"]
EOF

kubectl apply -f capability-pod.yaml
kubectl -n secure-lab wait --for=condition=Ready pod/capability-pod --timeout=90s
kubectl -n secure-lab exec capability-pod -- sh -c 'grep CapEff /proc/1/status'

# NET_BIND_SERVICE is the narrow capability used for binding low TCP/UDP ports.
# Do not add broad capabilities such as SYS_ADMIN in normal application Pods.

Cluster Security Checklist for Trainees

RBAC checklist

  • Use named service accounts per workload.
  • Prefer namespaced Roles over ClusterRoles where possible.
  • Avoid wildcard verbs and resources unless absolutely required.
  • Never bind cluster-admin to application service accounts.
  • Validate with kubectl auth can-i.

Pod hardening checklist

  • Set runAsNonRoot: true.
  • Set allowPrivilegeEscalation: false.
  • Drop all capabilities; add back only what is required.
  • Use seccompProfile: RuntimeDefault.
  • Use read-only root filesystem where possible.
  • Use Pod Security Standards labels on namespaces.

Assessment questions

  1. What is the difference between authentication and authorization in Kubernetes?
  2. Why does creating a TLS client certificate not automatically grant permissions?
  3. What is the difference between RoleBinding and ClusterRoleBinding?
  4. Why should an application Pod not use the default service account?
  5. What does allowPrivilegeEscalation: false prevent?
  6. Why is capabilities.drop: ["ALL"] a good default?
  7. When is seLinuxOptions practically useful?
  8. How do you prove whether a service account can create Deployments?
