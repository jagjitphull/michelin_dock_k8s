Docker Containers — Reference Guide with Labs

For trainees learning how to create, run, manage, network, limit, and persist Docker containers using practical commands. The examples are intentionally simple and use common images such as hello-world, nginx, alpine, and busybox.

0. Prerequisites and Safety Checks

Required

Docker Engine or Docker Desktop installed and running.

docker version
docker info

Linux permission check

If you see permission denied for /var/run/docker.sock, use sudo or add the user to the Docker group.

sudo usermod -aG docker $USER
newgrp docker
docker run hello-world

Images used

docker pull hello-world
docker pull nginx:alpine
docker pull alpine:latest
docker pull busybox:latest
Security note: membership in the docker group is effectively root-equivalent on a Linux host. Use it only on trusted training machines.

1. Creating Containers from Images

A Docker image is a read-only template. A container is a running or stopped instance created from that image.

Imagenginx:alpine
Containerweb1
Processnginx running

Basic creation and execution

# Pull image locally
docker pull nginx:alpine

# Create and run a new container
docker run --name web1 nginx:alpine

# In another terminal, check it
docker ps

When you use docker run, Docker performs image lookup, container creation, and process startup in one command.

Create without starting

docker create --name web-created nginx:alpine
docker ps -a
docker start web-created
docker ps
When to use docker create: when you want to prepare container configuration first and start it later.

3. Container Lifecycle Management

A container moves through lifecycle states such as created, running, paused, stopped, and removed.

Createddocker create
Runningdocker start / run
Pauseddocker pause
Stoppeddocker stop
Removeddocker rm

Lifecycle command set

# Start a long-running container
docker run -d --name life-demo nginx:alpine

# List running containers
docker ps

# List all containers, including stopped
docker ps -a

# Inspect metadata
docker inspect life-demo

# See logs
docker logs life-demo

# Pause and unpause
docker pause life-demo
docker ps

docker unpause life-demo

# Stop and start
docker stop life-demo
docker start life-demo

# Restart
docker restart life-demo

# Remove: must stop first unless using -f
docker stop life-demo
docker rm life-demo

Executing commands inside a running container

docker run -d --name exec-demo nginx:alpine

docker exec exec-demo nginx -v
docker exec -it exec-demo sh
# Inside container:
ps
exit

docker rm -f exec-demo

Restart policies

docker run -d --name restart-demo --restart unless-stopped nginx:alpine

docker inspect -f '{{.HostConfig.RestartPolicy.Name}}' restart-demo

docker rm -f restart-demo
Common restart policies: no, on-failure, always, and unless-stopped.

4. Container Resource Constraints: CPU and Memory

By default, containers can use host CPU and memory as allowed by the host kernel scheduler. Resource limits prevent one container from consuming too much.

Memory limit

docker run -d --name mem-demo --memory 128m nginx:alpine

docker inspect -f '{{.HostConfig.Memory}}' mem-demo

docker stats mem-demo

docker rm -f mem-demo

CPU limit

# Limit container to approximately half a CPU
docker run -d --name cpu-demo --cpus="0.5" nginx:alpine

docker inspect -f '{{.HostConfig.NanoCpus}}' cpu-demo

docker stats cpu-demo

docker rm -f cpu-demo

CPU share / relative priority

docker run -d --name cpu-share-demo --cpu-shares 512 nginx:alpine

docker inspect -f '{{.HostConfig.CpuShares}}' cpu-share-demo

docker rm -f cpu-share-demo
Training caution: do not intentionally stress memory on shared or production machines. Out-of-memory tests can affect host stability.

5. Container Networking

Docker networks allow containers to communicate with each other, the host, and external systems.

app1
container
app2
container
Docker Host

5.1 Bridge Network

The bridge driver is used for containers on the same Docker host. User-defined bridge networks provide built-in DNS resolution by container name.

# Create a user-defined bridge network
docker network create app-net

# Run two containers on the same network
docker run -d --name web-a --network app-net nginx:alpine
docker run --rm --name test-client --network app-net alpine:latest sh -c "wget -qO- http://web-a"

# Inspect network
docker network inspect app-net

# Cleanup
docker rm -f web-a
docker network rm app-net
Validation: the wget command should print the NGINX welcome page because web-a resolves by container name.

5.2 Host Network

Host networking removes Docker network isolation and lets the container use the host network namespace directly.

# Linux only: host network mode
docker run --rm --network host nginx:alpine
Important: host networking behaves differently on Docker Desktop compared with native Linux. For classroom labs, prefer Linux hosts or explain platform limitations.

5.3 Overlay Network

Overlay networks are used for communication across multiple Docker daemon hosts. In Docker Engine, a practical overlay lab normally uses Swarm mode.

# Single-machine Swarm demo. Do this only on a lab machine.
docker swarm init

# Create overlay network
docker network create -d overlay app-overlay

# Create a service attached to overlay network
docker service create --name web-svc --network app-overlay -p 8080:80 nginx:alpine

# Validate
docker service ls
curl http://localhost:8080

# Cleanup
docker service rm web-svc
docker network rm app-overlay
docker swarm leave --force
Do not run docker swarm init casually on production Docker hosts. It changes the Docker host into a Swarm manager.

Useful network commands

docker network ls
docker network inspect bridge
docker network create mynet
docker network connect mynet CONTAINER_NAME
docker network disconnect mynet CONTAINER_NAME
docker network rm mynet

6. Container Volumes

Container writable layers are temporary. For important data, use volumes or bind mounts.

Host Storage
volume / folder
Container
/data mounted

6.1 Bind Mounts

A bind mount maps a known host path into a container.

mkdir -p ~/docker-bind-demo
echo "Hello from host" > ~/docker-bind-demo/index.html

docker run --rm -p 8081:80 \
  --mount type=bind,source=$HOME/docker-bind-demo,target=/usr/share/nginx/html,readonly \
  nginx:alpine

# In another terminal:
curl http://localhost:8081
Validation: browser or curl should show Hello from host.

6.2 Named Volumes

A named volume is managed by Docker and can be reused by different containers.

docker volume create app-data

# Write data into the volume
docker run --rm -v app-data:/data alpine sh -c "echo persistent-data > /data/message.txt"

# Read data from a new container
docker run --rm -v app-data:/data alpine cat /data/message.txt

# Inspect volume
docker volume inspect app-data

# Cleanup only after lab is complete
docker volume rm app-data

6.3 Anonymous Volumes

An anonymous volume is created without an explicit name. It is useful for temporary data but can become confusing if not cleaned up.

docker run --name anon-demo -v /data alpine sh -c "echo temp-data > /data/file.txt"

docker inspect anon-demo --format '{{json .Mounts}}'

docker rm -f anon-demo

# Remove unused anonymous volumes
docker volume prune
Best practice: use named volumes for persistent data. Use bind mounts for development when the exact host path matters.

7. Managing Container Data Persistence

Data written inside a container disappears when the container is removed unless it is stored in a volume or bind mount.

Proof: data lost without volume

docker run --name no-persist alpine sh -c "echo lost-data > /tmp/data.txt && cat /tmp/data.txt"
docker rm no-persist

# New container will not have the previous /tmp/data.txt
docker run --rm alpine sh -c "ls /tmp/data.txt || echo 'file not found'"

Proof: data survives with named volume

docker volume create persist-vol

docker run --rm -v persist-vol:/appdata alpine sh -c "echo saved-data > /appdata/data.txt"
docker run --rm -v persist-vol:/appdata alpine cat /appdata/data.txt

docker volume rm persist-vol

Backup and restore a named volume

# Create sample data
docker volume create backup-demo
docker run --rm -v backup-demo:/data alpine sh -c "echo important > /data/file.txt"

# Backup volume to tar file in current directory
docker run --rm \
  -v backup-demo:/data \
  -v "$PWD":/backup \
  alpine tar czf /backup/backup-demo.tar.gz -C /data .

# Restore into new volume
docker volume create restore-demo
docker run --rm \
  -v restore-demo:/data \
  -v "$PWD":/backup \
  alpine tar xzf /backup/backup-demo.tar.gz -C /data

# Validate restore
docker run --rm -v restore-demo:/data alpine cat /data/file.txt

# Cleanup
docker volume rm backup-demo restore-demo
rm -f backup-demo.tar.gz

9. Troubleshooting Guide

ProblemLikely CauseFix
permission denied /var/run/docker.sockUser is not allowed to access Docker daemon.Use sudo or add user to docker group, then re-login or run newgrp docker.
Container exits immediatelyMain process completed.Use long-running process such as NGINX or run interactive shell with -it.
Port already allocatedHost port is already in use.Use another host port, for example -p 8081:80, or stop the process using the port.
Cannot resolve container by nameContainers are on default bridge or different networks.Create user-defined bridge network and attach both containers.
Volume cannot be removedVolume is still used by a container.Remove dependent container first: docker rm -f CONTAINER.
Overlay network command failsSwarm mode not enabled.Use docker swarm init on a lab machine, then create overlay network.

Cleanup all lab leftovers

# Carefully remove containers created in labs
docker rm -f web1 web-created web-bg life-demo exec-demo restart-demo mem-demo cpu-demo cpu-share-demo lab-nginx lab-life lab-limit lab-web bind-web 2>/dev/null || true

# Remove networks created in labs
docker network rm app-net lab-net 2>/dev/null || true

# Remove volumes created in labs
docker volume rm app-data persist-vol backup-demo restore-demo lab-data 2>/dev/null || true

# Optional: remove all unused objects. Use carefully.
docker system prune

10. Quick Command Cheat Sheet

docker run IMAGE                         # create and run container foreground
docker run -d IMAGE                      # detached/background
docker run --name NAME IMAGE             # assign name
docker run -p HOSTPORT:CONTAINERPORT IMAGE
docker ps                               # running containers
docker ps -a                             # all containers
docker logs CONTAINER                    # logs
docker exec -it CONTAINER sh             # shell inside container
docker stop CONTAINER                    # graceful stop
docker start CONTAINER                   # start stopped container
docker restart CONTAINER                 # restart
docker rm CONTAINER                      # remove stopped container
docker rm -f CONTAINER                   # force remove running container
docker inspect CONTAINER                 # detailed metadata
docker stats                             # live resource usage
docker network ls / inspect / create / rm
docker volume ls / inspect / create / rm

11. Trainer Validation Checklist

  • Run docker version and confirm client and server sections are visible.
  • Run docker run hello-world before starting the lab.
  • Verify ports 8080, 8081, and 8082 are free or change the examples.
  • For Linux trainees, fix Docker socket permission before class.
  • Do not run Swarm overlay lab on production or shared Docker hosts.
  • Use docker system prune only after confirming no important local containers/images are needed.
