Helm Reference Guide for Kubernetes Trainees

This standalone guide explains Helm concepts with animated visual explanations and hands-on labs. It is designed for classroom delivery using a local Kubernetes cluster such as Kind, Minikube, Docker Desktop Kubernetes, or any disposable training cluster.

1. What Helm Solves

Main idea: Helm is the package manager for Kubernetes. Instead of maintaining many raw YAML files for Deployment, Service, ConfigMap, Ingress, RBAC, and other objects separately, Helm packages them into a versioned chart and installs a running instance called a release.

ChartTemplates + defaults
ValuesEnvironment settings
Rendered YAMLFinal manifests
ReleaseObjects in cluster
Chart
A reusable package containing templates, default values, metadata, tests, hooks, and optional dependencies.
Release
An installed instance of a chart. The same chart can be installed many times using different release names and values.
Revision
Each upgrade creates a new revision. Helm can inspect history and roll back to older revisions.

Helm object flow

StageCommandWhat happens
Validate charthelm lint ./chartChecks chart structure and common template problems.
Render locallyhelm template app ./chartRenders templates into plain Kubernetes YAML without installing.
Installhelm install app ./chartCreates Kubernetes objects and stores release metadata.
Upgradehelm upgrade app ./chartApplies changed templates or values and creates a new revision.
Rollbackhelm rollback app 1Reverts to a previous revision.
Uninstallhelm uninstall appDeletes release-managed resources unless a resource has keep policy.

2. Installing Helm

Use the official Helm binary or script for production-like training. Package-manager installs are convenient but may lag behind the latest release.

Trainer note: Use a disposable lab cluster. Do not run these labs directly in a production namespace.

Prerequisites

  • kubectl installed and configured.
  • A Kubernetes cluster is reachable through the current kubeconfig context.
  • Linux/macOS shell or Windows PowerShell/Git Bash.
  • Internet access for pulling container images and public charts.

Check Kubernetes access first

Check cluster context 
kubectl config current-context
kubectl get nodes
kubectl cluster-info

Install Helm on Linux using the official script

The current official Helm documentation provides a Helm 4 installer script. Many enterprises still standardize on Helm 3; both use the same core lab workflow below.

3. Using Helm

Helm is used in two broad modes: consuming existing charts and developing your own charts.

Common commands

CommandPurposeExample
helm repo addAdd a classic chart repository.helm repo add bitnami https://charts.bitnami.com/bitnami
helm repo updateRefresh local chart repository index.helm repo update
helm search repoSearch charts in added repositories.helm search repo nginx
helm show valuesView default configurable values.helm show values bitnami/nginx
helm installInstall a new release.helm install web bitnami/nginx -n web --create-namespace
helm upgrade --installInstall if missing, upgrade if present.helm upgrade --install web ./chart -n web
helm listList releases.helm list -A
helm statusInspect one release.helm status web -n web
helm historySee release revisions.helm history web -n web
helm rollbackRollback to an older revision.helm rollback web 1 -n web
helm uninstallDelete a release.helm uninstall web -n web

Safe install pattern

Render and dry-run before real install 
helm lint ./mychart
helm template myapp ./mychart --namespace demo --debug
helm install myapp ./mychart --namespace demo --create-namespace --dry-run --debug
helm install myapp ./mychart --namespace demo --create-namespace --wait
Best classroom habit: Make trainees run helm lint, helm template, and helm install --dry-run --debug before every real install or upgrade.

4. Charts

A chart is a directory with metadata, templates, values, optional dependencies, optional CRDs, tests, and helpers.

Typical chart structure

Chart directory layout 
mychart/
  Chart.yaml             # Chart metadata: name, version, appVersion, type
  values.yaml            # Default configuration values
  values.schema.json     # Optional schema validation for values.yaml
  charts/                # Optional packaged dependencies/subcharts
  crds/                  # Optional CustomResourceDefinitions
  templates/             # Kubernetes templates
    _helpers.tpl         # Reusable helper templates; not rendered as a manifest
    deployment.yaml
    service.yaml
    configmap.yaml
    tests/
      test-connection.yaml

Important files

File/folderPurposeCommon mistake
Chart.yamlDefines chart name and version.Forgetting to bump version before packaging/releasing.
values.yamlStores default configuration.Putting environment-specific secrets directly in defaults.
templates/Stores Kubernetes YAML templates.Broken indentation after Go template rendering.
_helpers.tplStores reusable naming and label templates.Hardcoding names instead of using helpers.
templates/tests/Stores Helm test workloads.Writing tests that never exit.
charts/Stores dependencies.Vendoring dependencies without version control.
crds/Installs CRDs before templates.Expecting Helm to upgrade/delete CRDs automatically like normal resources.

Values override order

Chart defaults
values.yaml
File override
-f dev.yaml
More files
-f prod.yaml
CLI override
--set
Rendered YAML
Kubernetes API
Example values override 
helm upgrade --install web ./hello-helm \
  --namespace helm-lab \
  --create-namespace \
  -f values-dev.yaml \
  --set replicaCount=2 \
  --set message="Hello from CLI override" \
  --wait

5. Chart Development Tips and Tricks

Use helper templates.
Put reusable names and labels in templates/_helpers.tpl. This keeps manifests consistent.
Quote strings, not integers.
Use | quote for string values. Keep numeric fields such as ports and replica counts unquoted.
Use required for mandatory values.
Fail template rendering early when a required value is missing.
Use checksum annotations.
Trigger Deployment rollout when a ConfigMap or Secret template changes.
Prefer upgrade --install.
It simplifies CI/CD because the same command works for first deployment and later deployment.
Avoid random values during upgrades.
Randomly generated template values can cause unnecessary resource changes on every upgrade.

Good template patterns

Use required and quote correctly 
env:
  - name: APP_MODE
    value: {{ required "appMode is required" .Values.appMode | quote }}
ports:
  - containerPort: {{ .Values.service.port }}
Roll pods when ConfigMap changes 
spec:
  template:
    metadata:
      annotations:
        checksum/config: {{ include (print $.Template.BasePath "/configmap.yaml") . | sha256sum }}

Anti-patterns to avoid

Anti-patternWhy it hurtsBetter approach
Hardcoding namespace inside templatesBreaks installs into different namespaces.Use helm install -n namespace and .Release.Namespace only when needed.
Hardcoding release namesPrevents multiple releases of the same chart.Use helpers based on .Release.Name.
Using latest image tags in productionUpgrades become unpredictable.Pin application image tags.
Skipping helm templateYAML errors appear only during install.Render locally first.
Storing real secrets in values.yamlLeaks secrets through Git.Use external secret managers or sealed/encrypted secrets.

6. Syncing Your Chart Repository

A classic Helm chart repository is a simple HTTP location that serves index.yaml and chart packages such as hello-helm-0.1.0.tgz. Helm clients download index.yaml to discover chart versions.

Package chart
helm package
Generate index
helm repo index
Publish HTTP/GitHub Pages/S3/GCS

Local repository demo

Create and serve a local chart repo 
mkdir -p repo
helm package ./hello-helm --destination repo
helm repo index repo --url http://127.0.0.1:8879
python3 -m http.server 8879 --directory repo

Open a second terminal:

Consume local chart repo 
helm repo add localrepo http://127.0.0.1:8879
helm repo update
helm search repo localrepo
helm install local-demo localrepo/hello-helm \
  --namespace helm-lab \
  --create-namespace \
  --wait

Repository sync patterns

TargetPatternTypical command
GitHub PagesCommit chart source to main; publish packaged chart/index to gh-pages.Automate using Chart Releaser Action.
GCS bucketGenerate index locally, then sync to bucket.gsutil rsync -d local-dir gs://bucket-name
AWS S3Generate index locally, then sync to S3 static website or CloudFront.aws s3 sync repo/ s3://bucket-name/
OCI registryPackage chart and push to registry using oci://.helm push chart.tgz oci://registry.example.com/charts
Classic repo vs OCI: Classic repositories use index.yaml. OCI registries use registry APIs and oci:// references. For modern private distribution, OCI is often cleaner; for GitHub Pages demos, classic repositories are easier to teach.

7. Chart Releaser Action to Automate GitHub Page Charts

Chart Releaser Action turns a GitHub repository into a self-hosted Helm chart repository using GitHub Releases and GitHub Pages. Recommended repository layout:

GitHub chart repository layout 
helm-charts/
  charts/
    hello-helm/
      Chart.yaml
      values.yaml
      templates/
  .github/
    workflows/
      release.yml

GitHub Actions workflow

.github/workflows/release.yml 
name: Release Charts

on:
  push:
    branches:
      - main

jobs:
  release:
    permissions:
      contents: write
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4
        with:
          fetch-depth: 0

      - name: Configure Git
        run: |
          git config user.name "$GITHUB_ACTOR"
          git config user.email "$GITHUB_ACTOR@users.noreply.github.com"

      - name: Run chart-releaser
        uses: helm/chart-releaser-action@v1.7.0
        env:
          CR_TOKEN: "${{ secrets.GITHUB_TOKEN }}"

Repository setup checklist

  1. Create a GitHub repository such as helm-charts.
  2. Place all chart sources under charts/.
  3. Enable GitHub Pages from the gh-pages branch after the first successful release.
  4. Bump Chart.yaml version before each chart release.
  5. Push to main. The workflow creates release assets and updates index.yaml.
  6. Consumers install charts using helm repo add <alias> https://<org>.github.io/helm-charts.

Consumer commands after publishing

Install from published GitHub Pages chart repo 
helm repo add mycharts https://YOUR_ORG.github.io/helm-charts
helm repo update
helm search repo mycharts
helm install demo mycharts/hello-helm --namespace helm-lab --create-namespace --wait
Common failure: Chart Releaser skips a chart if the chart version already exists as a GitHub release. Increment version in Chart.yaml for every new release.

8. Chart Hooks

Hooks let chart authors run Kubernetes resources at specific points of a release lifecycle, such as before install, after install, before upgrade, after rollback, or during helm test.

Render templates
pre-install hook
Create resources
post-install hook
Release ready
helm test

Common hook annotations

AnnotationPurpose
helm.sh/hook: pre-installRun after rendering but before normal resources are created.
helm.sh/hook: post-installRun after normal resources are created.
helm.sh/hook: pre-upgradeRun before upgrade changes are applied.
helm.sh/hook: post-upgradeRun after upgrade changes are applied.
helm.sh/hook: pre-deleteRun before release resources are deleted.
helm.sh/hook: testRun when helm test RELEASE is invoked.
helm.sh/hook-weightControls execution order. Lower weight runs first.
helm.sh/hook-delete-policyControls when hook resources are deleted.

Example pre-install hook Job

templates/preinstall-job.yaml 
apiVersion: batch/v1
kind: Job
metadata:
  name: "{{ include "hello-helm.fullname" . }}-preinstall"
  annotations:
    "helm.sh/hook": pre-install
    "helm.sh/hook-weight": "-5"
    "helm.sh/hook-delete-policy": before-hook-creation,hook-succeeded
spec:
  template:
    spec:
      restartPolicy: Never
      containers:
        - name: preinstall
          image: busybox:1.36
          command: ["sh", "-c", "echo pre-install hook running for {{ .Release.Name }}"]
  backoffLimit: 1
Use hooks sparingly. Hooks are powerful, but they can make upgrades harder to reason about. For normal application resources, prefer regular templates.

9. Chart Tests

A Helm chart test is a Kubernetes workload, commonly a Pod or Job, annotated with helm.sh/hook: test. Helm considers the test successful when the workload exits with status code 0.

Example test Pod

templates/tests/test-connection.yaml 
apiVersion: v1
kind: Pod
metadata:
  name: "{{ include "hello-helm.fullname" . }}-test"
  labels:
    {{- include "hello-helm.labels" . | nindent 4 }}
  annotations:
    "helm.sh/hook": test
spec:
  restartPolicy: Never
  containers:
    - name: wget
      image: busybox:1.36
      command: ["wget"]
      args: ["-qO-", "http://{{ include "hello-helm.fullname" . }}:{{ .Values.service.port }}"]

Run tests

Test an installed release 
helm test demo --namespace helm-lab
kubectl get pods --namespace helm-lab
kubectl logs --namespace helm-lab pod/demo-hello-helm-test
Good tests for trainees: connection test, configuration injection test, service DNS test, and application health endpoint test.

Lab 2: Creating Helm Charts from Scratch to Deploy Application

Goal: Create a simple NGINX-based web application chart from scratch, render it, lint it, install it, test it, upgrade it, package it, publish it locally, and clean up.

Create chart directories

Directories 
mkdir -p hello-helm/templates/tests
cd hello-helm

Create Chart.yaml

Chart.yaml 
cat > Chart.yaml <<'EOF'
apiVersion: v2
name: hello-helm
description: A simple NGINX application deployed with Helm
type: application
version: 0.1.0
appVersion: "1.27"
EOF

Create values.yaml

values.yaml 
cat > values.yaml <<'EOF'
replicaCount: 1

image:
  repository: nginx
  tag: "1.27-alpine"
  pullPolicy: IfNotPresent

service:
  type: ClusterIP
  port: 80

message: "Hello from Helm values.yaml"
EOF

Create helper templates

templates/_helpers.tpl 
cat > templates/_helpers.tpl <<'EOF'
{{- define "hello-helm.name" -}}
{{- default .Chart.Name .Values.nameOverride | trunc 63 | trimSuffix "-" -}}
{{- end -}}

{{- define "hello-helm.fullname" -}}
{{- if .Values.fullnameOverride -}}
{{- .Values.fullnameOverride | trunc 63 | trimSuffix "-" -}}
{{- else -}}
{{- $name := default .Chart.Name .Values.nameOverride -}}
{{- if contains $name .Release.Name -}}
{{- .Release.Name | trunc 63 | trimSuffix "-" -}}
{{- else -}}
{{- printf "%s-%s" .Release.Name $name | trunc 63 | trimSuffix "-" -}}
{{- end -}}
{{- end -}}
{{- end -}}

{{- define "hello-helm.labels" -}}
helm.sh/chart: {{ .Chart.Name }}-{{ .Chart.Version | replace "+" "_" }}
app.kubernetes.io/name: {{ include "hello-helm.name" . }}
app.kubernetes.io/instance: {{ .Release.Name }}
app.kubernetes.io/version: {{ .Chart.AppVersion | quote }}
app.kubernetes.io/managed-by: {{ .Release.Service }}
{{- end -}}

{{- define "hello-helm.selectorLabels" -}}
app.kubernetes.io/name: {{ include "hello-helm.name" . }}
app.kubernetes.io/instance: {{ .Release.Name }}
{{- end -}}
EOF

Create ConfigMap for web content

templates/configmap.yaml 
cat > templates/configmap.yaml <<'EOF'
apiVersion: v1
kind: ConfigMap
metadata:
  name: {{ include "hello-helm.fullname" . }}-html
  labels:
    {{- include "hello-helm.labels" . | nindent 4 }}
data:
  index.html: |
    <!doctype html>
    <html>
    <head><title>Hello Helm</title></head>
    <body style="font-family: Arial; padding: 40px;">
      <h1>{{ required "values.message is required" .Values.message }}</h1>
      <p>Release: {{ .Release.Name }}</p>
      <p>Namespace: {{ .Release.Namespace }}</p>
      <p>Chart: {{ .Chart.Name }} {{ .Chart.Version }}</p>
    </body>
    </html>
EOF

Create Deployment

templates/deployment.yaml 
cat > templates/deployment.yaml <<'EOF'
apiVersion: apps/v1
kind: Deployment
metadata:
  name: {{ include "hello-helm.fullname" . }}
  labels:
    {{- include "hello-helm.labels" . | nindent 4 }}
spec:
  replicas: {{ .Values.replicaCount }}
  selector:
    matchLabels:
      {{- include "hello-helm.selectorLabels" . | nindent 6 }}
  template:
    metadata:
      labels:
        {{- include "hello-helm.selectorLabels" . | nindent 8 }}
      annotations:
        checksum/config: {{ include (print $.Template.BasePath "/configmap.yaml") . | sha256sum }}
    spec:
      containers:
        - name: nginx
          image: "{{ .Values.image.repository }}:{{ .Values.image.tag }}"
          imagePullPolicy: {{ .Values.image.pullPolicy }}
          ports:
            - name: http
              containerPort: 80
              protocol: TCP
          volumeMounts:
            - name: web-content
              mountPath: /usr/share/nginx/html
      volumes:
        - name: web-content
          configMap:
            name: {{ include "hello-helm.fullname" . }}-html
EOF

Create Service

templates/service.yaml 
cat > templates/service.yaml <<'EOF'
apiVersion: v1
kind: Service
metadata:
  name: {{ include "hello-helm.fullname" . }}
  labels:
    {{- include "hello-helm.labels" . | nindent 4 }}
spec:
  type: {{ .Values.service.type }}
  ports:
    - port: {{ .Values.service.port }}
      targetPort: http
      protocol: TCP
      name: http
  selector:
    {{- include "hello-helm.selectorLabels" . | nindent 4 }}
EOF

Create pre-install hook Job

templates/preinstall-job.yaml 
cat > templates/preinstall-job.yaml <<'EOF'
apiVersion: batch/v1
kind: Job
metadata:
  name: "{{ include "hello-helm.fullname" . }}-preinstall"
  annotations:
    "helm.sh/hook": pre-install
    "helm.sh/hook-weight": "-5"
    "helm.sh/hook-delete-policy": before-hook-creation,hook-succeeded
spec:
  template:
    spec:
      restartPolicy: Never
      containers:
        - name: preinstall
          image: busybox:1.36
          command: ["sh", "-c", "echo pre-install hook running for {{ .Release.Name }}"]
  backoffLimit: 1
EOF

cat > templates/tests/test-connection.yaml <<'EOF'

10. Troubleshooting Guide

ProblemLikely causeFix
Kubernetes cluster unreachableWrong kubeconfig context or cluster down.Run kubectl config current-context, kubectl get nodes, and switch context if needed.
cannot re-use a name that is still in useRelease name already exists.Use helm upgrade --install, choose another release name, or uninstall old release.
YAML parse errorBad indentation after template rendering.Run helm template --debug and inspect the rendered YAML around the reported line.
Pod stuck in ImagePullBackOffWrong image tag, no internet, proxy issue, or private registry auth missing.Run kubectl describe pod, check image name/tag, configure proxy or imagePullSecret.
Service not reachableService selector does not match pod labels or port-forward targets wrong service.Compare kubectl get svc -o yaml selectors with pod labels.
helm test failsTest Pod cannot reach service or app returns non-200 response.Check kubectl logs pod/...-test and DNS/service name.
Chart Releaser does not publishChart version already released, permissions missing, GitHub Pages not enabled.Bump Chart.yaml version, ensure contents: write, and enable Pages from gh-pages.
Upgrade does not restart pods after ConfigMap changeDeployment Pod template did not change.Add checksum annotation based on the ConfigMap template.
