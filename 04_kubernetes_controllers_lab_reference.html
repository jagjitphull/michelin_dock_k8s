3. Controller Fundamentals

3.1 What is a Kubernetes Controller?

A Kubernetes controller is a control loop. It continuously watches cluster state through the API server and takes action to move the current state closer to the desired state.

Animated Concept: Desired State → Reconciliation → Actual State

YAML Desired Statereplicas: 3
API Servervalidates object
etcdstores desired state
Controller Loopwatches + compares
Pods Runningactual state becomes 3

3.2 Desired State vs Current State

TermMeaningExample
Desired stateWhat the user declares in the Kubernetes object.replicas: 3 in a ReplicaSet or Deployment.
Current stateWhat is actually running in the cluster right now.Only 2 Pods are running because 1 Pod crashed.
ReconciliationThe controller detects the difference and takes corrective action.Controller creates 1 replacement Pod.

3.3 Controller Pattern

The common pattern is:

1. Watch Kubernetes API for object changes.
2. Read desired state from object spec.
3. Read current state from cluster resources.
4. Compare desired state and current state.
5. Create, update, delete, or request changes.
6. Repeat continuously.

4. Various Types of Kubernetes Controllers

ControllerMain purposeCommon use case
ReplicaSetMaintains a stable number of identical Pods.Keep 3 web Pods running.
DeploymentManages ReplicaSets and application rollouts.Rolling update, rollback, scaling.
DaemonSetRuns one Pod on every node or selected nodes.Log agent, monitoring agent, CNI agent.
StatefulSetRuns Pods with stable identity and ordered rollout.Databases, queues, clustered systems.
JobRuns one-time tasks to completion.Batch processing, migration, report generation.
CronJobRuns Jobs repeatedly on a schedule.Nightly backup, cleanup, scheduled report.
HPAScales number of Pods horizontally.Add Pods when CPU usage rises.
VPAAdjusts Pod CPU/memory requests vertically.Right-size resource requests over time.

5. Kube-Controller-Manager

kube-controller-manager is a control plane component that runs many core Kubernetes controllers. Examples include the node controller, job controller, endpoint slice controller, service account controller, deployment controller, ReplicaSet controller, and namespace controller.

Why centralized?

Multiple control loops are compiled into one binary and run as part of the control plane for simpler operation.

Important behavior

It does not run your app containers. It watches API objects and creates or updates other Kubernetes objects.

5.1 Observe controller-created objects

After you create a Deployment, the Deployment controller creates a ReplicaSet, and the ReplicaSet controller creates Pods.

kubectl get deployment,replicaset,pod -n controllers-lab
kubectl describe deployment <deployment-name> -n controllers-lab
kubectl get events -n controllers-lab --sort-by=.metadata.creationTimestamp

7. Cloud-Controller Manager

The cloud-controller-manager separates cloud-provider-specific control loops from the core Kubernetes control plane. It is mainly relevant when Kubernetes runs on public or private cloud infrastructure.

Cloud controllerPurposeExample
Node ControllerChecks and updates cloud-backed Node information.If a VM is deleted from the cloud, the node can be removed or marked unavailable.
Route ControllerConfigures cloud network routes where required.Ensures Pods on different nodes can communicate through cloud routing.
Service ControllerCreates cloud load balancers for Service type: LoadBalancer.Provision an AWS ELB, Azure Load Balancer, or Google Cloud Load Balancer.
Lab note: Local clusters usually do not have a real cloud-controller-manager. For local training, focus on ClusterIP/NodePort/Ingress. Use managed cloud clusters to demonstrate LoadBalancer.

Lab: Deploying Application using ReplicaSet

This lab shows how a ReplicaSet maintains a fixed number of identical Pods.

  1. Create the ReplicaSet manifest.
    apiVersion: apps/v1
kind: ReplicaSet
metadata:
  name: rs-web
  namespace: controllers-lab
  labels:
    app: rs-web
spec:
  replicas: 3
  selector:
    matchLabels:
      app: rs-web
      tier: frontend
  template:
    metadata:
      labels:
        app: rs-web
        tier: frontend
    spec:
      containers:
        - name: nginx
          image: nginx:1.27-alpine
          ports:
            - containerPort: 80
          resources:
            requests:
              cpu: 20m
              memory: 32Mi
            limits:
              cpu: 100m
              memory: 128Mi
  2. Apply and verify.
    kubectl apply -f rs-nginx.yaml
kubectl get rs -n controllers-lab
kubectl get pods -l app=rs-web -n controllers-lab -o wide
kubectl describe rs rs-web -n controllers-lab
  3. Expose the ReplicaSet using a Service.
    kubectl expose rs rs-web --name=rs-web-svc --port=80 --target-port=80 -n controllers-lab
kubectl get svc rs-web-svc -n controllers-lab
kubectl run curl-test --rm -i --restart=Never --image=curlimages/curl:8.10.1 -n controllers-lab --command -- curl -s http://rs-web-svc
  4. Test self-healing.
    POD=$(kubectl get pod -l app=rs-web -n controllers-lab -o jsonpath='{.items[0].metadata.name}')
echo "Deleting pod: $POD"
kubectl delete pod "$POD" -n controllers-lab
kubectl get pods -l app=rs-web -n controllers-lab -w

    Expected result: a replacement Pod is created automatically.

  5. Scale the ReplicaSet.
    kubectl scale rs rs-web --replicas=5 -n controllers-lab
kubectl get rs rs-web -n controllers-lab
kubectl get pods -l app=rs-web -n controllers-lab
  6. Cleanup this lab.
    kubectl delete svc rs-web-svc -n controllers-lab
kubectl delete rs rs-web -n controllers-lab

Lab: Working with Deployments – Rolling

This lab shows Deployment creation, Service exposure, scaling, rollout, revision history, and rollback.

  1. Create a Deployment and Service.
    apiVersion: apps/v1
kind: Deployment
metadata:
  name: rolling-web
  namespace: controllers-lab
  labels:
    app: rolling-web
  annotations:
    kubernetes.io/change-cause: "Initial release: nginx 1.25"
spec:
  replicas: 4
  revisionHistoryLimit: 5
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1
      maxUnavailable: 1
  selector:
    matchLabels:
      app: rolling-web
  template:
    metadata:
      labels:
        app: rolling-web
    spec:
      containers:
        - name: nginx
          image: nginx:1.25-alpine
          ports:
            - containerPort: 80
          readinessProbe:
            httpGet:
              path: /
              port: 80
            initialDelaySeconds: 5
            periodSeconds: 5
          resources:
            requests:
              cpu: 20m
              memory: 32Mi
            limits:
              cpu: 100m
              memory: 128Mi
---
apiVersion: v1
kind: Service
metadata:
  name: rolling-web-svc
  namespace: controllers-lab
  labels:
    app: rolling-web
spec:
  type: ClusterIP
  selector:
    app: rolling-web
  ports:
    - name: http
      port: 80
      targetPort: 80
  2. Apply and verify rollout.
    kubectl apply -f rolling-deployment.yaml
kubectl rollout status deployment/rolling-web -n controllers-lab
kubectl get deploy,rs,pod,svc -l app=rolling-web -n controllers-lab
kubectl run curl-test --rm -i --restart=Never --image=curlimages/curl:8.10.1 -n controllers-lab --command -- curl -I http://rolling-web-svc
  3. Scale the Deployment.
    kubectl scale deployment rolling-web --replicas=6 -n controllers-lab
kubectl rollout status deployment/rolling-web -n controllers-lab
kubectl get deploy rolling-web -n controllers-lab
kubectl get pods -l app=rolling-web -n controllers-lab -o wide
  4. Update the Deployment image.
    kubectl annotate deployment rolling-web kubernetes.io/change-cause="Update nginx to 1.27-alpine" -n controllers-lab --overwrite
kubectl set image deployment/rolling-web nginx=nginx:1.27-alpine -n controllers-lab
kubectl rollout status deployment/rolling-web -n controllers-lab
kubectl get rs -l app=rolling-web -n controllers-lab
kubectl rollout history deployment/rolling-web -n controllers-lab
  5. Inspect revision details.
    kubectl rollout history deployment/rolling-web --revision=1 -n controllers-lab
kubectl rollout history deployment/rolling-web --revision=2 -n controllers-lab
kubectl describe deployment rolling-web -n controllers-lab | sed -n '/Events:/,$p'
  6. Rollback to the previous revision.
    kubectl rollout undo deployment/rolling-web -n controllers-lab
kubectl rollout status deployment/rolling-web -n controllers-lab
kubectl get rs -l app=rolling-web -n controllers-lab
kubectl describe deployment rolling-web -n controllers-lab | grep -i Image -A2
  7. Cleanup this lab.
    kubectl delete deployment rolling-web -n controllers-lab
kubectl delete service rolling-web-svc -n controllers-lab

Lab: Deploying Application using DaemonSet

This lab deploys a simple node agent. A DaemonSet is appropriate for node-local agents such as log collectors, metrics collectors, and network plugins.

  1. Create the DaemonSet manifest.
    apiVersion: apps/v1
kind: DaemonSet
metadata:
  name: node-agent
  namespace: controllers-lab
  labels:
    app: node-agent
spec:
  selector:
    matchLabels:
      app: node-agent
  template:
    metadata:
      labels:
        app: node-agent
    spec:
      tolerations:
        - key: node-role.kubernetes.io/control-plane
          operator: Exists
          effect: NoSchedule
        - key: node-role.kubernetes.io/master
          operator: Exists
          effect: NoSchedule
      containers:
        - name: agent
          image: busybox:1.36
          command:
            - /bin/sh
            - -c
            - |
              while true; do
                echo "$(date) node-agent running on ${NODE_NAME}";
                sleep 30;
              done
          env:
            - name: NODE_NAME
              valueFrom:
                fieldRef:
                  fieldPath: spec.nodeName
          resources:
            requests:
              cpu: 10m
              memory: 16Mi
            limits:
              cpu: 50m
              memory: 64Mi
  2. Apply and verify one Pod per schedulable node.
    kubectl apply -f daemonset-node-agent.yaml
kubectl get ds node-agent -n controllers-lab
kubectl get pods -l app=node-agent -n controllers-lab -o wide
kubectl logs -l app=node-agent -n controllers-lab --tail=10
  3. Compare nodes and DaemonSet Pods.
    kubectl get nodes
kubectl get pods -l app=node-agent -n controllers-lab --sort-by=.spec.nodeName -o custom-columns=NAME:.metadata.name,NODE:.spec.nodeName,STATUS:.status.phase
  4. Cleanup this lab.
    kubectl delete ds node-agent -n controllers-lab

Mini Lab: StatefulSet – Stable Pod Identity

A StatefulSet gives Pods predictable names such as stateful-nginx-0 and stateful-nginx-1. For real stateful applications, add persistent volumes and proper backup/restore design.

  1. Create a headless Service and StatefulSet.
    apiVersion: v1
kind: Service
metadata:
  name: stateful-nginx
  namespace: controllers-lab
  labels:
    app: stateful-nginx
spec:
  clusterIP: None
  selector:
    app: stateful-nginx
  ports:
    - name: web
      port: 80
      targetPort: 80
---
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: stateful-nginx
  namespace: controllers-lab
spec:
  serviceName: stateful-nginx
  replicas: 2
  selector:
    matchLabels:
      app: stateful-nginx
  template:
    metadata:
      labels:
        app: stateful-nginx
    spec:
      containers:
        - name: nginx
          image: nginx:1.27-alpine
          ports:
            - name: web
              containerPort: 80
  2. Apply and verify stable Pod names.
    kubectl apply -f statefulset-nginx.yaml
kubectl rollout status statefulset/stateful-nginx -n controllers-lab

kubectl get pod stateful-nginx-0 -n controllers-lab -o wide
kubectl get pod stateful-nginx-1 -n controllers-lab -o wide
  3. Delete one Pod and observe that the same identity returns.
    kubectl delete pod stateful-nginx-0 -n controllers-lab

  4. Cleanup this lab.
    kubectl delete -f statefulset-nginx.yaml

7.1 Job – Run a one-time task

  1. Create the Job.
    apiVersion: batch/v1
kind: Job
metadata:
  name: batch-job
  namespace: controllers-lab
spec:
  completions: 3
  parallelism: 1

7.2 CronJob – Run recurring tasks

  1. Create the CronJob.
    apiVersion: batch/v1
kind: CronJob
metadata:
  name: hello-cron
  namespace: controllers-lab
spec:
  schedule: "*/2 * * * *"
  concurrencyPolicy: Forbid
  successfulJobsHistoryLimit: 2
  failedJobsHistoryLimit: 1
  jobTemplate:
    spec:
      template:
        spec:
          restartPolicy: OnFailure
          containers:
            - name: task
              image: busybox:1.36
              command:
                - /bin/sh
                - -c
                - |
                  echo "CronJob triggered at $(date)";
  2. Apply and verify schedule.
    kubectl apply -f cronjob.yaml
kubectl get cronjob hello-cron -n controllers-lab
kubectl get jobs -n controllers-lab --watch
  3. Trigger a manual Job from the CronJob template.
    kubectl create job manual-hello --from=cronjob/hello-cron -n controllers-lab
kubectl wait --for=condition=complete job/manual-hello -n controllers-lab --timeout=120s
kubectl logs -l job-name=manual-hello -n controllers-lab
  4. Cleanup the CronJob lab.
    kubectl delete cronjob hello-cron -n controllers-lab
kubectl delete job manual-hello -n controllers-lab --ignore-not-found=true

Lab: Autoscaling of Pods – HPA

This lab creates a CPU-based HorizontalPodAutoscaler. It requires Metrics Server.

  1. Confirm Metrics API works.
    kubectl top nodes
kubectl top pods -n controllers-lab
  2. Create the sample app, Service, and HPA.
    apiVersion: apps/v1
kind: Deployment
metadata:
  name: php-apache
  namespace: controllers-lab
spec:
  replicas: 1
  selector:
    matchLabels:
      run: php-apache
  template:
    metadata:
      labels:
        run: php-apache
    spec:
      containers:
        - name: php-apache
          image: registry.k8s.io/hpa-example
          ports:
            - containerPort: 80
          resources:
            requests:
              cpu: 200m
              memory: 64Mi
            limits:
              cpu: 500m
              memory: 128Mi
---
apiVersion: v1
kind: Service
metadata:
  name: php-apache
  namespace: controllers-lab
  labels:
    run: php-apache
spec:
  selector:
    run: php-apache
  ports:
    - name: http
      port: 80
      targetPort: 80
---
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: php-apache
  namespace: controllers-lab
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: php-apache
  minReplicas: 1
  maxReplicas: 10
  metrics:
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 50
  3. Apply and verify HPA status.
    kubectl apply -f hpa-php-apache.yaml
kubectl rollout status deployment/php-apache -n controllers-lab
kubectl get deploy,svc,hpa -n controllers-lab
kubectl describe hpa php-apache -n controllers-lab
  4. Generate load.

    Run this in a separate terminal and keep it running for a few minutes.

    kubectl run load-generator --rm -i --restart=Never --image=busybox:1.36 -n controllers-lab --command -- /bin/sh -c "while true; do wget -q -O- http://php-apache; done"
  5. Watch HPA scale Pods.
    kubectl get hpa php-apache -n controllers-lab --watch
kubectl get pods -l run=php-apache -n controllers-lab -w
kubectl describe hpa php-apache -n controllers-lab
  6. Stop load and observe scale-down.

    Stop the load generator with Ctrl+C. HPA scale-down can take several minutes because Kubernetes stabilizes scaling decisions.

    kubectl get hpa php-apache -n controllers-lab --watch
  7. Cleanup this lab.
    kubectl delete -f hpa-php-apache.yaml
kubectl delete pod load-generator -n controllers-lab --ignore-not-found=true
If HPA does not scale: check kubectl top pods, resource requests, HPA events, image pull status, and whether the load generator is actually hitting the Service.

9. Deployment Object Best Practices

Use declarative management

Prefer YAML stored in Git over manual one-off imperative commands. It supports review, repeatability, rollback, and audit.

Avoid Recreate for production web apps

Recreate intentionally causes downtime. Use RollingUpdate unless the application truly cannot run old and new versions together.

Define readiness probes

Readiness probes prevent traffic from being sent to Pods before they are ready.

Set resource requests

HPA CPU utilization requires CPU requests. Requests also help the scheduler place Pods correctly.

Use labels consistently

Services and controllers depend on selectors. Avoid accidental label overlap between workloads.

Keep revision history reasonable

Use revisionHistoryLimit to keep rollback history without accumulating unnecessary old ReplicaSets.

9.1 Safe Deployment workflow

# 1. Validate generated YAML client-side
kubectl apply --dry-run=client -f rolling-deployment.yaml

# 2. Apply declaratively
kubectl apply -f rolling-deployment.yaml

# 3. Wait for rollout
kubectl rollout status deployment/rolling-web -n controllers-lab

# 4. Verify Pods, Service, and events
kubectl get deploy,rs,pod,svc -n controllers-lab
kubectl get events -n controllers-lab --sort-by=.metadata.creationTimestamp

# 5. Roll back if needed
kubectl rollout undo deployment/rolling-web -n controllers-lab
