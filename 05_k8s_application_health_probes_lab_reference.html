Kubernetes Application Health

Understand how Kubernetes checks whether an application is alive, ready for traffic, and safely started before aggressive health checks begin.

✅ LivenessProbe ✅ ReadinessProbe ✅ StartupProbe ✅ Hands-on Labs ✅ Animated Concepts

1. Learning Objectives

By the end of this reference lab, trainees should be able to explain and implement application health checks in Kubernetes.

  • Explain why a running container is not always a healthy application.
  • Use livenessProbe to restart a stuck or broken container.
  • Use readinessProbe to stop sending traffic to an unready Pod.
  • Use startupProbe for slow-starting applications.
  • Verify probe behavior using kubectl get, kubectl describe, logs, events, Endpoints and EndpointSlices.
Lab use case: a small Python web application exposes /live and /ready. Trainees intentionally break readiness and liveness by deleting files inside the container, then observe how Kubernetes reacts.

LivenessProbe Is the app still alive, or is it stuck/deadlocked? Kubelet restarts the container. App process is running, but the application is frozen.

Important: Kubernetes does not automatically know whether your business application is healthy. You must expose a suitable endpoint or command that tells Kubernetes the truth.

3. Application LivenessProbe

3.1 What is LivenessProbe and why do you need it?

A livenessProbe tells Kubernetes whether the container should continue running. It is mainly used to recover from a broken internal state, such as a deadlock, frozen thread, or application that no longer responds correctly.

Use liveness when restart is the correct recovery action. If the app only needs time to warm up or temporarily cannot serve traffic, use readiness instead.

3.2 Implementing LivenessProbe

Kubernetes supports different probe mechanisms. HTTP is common for web apps; exec is useful for simple command checks; TCP checks only verify whether a port accepts connections.

HTTP liveness

YAML snippet 
livenessProbe:
  httpGet:
    path: /live
    port: 8080
  initialDelaySeconds: 5
  periodSeconds: 10
  timeoutSeconds: 2
  failureThreshold: 3

Exec liveness

YAML snippet 
livenessProbe:
  exec:
    command:
      - cat
      - /tmp/healthy
  periodSeconds: 5
  failureThreshold: 2

TCP liveness

YAML snippet 
livenessProbe:
  tcpSocket:
    port: 8080
  periodSeconds: 10
  failureThreshold: 3
Common mistake: Do not make liveness depend on external systems like a database, payment gateway, or another microservice. If that external system has a short outage, you can accidentally restart all application Pods and make the outage worse.

4. Application ReadinessProbe

4.1 Why do you need ReadinessProbe?

A readinessProbe tells Kubernetes whether the Pod is ready to receive traffic. When readiness fails, the container is not restarted. Instead, the Pod is marked NotReady and removed from Service endpoints until the probe succeeds again.

4.2 Implementing ReadinessProbe

Readiness probe example 
readinessProbe:
  httpGet:
    path: /ready
    port: 8080
  initialDelaySeconds: 2
  periodSeconds: 5
  timeoutSeconds: 2
  failureThreshold: 2
  successThreshold: 1
ScenarioShould readiness pass?Reason
App process is started but cache is still loadingNoIt cannot safely serve users yet.
App is connected to required local dependencies and can serve requestsYesIt is ready for Service traffic.
App is doing graceful shutdownNoIt should drain traffic before termination.
App is alive but temporarily overloadedUsually NoReadiness can protect users from bad responses while the Pod recovers.

5. Using ReadinessProbe and LivenessProbe Together

Use both when you need separate answers for two different questions:

  • Liveness: Should Kubernetes restart this container?
  • Readiness: Should Kubernetes send traffic to this Pod?
ConditionLivenessReadinessKubernetes behavior
App is healthyPassPassPod receives traffic.
App is warming upPassFailPod runs but receives no Service traffic.
App is deadlockedFailUsually FailContainer is restarted after configured failure threshold.
External dependency is downUsually PassMaybe FailDo not restart unless the app itself is broken.
Recommended pattern: make liveness less aggressive than readiness. Readiness should fail first to stop traffic. Liveness should restart only when the app cannot recover by itself.

6. Application StartupProbe

A startupProbe is useful for applications that take a long time to start. While startup probe is running and has not yet succeeded, Kubernetes does not use liveness or readiness checks for that container.

Startup probe example 
startupProbe:
  httpGet:
    path: /live
    port: 8080
  periodSeconds: 5
  failureThreshold: 12

In this example, Kubernetes allows up to about 60 seconds for startup: periodSeconds 5 × failureThreshold 12. After startup succeeds, normal liveness and readiness probes take over.

When to use StartupProbe: Java/Spring Boot apps, heavy initialization, ML model loading, legacy apps, apps that run migrations, or applications with unpredictable cold starts.

7. Lab: Working with Readiness and Liveness Probes

This lab deploys three replicas of a small web app. It becomes ready after 20 seconds. You will manually break readiness and liveness to observe Kubernetes behavior.

7.1 Prerequisites

  • A working Kubernetes cluster: Minikube, Kind, kubeadm, Docker Desktop Kubernetes, OpenShift local, or a managed cluster.
  • kubectl configured to access the cluster.
  • Internet access from nodes to pull python:3.12-alpine and curlimages/curl.

Create a namespace

Command 
kubectl create namespace health-lab

Create the application manifest

Save the following as health-demo.yaml.

health-demo.yaml 
apiVersion: apps/v1
kind: Deployment
metadata:
  name: health-demo
  namespace: health-lab
spec:
  replicas: 3
  selector:
    matchLabels:
      app: health-demo
  template:
    metadata:
      labels:
        app: health-demo
    spec:
      containers:
        - name: app
          image: python:3.12-alpine
          imagePullPolicy: IfNotPresent
          ports:
            - containerPort: 8080
          command:
            - /bin/sh
            - -c
          args:
            - |
              cat > /tmp/server.py <<'PY'
              from http.server import BaseHTTPRequestHandler, HTTPServer
              import os
              import socket

              class Handler(BaseHTTPRequestHandler):
                  def log_message(self, fmt, *args):
                      print("%s - %s" % (self.path, fmt % args), flush=True)

                  def send_text(self, status, text):
                      self.send_response(status)
                      self.send_header("Content-Type", "text/plain")
                      self.end_headers()
                      self.wfile.write(text.encode())

                  def do_GET(self):
                      if self.path == "/live":
                          if os.path.exists("/tmp/live"):
                              self.send_text(200, "live\n")
                          else:
                              self.send_text(500, "not live\n")
                      elif self.path == "/ready":
                          if os.path.exists("/tmp/ready"):
                              self.send_text(200, "ready\n")
                          else:
                              self.send_text(503, "not ready\n")
                      else:
                          self.send_text(200, "hello from %s\n" % socket.gethostname())

              open("/tmp/live", "w").close()
              os.system("(sleep 20; touch /tmp/ready) &")
              HTTPServer(("0.0.0.0", 8080), Handler).serve_forever()
              PY
              python /tmp/server.py
          readinessProbe:
            httpGet:
              path: /ready
              port: 8080
            initialDelaySeconds: 2
            periodSeconds: 5
            timeoutSeconds: 2
            failureThreshold: 2
            successThreshold: 1
          livenessProbe:
            httpGet:
              path: /live
              port: 8080
            initialDelaySeconds: 5
            periodSeconds: 5
            timeoutSeconds: 2
            failureThreshold: 2
          resources:
            requests:
              cpu: 50m
              memory: 64Mi
            limits:
              cpu: 200m
              memory: 128Mi
---
apiVersion: v1
kind: Service
metadata:
  name: health-demo-svc
  namespace: health-lab
spec:
  type: ClusterIP
  selector:
    app: health-demo
  ports:
    - name: http
      port: 80
      targetPort: 8080

Apply the manifest

Command 
kubectl apply -f health-demo.yaml

Expected result:

Expected output 
deployment.apps/health-demo created
service/health-demo-svc created

Watch Pod readiness

The Pods may first show 0/1 because the app intentionally waits 20 seconds before creating /tmp/ready.

Command 
kubectl get pods -n health-lab -l app=health-demo -w

Expected progression:

Expected output pattern 
NAME                           READY   STATUS    RESTARTS   AGE
health-demo-xxxxxxxxxx-abcde   0/1     Running   0          10s
health-demo-xxxxxxxxxx-abcde   1/1     Running   0          25s

Verify Service traffic

Command 
kubectl run curl -n health-lab --rm -i --tty \
  --image=curlimages/curl:8.10.1 \
  --restart=Never -- \
  curl -s http://health-demo-svc/

Expected output will show one Pod name:

Expected output 
hello from health-demo-xxxxxxxxxx-abcde

Check Service endpoints

Only ready Pods should appear as usable endpoints for the Service.

Command 
kubectl get endpointslices -n health-lab \
  -l kubernetes.io/service-name=health-demo-svc \
  -o wide

Alternative command for older clusters:

Command 
kubectl get endpoints health-demo-svc -n health-lab -o wide

Break readiness for one Pod

This simulates a Pod that is alive but not ready for user traffic.

Commands 
POD=$(kubectl get pod -n health-lab -l app=health-demo \
  -o jsonpath='{.items[0].metadata.name}')

echo "Testing Pod: $POD"

kubectl exec -n health-lab "$POD" -- rm -f /tmp/ready

Now check Pod status:

Command 
kubectl get pod -n health-lab "$POD"

Expected result after a few probe periods:

Expected output pattern 
NAME                           READY   STATUS    RESTARTS   AGE
health-demo-xxxxxxxxxx-abcde   0/1     Running   0          5m
Observation: The Pod is still running. Kubernetes did not restart it. It simply marked it NotReady and stopped using it for normal Service traffic.

Restore readiness

Command 
kubectl exec -n health-lab "$POD" -- touch /tmp/ready
kubectl get pod -n health-lab "$POD" -w

Expected result: the Pod returns from 0/1 to 1/1.

Break liveness for one Pod

This simulates an app that is no longer alive and needs restart.

Commands 
POD=$(kubectl get pod -n health-lab -l app=health-demo \
  -o jsonpath='{.items[0].metadata.name}')

echo "Breaking liveness in Pod: $POD"

kubectl exec -n health-lab "$POD" -- rm -f /tmp/live

Watch restart count:

Command 
kubectl get pod -n health-lab "$POD" -w

Expected result after liveness failure threshold is reached:

Expected output pattern 
NAME                           READY   STATUS    RESTARTS   AGE
health-demo-xxxxxxxxxx-abcde   0/1     Running   1          8m
health-demo-xxxxxxxxxx-abcde   1/1     Running   1          9m

Check events:

Command 
kubectl describe pod -n health-lab "$POD"

Look for an event similar to:

Expected event 
Warning  Unhealthy  kubelet  Liveness probe failed: HTTP probe failed with statuscode: 500
Normal   Killing    kubelet  Container app failed liveness probe, will be restarted

8. Lab: Working with StartupProbe

This lab simulates a slow-starting application that waits 35 seconds before opening its HTTP port. StartupProbe gives the app enough time to boot before liveness checks can kill it.

Create the startup probe manifest

Save this as startup-demo.yaml.

startup-demo.yaml 
apiVersion: apps/v1
kind: Deployment
metadata:
  name: startup-demo
  namespace: health-lab
spec:
  replicas: 1
  selector:
    matchLabels:
      app: startup-demo
  template:
    metadata:
      labels:
        app: startup-demo
    spec:
      containers:
        - name: app
          image: python:3.12-alpine
          imagePullPolicy: IfNotPresent
          ports:
            - containerPort: 8080
          command:
            - /bin/sh
            - -c
          args:
            - |
              cat > /tmp/server.py <<'PY'
              from http.server import BaseHTTPRequestHandler, HTTPServer
              import os

              class Handler(BaseHTTPRequestHandler):
                  def send_text(self, status, text):
                      self.send_response(status)
                      self.send_header("Content-Type", "text/plain")
                      self.end_headers()
                      self.wfile.write(text.encode())

                  def do_GET(self):
                      if self.path in ["/live", "/ready"]:
                          self.send_text(200, "ok\n")
                      else:
                          self.send_text(200, "slow app started\n")

              HTTPServer(("0.0.0.0", 8080), Handler).serve_forever()
              PY
              echo "Simulating slow application startup for 35 seconds..."
              sleep 35
              python /tmp/server.py

            periodSeconds: 5
            timeoutSeconds: 2

            timeoutSeconds: 2

9. Probe Parameters Explained

ParameterMeaningPractical guidance
initialDelaySecondsHow long to wait after container start before probing.Useful when not using StartupProbe, but StartupProbe is cleaner for slow-starting apps.
periodSecondsHow often to run the probe.5–10 seconds is common for many labs and small apps.
timeoutSecondsHow long Kubernetes waits for a response.Keep the health endpoint fast. A slow health endpoint causes false failures.
failureThresholdNumber of consecutive failures before Kubernetes acts.Use a higher value for liveness than readiness to avoid unnecessary restarts.
successThresholdNumber of successes needed after failure.For readiness, values greater than 1 can avoid flapping. For liveness and startup, it must be 1.

11. Deployment Best Practices

  • Use different endpoints for /live and /ready when the checks mean different things.
  • Keep probe endpoints lightweight. They should not perform heavy database queries.
  • Readiness may check essential dependencies; liveness should usually check only whether the app itself is recoverable.
  • Use StartupProbe for slow apps instead of making liveness delay very large.
  • Avoid aggressive liveness probes. Too many restarts can create cascading failures.
  • Use readiness during graceful shutdown so the Pod leaves Service traffic before it exits.
  • Document expected probe status codes with the application team.
  • In production, tune probe values based on real startup time, request latency, CPU throttling, and dependency behavior.

13. Quick Assessment Questions

Q1. A Pod is running but should not receive traffic yet. Which probe should fail?
Answer

ReadinessProbe.

Q2. A container is deadlocked and cannot recover. Which probe should trigger restart?
Answer

LivenessProbe.

Q3. A Java app needs 90 seconds to boot. What probe should you add?
Answer

StartupProbe, with enough periodSeconds × failureThreshold budget.

Q4. Should liveness fail just because the database is temporarily down?
Answer

Usually no. That can cause unnecessary restarts. Readiness may fail if the app cannot serve traffic without the database.

Q5. What command helps you confirm whether a Service has ready backends?
Answer

kubectl get endpointslices -n <namespace> -l kubernetes.io/service-name=<service-name> -o wide.
